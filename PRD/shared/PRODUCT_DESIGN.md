## 1. Isidoricaの動作モデル

Isidoricaは、読者の**状態**に応じて**アダプティブに振る舞う**知識プラットフォームである。

### 1.1 中核概念

```
┌─────────────────────────────────────────────────────────┐
│ 読者の状態（4次元）                                      │
│   K: 知識状態（ドメイン固有、動的）                      │
│   M: メタ学習スキル（比較的安定）                        │
│   P: アクセス目的（状況依存）                            │
│   T: 時間制約（状況依存）                                │
└─────────────────────────────────────────────────────────┘
                          ↓ 状態に応じて適応
┌─────────────────────────────────────────────────────────┐
│ Library（知識基盤）【中核】                              │
│   - アクセスモードの適応（P, T）                         │
│   - 知識状態への適応（K）                                │
│   - 学習支援の密度調整（M）                              │
│   - 学習パス（キュレーション＋動的調整）                 │
│                                                         │
│   └─ Laboratory（体験層）                               │
│        記事に埋め込まれる体験要素                        │
└─────────────────────────────────────────────────────────┘
                          ↕ 相互連携
┌─────────────────────────────────────────────────────────┐
│ Agora（対話基盤）                                        │
│   - Libraryコンテンツへの議論・批判                      │
│   - 議論結果がLibraryに還流                              │
└─────────────────────────────────────────────────────────┘
```

## 2. 読者モデル（4次元）

本セクションは4次元読者モデルの**正典**である。

### 2.1 なぜ4次元で捉えるか

読者の状態を**複数の独立した次元**で捉えることには、以下の利点がある：

1. **設計変数への直結**: 各次元が具体的な設計決定（コンテンツの詳細度、支援の密度など）に直接対応する
2. **状況変動の表現**: 同一人物が異なる場面で異なる状態を持つことを自然に表現できる
3. **理論的根拠**: 各次元が認知科学・社会学・辞書学の理論から演繹的に導出される

### 2.2 理論から導出される4次元

| 次元 | 由来する理論 | 設計への影響 |
|:---|:---|:---|
| **知識状態（K）** | ZPD（Vygotsky）、熟達化逆転効果（Kalyuga） | コンテンツの前提処理、詳細度 |
| **メタ学習スキル（M）** | 文化資本論（Bourdieu）、言語コード論（Bernstein） | 学習支援の密度 |
| **アクセス目的（P）** | 辞書機能論（Tarp）、状況的学習（Lave & Wenger） | 提供する情報の種類 |
| **時間制約（T）** | 満足化理論（Simon） | 要約度、詳細度 |

### 2.3 各次元の定義

#### K: 知識状態（Domain-specific, Dynamic）

当該トピックに対するユーザーの既存知識。**ドメインごとに異なり、学習により動的に変化**する。

| 値 | 定義 | 形式的条件 |
|:---|:---|:---|
| **K₀（未習得）** | 前提知識が不足している | $P_A \not\subseteq K_u$ |
| **K₁（発達中）** | 前提は充足だが当該概念は未習得 | $P_A \subseteq K_u, A \notin K_u$ |
| **K₂（習得済）** | 当該概念を習得している | $A \in K_u$ |

> **設計詳細**: [LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md) で形式モデルを定義

#### M: メタ学習スキル（Domain-general, Stable）

学習方法そのものに関するスキル。「どう学ぶか」を知っているか。**比較的安定した個人特性**。

| 値 | 定義 | 典型的な背景 |
|:---|:---|:---|
| **M_high** | 自律的に学習を計画・実行できる | 高等教育経験者、研究者、独学経験豊富 |
| **M_low** | 学習方法自体の支援が必要 | 学術的文献に不慣れ、学習経験が限られる |

> **設計詳細**: [LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md) で支援設計を定義

#### P: アクセス目的（Situation-specific）

このアクセスで何を達成したいか。**状況ごとに変化**する。

| 値 | 定義 | 典型的な問い |
|:---|:---|:---|
| **P_reception** | テキスト理解中に意味を知りたい | 「これは何？」 |
| **P_production** | 文章作成中に使い方を知りたい | 「これはどう使う？」 |
| **P_cognitive** | 体系的に概念を理解したい | 「なぜこうなる？」 |
| **P_solving** | 具体的問題を解決したい | 「どうすればできる？」 |

#### T: 時間制約（Situation-specific）

どれだけの時間をかける意思・余裕があるか。**状況ごとに変化**する。

| 値 | 定義 | 典型的状況 |
|:---|:---|:---|
| **T_quick** | 即座に答えが欲しい | 会議中、文章読解中、検索から来た |
| **T_extended** | 時間をかけて学ぶつもり | 自習時間、研究中、教養として |

### 2.4 次元の組み合わせと適応

読者を固定的な「タイプ」に分類するのではなく、**次元の組み合わせに応じて動的に設計を調整**する：

| K | M | P | T | システムの応答 |
|:---|:---|:---|:---|:---|
| K₀ | M_low | P_cognitive | T_extended | 前提から丁寧に説明、メタ学習支援付き、体系的構造 |
| K₀ | M_high | P_cognitive | T_extended | 前提から説明、ただし支援は最小限 |
| K₁ | any | P_cognitive | T_extended | 当該概念に集中、前提はリンクで提供 |
| K₂ | any | P_reception | T_quick | 定義のみ即時表示（TL;DR） |
| any | any | P_production | any | 用例、コロケーション、正しい使い方 |
| any | any | P_solving | any | 手順、適用例、トラブルシューティング |

## 3. ユーザー体験フロー

本セクションでは、ユーザーがIsidoricaをどのように体験するかを具体的に記述する。

### 3.1 状態の判定タイミング

4次元の各状態がいつ判定・更新されるか：

| 次元 | 判定タイミング | 判定方法 |
|:---|:---|:---|
| **K（知識状態）** | 記事表示時、学習完了時 | 明示的な完了マーク、理解度確認の結果、滞在時間 |
| **M（メタ学習スキル）** | 初回訪問時、明示的な設定時 | オンボーディングでの質問、ユーザー設定 |
| **P（アクセス目的）** | 各アクセス時 | 流入元（検索、学習パス、直接リンク）、UIでの明示的選択 |
| **T（時間制約）** | 各アクセス時 | UIでの明示的選択、過去の行動パターン |

### 3.2 典型的なユーザージャーニー

#### ジャーニーA: 新規ユーザーの初回訪問

```
1. 検索エンジンから「ナッシュ均衡」記事に到着
   → P = P_reception（検索から来た）, T = T_quick（デフォルト）
   
2. TL;DR（要約）が最初に表示される
   → 「もっと詳しく」リンクが見える
   
3. ユーザーが「もっと詳しく」をクリック
   → P = P_cognitive に遷移、T = T_extended に遷移
   
4. 前提知識チェック
   → 「この記事を理解するには『ゲーム理論の基礎』が必要です」
   → K = K₀ と判定
   
5. オプション提示
   a) 前提記事から読む → 学習パスを提案
   b) 簡略版で続ける → アコモデーション提供
   c) このまま読む → 警告のみで続行
```

#### ジャーニーB: 学習パスに沿った学習

```
1. ユーザーが「ゲーム理論入門」パスを選択
   → P = P_cognitive, T = T_extended, M = 既存値を使用
   
2. パス内の最初の概念「効用理論」を表示
   → K = K₁（前提は充足、当該概念は未習得）
   
3. 学習モードで記事を読む
   → Phase構造で表示、Laboratory埋め込み
   
4. 記事末尾で「理解できた」をマーク
   → K が更新: 効用理論 ∈ K_u
   
5. 次の概念「戦略的相互作用」に自動遷移
   → 「効用理論」は既知として扱われる
```

#### ジャーニーC: 専門家の参照利用

```
1. 専門家が直接URLで「情報の非対称性」記事にアクセス
   → 過去の行動から K = K₂, M = M_high と推定
   
2. 参照モードで表示
   → TL;DR + 定義 + 必要なら詳細展開
   
3. 関連概念「逆選択」へリンクをクリック
   → 同じく参照モード、即時表示
```

### 3.3 モード切り替えのUI概念

ユーザーは以下の方法でモードを切り替えられる：

| 切り替え | トリガー | UI要素 |
|:---|:---|:---|
| **参照→学習** | 「詳しく学ぶ」ボタン | TL;DRセクション内 |
| **学習→参照** | 「要約を見る」ボタン | 記事ヘッダー |
| **目的（P）の変更** | タブまたはドロップダウン | 「定義」「使い方」「手順」タブ |
| **時間（T）の変更** | 暗黙的（行動）または明示的 | 「時間がない」チェックボックス |

### 3.4 状態遷移の可視化

ユーザーに対して現在の状態を可視化する：

| 可視化 | 対象 | 理由 |
|:---|:---|:---|
| **学習進捗バー** | K（学習パス内） | 現在地と残りを把握（B-2: 外的構造） |
| **前提知識ステータス** | K（記事表示時） | 不足している前提を明示（B-1: 暗黙知明示化） |
| **モード表示** | P, T | 現在のモードを認識させる |
| **支援レベル設定** | M | ユーザーが自分で調整可能に（D-3: 認識的信念涵養） |

## 4. Libraryのアダプティブ動作

### 4.1 設計決定: アクセスモードの統合

**従来の分離モデル**（他サービス）:
- 「体系的学習」と「辞書引き」が別々のサービス/モード
- **問題**: 読者は固定的な「タイプ」ではなく、状況に応じて振る舞いを変える

**Isidoricaの統合モデル**:
- **単一のコンテンツ基盤**が、読者の状態（K, M, P, T）に応じて**異なるアクセスモード**を提供
- モードは別々の機能ではなく、同一コンテンツの異なる表示方法

### 4.2 アクセスモードの適応

| P（目的） | T（時間） | 表示モード | 表示内容 |
|:---|:---|:---|:---|
| P_cognitive | T_extended | **学習モード** | Phase 1-6構造、詳細な説明 |
| P_reception | T_quick | **参照モード** | TL;DR、定義、即時表示 |
| P_production | any | **産出支援モード** | 用例、正しい使い方 |
| P_solving | any | **問題解決モード** | 手順、適用例 |

### 4.3 知識状態への適応

| K（知識状態） | 適応 |
|:---|:---|
| K₀（前提不足） | アコモデーション：簡略説明を提供、前提記事へのリンク |
| K₁（発達中） | 当該概念に集中、前提は既知として扱う |
| K₂（習得済） | スキップ提案、または復習オプション |

> **設計詳細**: [LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md) を参照

## 5. 学習パス設計

### 5.1 設計決定: キュレーション＋動的調整

**3層モデル**:

```
Layer 1: 知識ネットワーク（基盤）
  - 概念間の依存関係（requires）
  - 客観的な構造

Layer 2: キュレーションされたパス（オーバーレイ）
  - 専門家/コミュニティが作成した「おすすめの旅程」
  - 例: 「ゲーム理論入門」「確率論から機械学習へ」
  - 【任意】パスなしでも自由探索は可能

Layer 3: 動的調整（エンジン）
  - ユーザーの K（知識状態）を参照
  - キュレーションされたパスを調整:
    ・既知の概念をスキップまたは簡略化
    ・前提未充足の概念を挿入または警告
```

### 5.2 なぜこの設計か

3つの設計選択肢を検討し、それぞれの問題点を分析した：

#### 選択肢A: 完全に動的生成のみ

ユーザーの知識状態（K）に基づき、システムがすべての学習パスを自動生成する。

| 問題 | 説明 |
|:---|:---|
| **教育的知識の喪失** | 熟達した教育者は「どの順序で学ぶと効果的か」を知っている。これはアルゴリズムには自明でない暗黙知である |
| **コールドスタート問題** | 新規ユーザーの K が未知の場合、適切なパスを生成できない |
| **B-2原則違反** | 外的構造の提供（B-2）は、特に M_low の読者に必要。動的生成のみでは「おすすめの道筋」が見えず、選択肢過多で認知負荷が増大する |

#### 選択肢B: キュレーションのみ

専門家が定義した固定的な学習パスのみを提供する。

| 問題 | 説明 |
|:---|:---|
| **K次元の無視** | ユーザーの既存知識を無視し、全員に同じ順序を強制する |
| **熟達化逆転効果** | 既に習得済みの内容を繰り返すと、学習効率が低下する（Kalyuga, 2007） |
| **ZPD原則違反** | 学習者の現在地点（Zone of Proximal Development）を考慮しない |

#### 選択肢C: キュレーション＋動的調整（採用）

キュレーションされたパスを基盤とし、ユーザーの K に応じて動的に調整する。

| 利点 | 説明 |
|:---|:---|
| **教育的知識の活用** | 専門家の暗黙知（効果的な学習順序）を保持 |
| **個別適応** | K に基づき、既知の内容をスキップ、未充足の前提を補完 |
| **B-2との両立** | キュレーションが「おすすめの道筋」を提供しつつ、適応により個別最適化 |

### 5.3 動作例

**キュレーションされたパス「ゲーム理論入門」**:
```
1. 効用理論 → 2. 戦略的相互作用 → 3. ナッシュ均衡 → 4. 混合戦略
```

**ユーザーA の K = {効用理論, 戦略的相互作用}**:
```
[既知] 1. 効用理論 → [スキップ提案]
[既知] 2. 戦略的相互作用 → [スキップ提案]
[未習得] 3. ナッシュ均衡 → [ここから開始]
[未習得] 4. 混合戦略 → [次のステップ]
```

> **設計詳細**: [LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md) を参照

## 7. LaboratoryとAgoraの動作

本セクションでは、LibraryとともにIsidoricaを構成するLaboratoryとAgoraの動作を概説する。

### 7.1 Laboratory（体験基盤）

#### 7.1.1 Laboratoryの役割

| 項目 | 内容 |
|:---|:---|
| **定義** | 能動的推論を通じた学習体験を提供する空間 |
| **配置** | Library記事内に埋め込み |
| **目的** | 抽象概念を操作・介入を通じて体験させる |

#### 7.1.2 ユーザー体験フロー

```
1. ユーザーがLibrary記事を読んでいる
   → 「概念X」の説明セクション

2. 記事内にLaboratory要素が現れる
   → 「実際に試してみよう」ボックス

3. パラメータを操作
   → スライダー、入力フィールド等

4. 結果を観察
   → ビジュアライゼーション、計算結果

5. 記事の説明と照合
   → 「予測と結果が一致しましたか？」
```

#### 7.1.3 Laboratoryの設計原則

| 原則 | 実現 |
|:---|:---|
| **A-1（先行知識活性化）** | 操作前に「どうなると思うか」を問う |
| **A-2（身体接地）** | 抽象概念を操作可能なインターフェースに |
| **C-1（生産的混乱）** | 直感に反する結果を意図的に設計 |

> **設計詳細**: [LABORATORY_DESIGN.md](../engine/LABORATORY_DESIGN.md) を参照

### 7.2 Agora（対話基盤）

#### 7.2.1 Agoraの役割

| 項目 | 内容 |
|:---|:---|
| **定義** | 探究対話を通じた知識の共同精錬の空間 |
| **目的** | 真理や最良の解を共同で見つける（探究対話） |
| **理論基盤** | 相互行為的語用論、アーギュメンテーション理論、コミュニティ設計論、対話の政治哲学、対話の倫理 |

> **理論的基盤**: [AGORA_FOUNDATIONS.md](../library/AGORA_FOUNDATIONS.md) を参照

#### 7.2.2 Agoraの3層構造

Agoraは記事単位に限定せず、以下の3層で構成される：

```
Agora
├─ Article Discussions（記事単位）
│    - 特定記事への質問、指摘、議論
│    - F-3（知識循環）の主要経路：議論→記事改善
│
├─ Topic Discussions（トピック/ドメイン単位）
│    - 「ゲーム理論」「確率論」等の横断的議論
│    - 「概念Aと概念Bの関係は？」のような探究対話
│
└─ Meta Discussions（サイト運営）
     - 編集方針、コミュニティ規範の議論
     - コミュニティの自己統治
```

#### 7.2.3 ユーザー体験フロー

**記事単位の議論**:
```
1. ユーザーがLibrary記事を読む
   → 記事下部に「議論」セクション

2. 質問や疑問を投稿
   → 「この説明が分かりにくい」「反例はあるか」

3. 他のユーザーや編集者が応答
   → 回答、補足説明、異なる視点

4. 議論が知識に還流
   → 有用な指摘は記事に反映（F-3: 知識循環）
```

**横断的議論**:
```
1. ユーザーがTopic Discussionsにアクセス
   → 「確率論」トピックの議論一覧

2. 横断的な問いを投稿
   → 「ベイズ主義と頻度主義、どちらが適切な場面は？」

3. 複数の記事を参照しながら議論
   → 関連記事へのリンク、引用

4. 議論結果がまとめられる
   → FAQ化、関連記事への反映
```

#### 7.2.4 AIファシリテーター

AGORA_FOUNDATIONSに基づき、AIが議論を支援する：

| 機能 | 説明 |
|:---|:---|
| **修復トリガー** | 論点がかみ合っていない場合、明確化を求める介入 |
| **フレーム管理** | 「論争フレーム」への移行を検出し、「探究対話フレーム」へリフレーミング |
| **フェイス保護** | 反論が相手のポジティブ・フェイスを損なわないよう表現の緩和を提案 |
| **構造化支援** | Toulminモデルに基づき「主張の根拠は？」等のプロンプト |
| **未回答検出** | 隣接ペア（質問→応答）の欠損を検出し、応答を促す |

#### 7.2.5 Agoraの設計原則

| 原則 | 実現 |
|:---|:---|
| **F-1（相互行為安全性）** | 優先構造支援、フェイス保護、AIファシリテーション |
| **F-2（議論構造化）** | Toulminプロンプト、質問/回答/反論の可視化 |
| **F-3（知識循環）** | 議論→記事改善の明確な経路 |
| **D-2（認識的多様性）** | 複数の視点・解釈の尊重、「探究対話」フレームの明示 |

> **設計詳細**: [DISCUSSIONS_DESIGN.md](../library/DISCUSSIONS_DESIGN.md) を参照

### 7.3 3基盤の連携

#### 7.3.1 Library ↔ Laboratory

| 方向 | 実現方法 |
|:---|:---|
| **概念→体験** | 記事内にLaboratoryを埋め込み、説明→実践の流れを構築 |
| **体験→理解** | Laboratory結果を記事説明と照合、理解を確認 |

#### 7.3.2 Library ↔ Agora

| 方向 | 実現方法 |
|:---|:---|
| **コンテンツ→議論** | 記事に紐づくコメント、質問機能 |
| **議論→コンテンツ** | 議論で得られた知見を記事に反映、改訂履歴で透明化 |

#### 7.3.3 Laboratory ↔ Agora

| 方向 | 実現方法 |
|:---|:---|
| **体験→議論** | Laboratoryでの体験結果をAgoraで共有、「こうなった」報告 |
| **議論→体験** | Agoraでの提案を受けてLaboratoryパラメータ追加・改善 |
