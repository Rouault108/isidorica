# リスク分析 (Risk Analysis)

本ドキュメントでは、Isidoricaプロジェクトにおける技術的・運用的・法的・学習設計・プロジェクト上のリスクを特定し、その対策を定義する。

**最終更新**: 2026-01-08
**定期レビュー**: 6ヶ月ごと（次回: 2026年7月）

---

## リスク評価の枠組み

### 評価基準

| 項目 | 定義 |
| :--- | :--- |
| **影響度 (Impact)** | リスクが顕在化した場合のプロジェクトへの影響（高/中/低） |
| **発生確率 (Probability)** | リスクが顕在化する可能性（高/中/低） |
| **リスクスコア** | 影響度 × 発生確率（H×H=最高、L×L=最低） |

### リスクスコアマトリクス

|  | 影響度: 低 | 影響度: 中 | 影響度: 高 |
| :--- | :--- | :--- | :--- |
| **確率: 高** | 中 | 高 | 最高 |
| **確率: 中** | 低 | 中 | 高 |
| **確率: 低** | 最低 | 低 | 中 |

## 技術的リスク

### [R-TECH-01] WASM/Laboratoryのパフォーマンス

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | Pyodide/webRのWASMロードがモバイル端末で遅延し、UXを損なう |
| **影響度** | 高（Laboratory体験はIsidoricaの差別化要素） |
| **発生確率** | 中（モバイル4G環境、低スペック端末で顕在化しやすい） |
| **リスクスコア** | 高 |
| **対策** | 1. プリレンダリング（静的プレビュー）で初期表示を高速化<br>2. 遅延ロード（ユーザーが「実行」をクリックするまでWASMをロードしない）<br>3. Web Workers使用でメインスレッドブロック回避<br>4. 軽量ランタイム（QuickJS）を優先し、Pyodideは必要時のみ |
| **残存リスク** | 低（対策により許容範囲内） |

### [R-TECH-02] SSGビルド時間の増大

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 記事数増加に伴いAstroのビルド時間が許容範囲を超える |
| **影響度** | 中（開発効率とデプロイ頻度に影響） |
| **発生確率** | 中（1000記事を超えた場合に顕在化） |
| **リスクスコア** | 中 |
| **対策** | 1. インクリメンタルビルド（差分のみ再生成）<br>2. コンテンツキャッシュ戦略<br>3. 閾値（500記事/10分）を設定し、超過時にアーキテクチャ再検討 |
| **残存リスク** | 低（監視と閾値で早期対応可能） |
| **関連ドキュメント** | [SYSTEM_ARCHITECTURE.md 1.3節](../engine/technical/SYSTEM_ARCHITECTURE.md)（スケーリング戦略の詳細） |

### [R-TECH-03] ブラウザ互換性

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 古いブラウザ（Safari 14以前、IE）でWASMやCSS機能が動作しない |
| **影響度** | 中（一部ユーザーがLaboratoryを利用できない） |
| **発生確率** | 低（対象ブラウザシェアは減少傾向） |
| **リスクスコア** | 低 |
| **対策** | 1. グレースフルデグラデーション（静的プレビューのみ表示）<br>2. サポートブラウザを明示し、ユーザーに通知 |
| **残存リスク** | 最低 |

### [R-TECH-04] 依存パッケージの脆弱性

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | npm依存パッケージに重大な脆弱性が発見される |
| **影響度** | 高（セキュリティ侵害、信頼性低下） |
| **発生確率** | 中（OSSエコシステムの常態） |
| **リスクスコア** | 高 |
| **対策** | 1. Dependabot/Renovateで自動検出<br>2. CI/CDでセキュリティスキャン（npm audit）<br>3. 重大な脆弱性は24時間以内に対応するSLA |
| **残存リスク** | 低（継続的監視で対応可能） |

### [R-TECH-05] 外部APIへの依存

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 執筆支援で使用する外部API（Google Books, CrossRef等）がダウンまたは仕様変更される |
| **影響度** | 低（執筆作業の一時的停滞） |
| **発生確率** | 低（主要サービスの信頼性は高い） |
| **リスクスコア** | 低 |
| **対策** | 1. 複数のAPIソースを並列利用（冗長化）<br>2. 手動入力モードへのシームレスなフォールバック機能<br>3. API仕様変更を検知するテストスイート |
| **残存リスク** | 最低 |

### [R-TECH-06] クライアントサイドストレージ制限

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 学習者のデバイスにデータを蓄積する設計（学習者主権）により、LocalStorageの5MB制限到達やSafari ITPによる7日削除でデータ消失が発生する |
| **影響度** | 高（ユーザーの学習記録消失、機能停止） |
| **発生確率** | 中（長期ヘビーユーザー、Safari利用者で顕在化） |
| **リスクスコア** | 高 |
| **対策** | 1. **OPFS移行（推奨）**: Origin Private File Systemへ移行。容量制限緩和かつSafari ITP対象外<br>2. **エクスポート/インポート機能**: ユーザーがデータをバックアップ可能に（ユーザー削除への唯一の対策）<br>3. **容量警告UI（フォールバック環境向け）**: IndexedDB/LocalStorageフォールバック時のみ、使用量が閾値を超えたら警告を表示<br>4. **段階的フォールバック**: OPFS → IndexedDB → LocalStorage |
| **残存リスク** | 低（OPFS移行により容量・ITP問題を解消。ユーザーによる意図的削除は不可避） |
| **関連ドキュメント** | [KPI_FRAMEWORK.md §6.5](./KPI_FRAMEWORK.md)（学習者向け機能とデータ保存） |

> **ストレージ選択の根拠**: OPFSはSafari ITPの対象外であり、容量も実質無制限。主要ブラウザ（Chrome 86+, Firefox 111+, Safari 15.2+）で対応済み。ユーザーが「サイトデータを削除」した場合はすべてのストレージが削除されるため、エクスポート機能が重要なバックアップ手段となる。

### [R-TECH-07] クライアントサイド・データマイグレーション

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 学習者主権（Local-First）アーキテクチャにおいて、データスキーマ変更時のマイグレーションに失敗し、ユーザーの学習記録が破損または消失する |
| **影響度** | 高（学習記録の不可逆的消失） |
| **発生確率** | 低（スキーマ変更は低頻度だが、発生時の影響は甚大） |
| **リスクスコア** | 中 |
| **対策** | 1. **スキーマバージョン管理**: データに`_version`フィールドを含め、マイグレーションの必要性を判定<br>2. **防御的読み込み**: 新規フィールドはオプショナル、古いデータでも動作するコード設計<br>3. **マイグレーションチェーン**: 任意の旧バージョンから段階的に変換（v1→v2→v3）<br>4. **事前バックアップ**: マイグレーション前にOPFS内に旧データを一時保存<br>5. **エクスポート促進**: 重大なスキーマ変更前にユーザーへエクスポートを促す通知 |
| **残存リスク** | 低（多層防御により軽減） |
| **関連ドキュメント** | [LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md)、[ADR-020](../engine/technical/ARCHITECTURE_DECISION_RECORD.md) |

### [R-TECH-08] GitHub Discussions API依存

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | Agora（議論プラットフォーム）のバックエンドであるGitHub Discussions APIが廃止・仕様変更された場合、Agoraが機能停止または大幅な改修が必要になる |
| **影響度** | 高（コミュニティ機能の喪失） |
| **発生確率** | 低（GitHubは主要プラットフォームであり、APIの安定性は高い） |
| **リスクスコア** | 中 |
| **対策** | 1. **抽象化レイヤー**: 議論データへのアクセスを抽象化し、バックエンド交換を容易に<br>2. **データエクスポート**: 定期的に議論データをバックアップ<br>3. **代替案の事前調査**: Giscus、独自実装等の移行先を把握<br>4. **APIバージョン監視**: GitHub API変更のアナウンスを監視 |
| **残存リスク** | 低（抽象化により移行コストを軽減） |
| **関連ドキュメント** | [DISCUSSIONS_DESIGN.md](../library/DISCUSSIONS_DESIGN.md)、[ADR-019](../engine/technical/ARCHITECTURE_DECISION_RECORD.md) |

### [R-TECH-09] Cloudflareベンダー集中

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | Hosting (Pages), CDN, DNS, Registrar, Workers, R2, Web Analytics, Access (Zero Trust) の全てをCloudflareに依存しており、サービス終了・規約変更・アカウント停止時に代替が困難になる |
| **影響度** | 高（サービス全体の基盤が影響を受ける） |
| **発生確率** | 低（Cloudflareは主要プロバイダであり、突然の撤退リスクは低い） |
| **リスクスコア** | 中 |
| **対策** | 1. **SSGアーキテクチャ**: 出力物（HTML/CSS/JS）は純粋な静的ファイルであり、任意のホスティングに移行可能<br>2. **Velite採用**: データ層がフレームワーク非依存のため、コンテンツパイプラインの移行コストも低い<br>3. **代替先の把握**: Vercel, Netlify, AWS S3+CloudFront を移行候補としてリストアップ<br>4. **ドメイン移管の容易性**: Cloudflare Registrarは移管制限がなく、他レジストラへの移行は標準手順で可能<br>5. **R2抽象化**: R2アクセスはS3互換APIを使用し、アセットURLをCNAME経由で抽象化することで、ストレージ移行時のURL変更を最小化する<br>6. **Workers抽象化**: Workers固有のAPI（KV, Durable Objects等）への依存を最小化し、移行先（AWS Lambda, Vercel Functions）でも動作するポータブルなコードを心がける |
| **残存リスク** | 中（静的サイト・DNSの移行は低コストだが、R2およびWorkersへの依存により、完全移行には中〜高のコストが発生する） |
| **関連ドキュメント** | [ADR-008](../engine/technical/ARCHITECTURE_DECISION_RECORD.md)、[ADR-011](../engine/technical/ARCHITECTURE_DECISION_RECORD.md)、[ADR-015](../engine/technical/ARCHITECTURE_DECISION_RECORD.md) |

## 運用的リスク

### [R-OPS-01] コンテンツ品質の劣化

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | レビュー体制の不備により、不正確な情報や低品質な記事が公開される |
| **影響度** | 高（信頼性がIsidoricaの生命線） |
| **発生確率** | 中（コントリビューター増加時に顕在化） |
| **リスクスコア** | 高 |

| **残存リスク** | 中（完全な防止は不可能、検出・修正を迅速化） |

### [R-OPS-02] コミュニティの停滞

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | コントリビューターが集まらず、コンテンツ拡充が停滞する |
| **影響度** | 高（持続可能性に直結） |
| **発生確率** | 中（初期段階で特に顕在化しやすい） |
| **リスクスコア** | 高 |
| **対策** | 1. First Domain（統計リテラシー）で成功事例を作る<br>2. 記事執筆ガイドラインとテンプレートの整備<br>3. 大学・教育機関との連携<br>4. Agoraによるコミュニティ形成 |

### [R-OPS-03] サービス可用性

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | Cloudflare Pages/R2の障害によりサービスがダウンする |
| **影響度** | 中（読者体験の中断） |
| **発生確率** | 低（Cloudflareの信頼性は高い） |
| **リスクスコア** | 低 |
| **対策** | 1. 静的サイトのためCDNエッジキャッシュで耐障害性確保<br>2. 月間稼働率99.9%を目標（NON_FUNCTIONAL_REQUIREMENTS.md） |

### [R-OPS-04] 知識エコシステム断片化

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 類似ミッションのサービスが乱立し、貢献者の分散・寄付者疲れ・低品質模倣による評判リスクが発生する |
| **影響度** | 中（持続可能性と品質認知に影響） |
| **発生確率** | 低（参入障壁は高いが、AI時代に変化の可能性） |
| **リスクスコア** | 低 |
| **対策** | 1. **協調優先**: 同志サービスとは競争ではなく相互参照・連携を志向<br>2. **差別化維持**: 理論的基盤、アクセシビリティ、日本語品質での独自性<br>3. **オープンライセンス（G-3）**: CC BY-SAにより相互利用を促進<br>4. **透明なガバナンス**: 品質基準と意思決定プロセスの公開 |
| **残存リスク** | 最低 |
| **関連ドキュメント** | [IMPLEMENTATION_STRATEGY.md §6, §9](./IMPLEMENTATION_STRATEGY.md)（競合参入対策、知識エコシステムにおける位置づけ） |

> **公共財としての立場**: Isidoricaにとって「類似サービスの出現」は商業的脅威ではなく、ミッション達成の証明である。真の成功とは、質の高い知識が社会に広く行き渡ること（Isidorica経由でなくても）である。

### [R-OPS-05] コミュニティの健全性悪化

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | Agora（議論プラットフォーム）において、対立的な議論、トロール行為、または外部からの組織的攻撃により、建設的な議論が不可能になり、良質な貢献者が離脱する |
| **影響度** | 高（コミュニティの持続可能性と品質に直結） |
| **発生確率** | 中（GitHub認証により匿名攻撃は抑制されるが、専門家間・イデオロギー対立は起こりうる） |
| **リスクスコア** | 高 |
| **対策** | 1. **行動規範（Code of Conduct）**: 許容される行動と禁止行為の明文化<br>2. **段階的モデレーション**: 警告→一時停止→永久BANの段階的対応<br>3. **構造化議論（Toulminモデル）**: 感情的対立を論理的分析に誘導<br>4. **ポライトネス支援**: 投稿前のトーンチェック（将来実装）<br>5. **段階的参加権限**: 新規ユーザーは閲覧中心、貢献実績で発言権拡大 |
| **残存リスク** | 中（完全な防止は不可能、早期検知と迅速対応で被害を最小化） |
| **関連ドキュメント** | [AGORA_FOUNDATIONS.md §2](../library/AGORA_FOUNDATIONS.md)（証言的不正義）、[DISCUSSIONS_DESIGN.md](../library/DISCUSSIONS_DESIGN.md)、[GOVERNANCE.md](../library/GOVERNANCE.md) |

## 法的リスク

### [R-LEGAL-01] 著作権侵害

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 引用の範囲を超えた転載、画像の無断使用により著作権侵害が発生する |

| **発生確率** | 中（コントリビューターの理解不足により発生しうる） |
| **リスクスコア** | 高 |
| **対策** | 1. 引用ガイドラインの整備（LEGAL_REQUIREMENTS.md）<br>2. 画像追加時の許諾証拠添付必須（GOVERNANCE.md）<br>3. DMCA対応プロセスの明確化<br>4. レビュー時の著作権チェック項目 |

### [R-LEGAL-02] プライバシー侵害

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | アクセス解析やAgora機能で個人情報を不適切に収集・処理する |
| **影響度** | 高（法的責任、信頼性低下） |
| **発生確率** | 最低（設計段階でプライバシーファーストを採用、運営目的での学習データ収集を全面禁止） |
| **リスクスコア** | 低 |
| **対策** | 1. **学習者主権の原則**: 学習データ（Quiz履歴、LP進捗等）は学習者のデバイス（LocalStorage）にのみ保存、サーバー送信なし<br>2. **最小限のデータ収集**: 読者評価（記事ID + Yes/No）のみ匿名イベントとして収集、ユーザー識別子なし<br>3. アクセス解析はCloudflare Web Analytics（Cookie不使用、匿名化）<br>4. Agoraの認証はGitHub OAuth（個人情報の直接保持を回避）<br>5. プライバシーポリシーの明示 |
| **残存リスク** | 最低 |
| **関連ドキュメント** | [KPI_FRAMEWORK.md §6](./KPI_FRAMEWORK.md)（学習者主権、プライバシー原則） |

| **残存リスク** | 最低 |

## 学習設計リスク

本セクションは、[PRODUCT_DESIGN.md](./PRODUCT_DESIGN.md)で定義されたアダプティブ動作と4次元読者モデル（K/M/P/T）に関連するリスクを扱う。

### [R-LEARN-01] 知識状態（K次元）の誤判定

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | ユーザーの知識状態（K）を誤判定し、不適切な適応を行う（前提があるのに前提説明を表示、または前提がないのに省略） |
| **影響度** | 高（学習体験の質低下、離脱リスク） |
| **発生確率** | 中（明示的なマーク以外の判定は不確実） |
| **リスクスコア** | 高 |
| **対策** | 1. ユーザーによる明示的な「理解した」マークを主要シグナルとする<br>2. 誤判定時の簡単な修正手段を提供（「私はこれを知っています」「前提から学びたい」）<br>3. 判定に確信がない場合は安全側（前提説明を提供）に倒す |
| **残存リスク** | 低（ユーザーによる修正が可能） |
| **関連ドキュメント** | [PRODUCT_DESIGN.md §4.3](./PRODUCT_DESIGN.md)、[LEARNER_STATE_MODEL.md](./LEARNER_STATE_MODEL.md) |

### [R-LEARN-02] メタ学習スキル（M次元）への過度な仮定

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | M_high（高メタ学習スキル）と誤判定し、M_lowのユーザーに十分な支援を提供しない |
| **影響度** | 高（教育的公正の観点から深刻） |
| **発生確率** | 中（自己申告やオンボーディングの精度に依存） |
| **リスクスコア** | 高 |
| **対策** | 1. デフォルトはM_low扱い（支援は多めに提供）<br>2. ユーザーが明示的に「支援を減らす」を選択可能に<br>3. メタ学習支援（Before/After Readingガイド）は非強制的に提供 |
| **残存リスク** | 低（安全側にデフォルト設定） |
| **理論的根拠** | [DESIGN_PRINCIPLES.md](./DESIGN_PRINCIPLES.md) B-1（暗黙知明示化）、B-2（外的構造提供） |

### [R-LEARN-03] 流暢性の錯覚

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 読者が「読んだだけで分かったつもり」になり、転移可能な知識を獲得できない |
| **影響度** | 高（学習効果の欠如） |
| **発生確率** | 高（流暢な読解はデフォルトで錯覚を生む） |
| **リスクスコア** | 最高 |
| **対策** | 1. 先行知識活性化シーケンス（A-1）：「どうなると思うか」を問う<br>2. Laboratory埋め込み：操作を通じた体験的理解<br>3. 確認問題機能の提供（学習者自身のため、データは非収集） |
| **残存リスク** | 中（完全な防止は不可能） |
| **理論的根拠** | [LEARNING_FOUNDATIONS.md §5.2.2](./LEARNING_FOUNDATIONS.md)、[INTEGRATED_DESIGN_GUIDELINES.md §4.3](./INTEGRATED_DESIGN_GUIDELINES.md) A-1 |

> **注**: Quiz正答率等の学習成果指標は、プライバシー保護のためKPIとして収集しない。Quiz機能は学習者自身のために提供し、データは学習者のデバイスにのみ保存される。

### [R-LEARN-04] 熟達化逆転効果

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | K₂（習得済）のユーザーに対して過剰な説明を提供し、認知負荷を増大させる |
| **影響度** | 中（熟達者の離脱リスク） |
| **発生確率** | 中（K次元の判定精度に依存） |
| **リスクスコア** | 中 |
| **対策** | 1. K次元に応じたスキップ提案（R-LEARN-01の対策と連携）<br>2. 参照モード（P_reception × T_quick）でのTL;DR即時表示<br>3. 目次による任意セクションへのジャンプ |
| **残存リスク** | 低 |
| **理論的根拠** | [LEARNING_FOUNDATIONS.md §5.2.4](./LEARNING_FOUNDATIONS.md)（Kalyuga, 2007） |

### [R-LEARN-05] アクセス目的（P次元）と時間制約（T次元）の誤読

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | ユーザーのアクセス目的（P）や時間制約（T）を誤読し、不適切な表示モードを提供する（学習したいのに参照モード、急いでいるのに詳細表示など） |
| **影響度** | 中（UX低下、目的達成の遅延） |
| **発生確率** | 中（流入元からの推定は完璧ではない） |
| **リスクスコア** | 中 |
| **対策** | 1. モード切り替えUIを常に提供（「詳しく学ぶ」「要約を見る」）<br>2. 流入元（検索、学習パス、直接リンク）からの推定は初期値のみ<br>3. ユーザーの明示的選択を最優先 |
| **残存リスク** | 低（即時の切り替えが可能） |
| **関連ドキュメント** | [PRODUCT_DESIGN.md §3.4](./PRODUCT_DESIGN.md) |

### [R-LEARN-06] 読者評価の回答率低下

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 学習効果の唯一のKPIである読者評価の回答率が低すぎると、有意なデータが得られない |
| **影響度** | 中（学習効果の評価が困難になる） |
| **発生確率** | 中（任意参加のフィードバックは一般的に回答率が低い） |
| **リスクスコア** | 中 |
| **対策** | 1. UIを目立たせつつも邪魔にならない設計<br>2. 回答しなくても読者体験に影響なし（強制しない）<br>3. 絶対値より傾向を重視する分析アプローチ |
| **残存リスク** | 低（入力品質による間接保証がバックアップ） |
| **関連ドキュメント** | [KPI_FRAMEWORK.md §6.3](./KPI_FRAMEWORK.md)（読者評価の設計上の注意） |

## プロジェクトリスク

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | 機能追加要望により、スコープが際限なく拡大する |
| **影響度** | 中（リソース分散、品質低下） |
| **発生確率** | 中（柔軟性を重視するプロジェクトで発生しやすい） |
| **リスクスコア** | 中 |
| **対策** | 1. Phase定義による段階的導入<br>2. 「将来検討」カテゴリの明確化<br>3. PRD（本ドキュメント群）による合意形成 |

| 項目 | 内容 |
| :--- | :--- |

| **影響度** | 中（長期的な開発効率低下） |
| **発生確率** | 中（スタートアップ的な開発スタイルで発生しやすい） |
| **リスクスコア** | 中 |
| **対策** | 1. テストカバレッジ80%以上（NON_FUNCTIONAL_REQUIREMENTS.md）<br>2. コードレビュー必須<br>3. 定期的なリファクタリングスプリント |
| **残存リスク** | 低 |

### [R-PROJ-04] 属人化（バス係数）

| 項目 | 内容 |
| :--- | :--- |
| **リスク** | キーパーソン（主要開発者）の離脱によりプロジェクトが継続不能になる |
| **影響度** | 最高（プロジェクト存続の危機） |
| **発生確率** | 中（少人数プロジェクトとしての宿命） |
| **リスクスコア** | 高 |
| **対策** | 1. 徹底的なドキュメント化（本PRD群の整備）<br>2. "Why"を含めた意思決定プロセスの記録（ADR）<br>3. 開発環境のコンテナ化（DevContainers）によるオンボーディング平準化 |
| **残存リスク** | 中（ドキュメントがあっても知識移転にはコストがかかる） |
| **関連ドキュメント** | [KPI_FRAMEWORK.md §8.5](./KPI_FRAMEWORK.md)（リスク指標） |
