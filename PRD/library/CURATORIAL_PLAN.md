# キュレーション計画 (Curatorial Plan)

Isidoricaは、世界のすべての知識を扱うことを目指すプロジェクトである。しかし、初期段階のリソースは限られている。
本ドキュメントでは、**「どの知識から優先的にIsidoricaに取り込むか」**というキュレーション（選書・展示計画）の方針を定義する。

**最終更新**: 2026-01-09

---

## 1. キュレーションの基本方針

### 1.1 戦略的意図

以下の観点から初期コンテンツを選定する。

1.  **公共知としての優先度**: 「すべての人」に必要な知識か。市民生活に直結するか。
2.  **Isidorica独自の価値**: 既存リソースでは不足しており、Isidoricaが作る意義がある。
3.  **Laboratoryの実証**: インタラクティブな学習体験（Laboratory）の効果が最大限発揮される分野を選ぶ。
4.  **実行可能性**: Core Maintainer（技術者）が高品質なコンテンツを書けるか。

### 1.2 選定基準 (Selection Criteria)

| 基準 | 説明 | 重み |
| :--- | :--- | :--- |
| **公共知としての必要性** | すべての市民に必要な知識か。民主主義社会の基盤となるか。 | 高 |
| **既存リソースの不足** | 質の高い「教科書」が既に存在するか。Isidoricaが埋めるべき空白があるか。 | 高 |
| **Laboratory親和性** | シミュレーション、可視化が学習効果を劇的に高めるか | 高 |
| **情報の恒久性** | 10年後も価値が変わらない知識か | 中 |
| **実行可能性** | Core Maintainerが高品質なコンテンツを執筆可能か | 高 |

## 2. 初期領域 (First Domain)

Phase 5（Launch）において、リソースを集中投下する領域を**「認知バイアス」**に決定する。

### 2.1 選定理由

1.  **公共知としての優先度 (◎)**
    *   意思決定、メディア消費、民主主義参加のすべてに関連する。
    *   「バイアスを知っている」と「体験した」では学習効果が全く異なる。
    *   CORE_FOUNDATIONS.md §1の「教育的公正（Equity）」に最も合致する。

2.  **Isidorica独自の価値 (◎)**
    *   入門書（「ファスト&スロー」等）は存在するが、Laboratoryで「バイアスを体験する」プラットフォームは存在しない。
    *   **読むだけでは得られない「体験的理解」がIsidoricaの真価**を発揮する領域。
    *   認知バイアスの理解は、後続の統計リテラシー・科学リテラシーの前提となる。

3.  **Laboratory適合性 (◎)**
    *   確証バイアス、アンカリング、基準率の無視などをシミュレーションで体験可能。
    *   **「自分がバイアスに陥った」瞬間を経験させる**ことが最大の学習効果をもたらす。
    *   先行知識活性化原則（A-1）を最も純粋に実現できる領域。

4.  **実行可能性 (◯)**
    *   Core Maintainerの心理学・認知科学への知見は十分。
    *   研究文献が豊富であり、Tier 1-2出典に基づくコンテンツ作成が可能。

### 2.2 学習パスの構成

以下の3つの「学習パス（Learning Path）」を整備する。

| Path | テーマ | 概要 |
| :--- | :--- | :--- |
| **Path A** | 判断のショートカット | ヒューリスティクスと系統的誤り |
| **Path B** | 見たいものを見る | 信念と情報処理の歪み |
| **Path C** | 数字と確率の罠 | 確率・リスク認知の誤り |

> 各Pathの詳細な記事リスト・Laboratory構成は、Phase 4完了後に策定する。

> **統計リテラシーへの接続**: Path Cは統計リテラシー（Phase 6）への自然な導入となる。

> **教育設計方針**: 認知バイアス領域の教育設計は [LEARNING_DESIGN.md](../shared/LEARNING_DESIGN.md) の一般原則（特に§1「予測誤差による学習駆動」）に従う。「体験ファースト設計」として、まずLaboratoryでバイアスに「陥る」体験をさせ、その後に解説を提供する。

### 2.3 検討した他候補

以下の領域も検討したが、現段階ではFirst Domainとして選定しなかった。

| 候補 | 見送り理由 |
| :--- | :--- |
| **統計リテラシー** | 入門書・読み物が既に豊富。Isidorica独自の価値が相対的に低い。Phase 6で扱う |
| **論理学・批判的思考** | Laboratory親和性が認知バイアスより低い |
| **経済学基礎** | 公共知としての優先度は高いが、価値観が分かれやすく初期には不適 |

## 3. 拡張ロードマップ (Expansion Roadmap)

### Phase 5 (Launch): 認知バイアス

**認知バイアス領域**にリソースを集中し、「自分の判断の限界を知る市民」を育てる教科書を確立する。

*   **目標**:
    *   主要記事数: 20〜30記事（各Path 7〜10記事程度）
    *   Laboratory実装: 10個以上
    *   **品質基準**: 全記事がTier 1-3出典に基づき、体験型Laboratoryを含むこと。

### Phase 6 (Growth): 統計リテラシー

認知バイアスで「なぜ私たちは間違えるか」を学んだ読者に、**データを正しく読む方法**を提供する。

*   **接続**: Phase 5の「数字と確率の罠」から自然に移行
*   **目標**: データリテラシー、確率的思考、因果推論の基礎

### Phase 7 (Maturity): 科学リテラシー + 他分野展開

統計リテラシーで土台を固めた上で、**科学リテラシー**および**他分野**へ展開する。

*   **科学リテラシー**: エビデンス評価、査読プロセス、疑似科学の識別
*   **他分野候補**: コンピュータサイエンス、経済学基礎、論理学・批判的思考、数学基礎
*   **アクション**: 分野別エディター（Curator）の募集開始、コミュニティ主導の領域拡大

## 4. 数量目標 (Quantitative Goals)

### Phase 5 (Launch) 目標

| 指標 | 目標値 | 備考 |
| :--- | :--- | :--- |
| **公開記事数** | 20〜30記事 | 3つのLearning Path × 7-10記事 |
| **Laboratory数** | 10個以上 | 各Pathに3-4個の体験型Laboratory |
| **カバー領域** | 認知バイアス基礎 | 「自分の判断の限界を知る」入門 |

### 成功基準 (Success Criteria)

成功を**「公開時」**と**「公開6ヶ月後」**の2段階で評価する。

#### 公開時（Phase 5完了）

「Isidoricaが約束したものを、約束した品質で出せたか」を評価。

| カテゴリ | 指標 | 目標値 | 測定方法 |
| :--- | :--- | :--- | :--- |
| **量** | 公開記事数 | 20記事以上 | ファイル数カウント |
| **量** | Laboratory実装数 | 10個以上 | コンポーネント数カウント |
| **質** | 出典Tier準拠率 | 100% | 全記事がTier 1-3出典のみ |
| **質** | アクセシビリティ | Lighthouse 100 | CI自動テスト |
| **質** | パフォーマンス | Lighthouse 90+ | CI自動テスト |
| **プロセス** | 執筆ワークフロー完走 | 3パス全て公開 | 目視確認 |
| **プロセス** | バーンアウトなし | Core Maintainerが継続意欲あり | 自己申告 |

#### 公開6ヶ月後

「読者に価値を届けられているか」を評価。

| カテゴリ | 指標 | 目標値 | 測定方法 |
| :--- | :--- | :--- | :--- |
| **リーチ** | 月間ページビュー | 1,000 PV以上 | Cloudflare Web Analytics |
| **エンゲージメント** | Laboratory操作率 | 30%以上 | Cloudflare Web Analytics カスタムイベント |
| **信頼性** | 誤り報告数 | 重大な誤り0件 | GitHub Issue |
| **評判** | ポジティブな言及 | 1件以上（SNS/ブログ） | 外部監視 |
| **拡張準備** | Phase 6計画策定 | 完了 | ドキュメント存在 |

> **KPI_FRAMEWORK.mdとの接続**: 北極星指標「知識の還元数」は、「Laboratory操作率」を代理指標（Proxy）として評価する。

## 5. 実行計画 (Execution Plan)

### 5.1 執筆体制

*   **担当**: Core Maintainer
*   **方針**: ドッグフーディングを兼ねて、プラットフォーム開発者自身が最初のコンテンツを執筆する。これにより、執筆者体験（DX）へのフィードバックを最速で行う。

### 5.2 詳細記事リストの策定

*   **時期**: Phase 4（実装フェーズ）完了直後
*   **内容**: 各Learning Pathの具体的な目次案（Table of Contents）を策定する。
*   **理由**: 実装された機能（Markdown拡張、Laboratory機能）の全容が確定した状態で、それに最適化された構成を組むため。
