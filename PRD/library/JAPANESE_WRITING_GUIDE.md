# 日本語文章スタイルガイド

Isidorica記事執筆時の日本語表記ルールを定めたものです。
JTF日本語標準スタイルガイド、JTCA日本語スタイルガイド、記者ハンドブック等を参考に策定しています。

> **自動チェック**: 本ガイドのルールの多くは **textlint** で自動チェックされます。`pnpm textlint articles/` を実行してください。
> **クイックリファレンス**: 必須ルールのみ確認したい場合は [WRITER_HANDBOOK.md](./WRITER_HANDBOOK.md) を参照。

## 1. 文体・口調

### 1.1 基本の文体

記事の性質に応じて文体を使い分けます。**同一記事内での混在は避けてください**。

| 文体 | 使用場面 | 例 |
|------|----------|-----|
| **です・ます調（敬体）** | チュートリアル、入門記事、ドキュメント | 「〜です」「〜します」「〜ください」 |
| **だ・である調（常体）** | 学術的記事、論文調の解説、リファレンス | 「〜である」「〜する」「〜だ」 |

**フロントマターでの指定（推奨）**:
```yaml
---
title: 記事タイトル
style: formal  # formal（である調）/ polite（ですます調）
---
```

### 1.2 読者への呼びかけ

- 読者への直接的な呼びかけ（「あなた」「皆さん」）は**使用しません**
- 主語が必要な場合は「読者」「利用者」「開発者」などの一般名詞を使用

```markdown
<!-- ❌ 避ける -->
あなたはこのコードを実行できます。

<!-- ✅ 推奨 -->
このコードを実行できる。
開発者はこのコードを実行できる。
```

### 1.3 一人称

- 一人称（「私」「筆者」「著者」）は**原則として使用しません**
- 客観的な記述を心がけ、主語を省略するか、受動態・能動態で表現

```markdown
<!-- ❌ 避ける -->
私はこの方法を推奨します。

<!-- ✅ 推奨 -->
この方法が推奨される。
この方法を推奨する。
```

### 1.4 文末表現の統一

- 同一段落・同一セクション内で文末表現を統一
- 「〜です」「〜ます」「〜である」「〜だ」の混在を避ける

## 2. 句読点・記号

### 2.1 句点・読点

**基本ルール**: 句点は「。」、読点は「、」を使用

| 記号 | 基本 | 特殊（学術分野） |
|------|:----:|:----------------:|
| 句点 | 。 | .（ピリオド） |
| 読点 | 、 | ,（カンマ） |

**例外**:
- 計算機科学、数学、物理学などの学術的記事では「.」「,」も許容
- 分野ごとに統一し、**同一記事内での混在は避ける**

### 2.2 括弧

| 括弧の種類 | 用途 | 例 |
|------------|------|-----|
| **「」（かぎ括弧）** | 引用、用語の強調、会話 | 「サーバサイドレンダリング」 |
| **『』（二重かぎ括弧）** | 書名、作品名、「」内での引用 | 『プログラミング言語C』 |
| **（）（全角丸括弧）** | 補足説明、言い換え | 関数（function） |
| **〈〉（山括弧）** | 強調、特殊な用語 | 使用頻度は低い |

**英数字との組み合わせ**:
- 括弧内が英数字のみの場合は半角 `()` も許容: `(1)`, `(a)`
- 日本語を含む場合は全角を使用: （注意）

### 2.3 引用符

| 言語 | 引用符 | 例 |
|------|--------|-----|
| 日本語 | 「」『』 | 「引用文」 |
| 英語 | "" '' | "quoted text" |

- 英文を記述する際は英語の引用符ルールに従う
- 日本語文中での英単語の強調にはバッククォート `` ` `` を使用

### 2.4 中黒（・）

| 用途 | 例 |
|------|-----|
| **外国人名の区切り** | ドナルド・クヌース |
| **並列表現** | HTML・CSS・JavaScript |
| **カタカナ複合語の区切り** | プログラミング・スタイル |

**外国人名の表記方針**:
- 原則として**原語表記**を推奨（表記ゆれを避けるため）
- カタカナ表記が必要な場合は中黒で区切る
- 表記ゆれが問題になる場合は原語を併記: ワーグナー（Richard Wagner）

**並列表現**:
- 2〜3項目の短い並列: 中黒を使用（HTML・CSS・JavaScript）
- 4項目以上または長い項目: 読点を使用（HTML、CSS、JavaScript、TypeScript）

### 2.5 コロン・セミコロン

| 記号 | 表記 | 用途 |
|------|------|------|
| コロン | ：（全角） | 説明の導入、対応関係 |
| セミコロン | ；（全角） | 使用頻度は低い |

- 英数字の直後は半角 `:` も許容: `例: sample`
- リスト項目の導入には全角を使用: 「主な機能：」

### 2.6 感嘆符・疑問符

| 記号 | 使用可否 | 注意点 |
|------|:--------:|--------|
| ！（感嘆符） | △ 控えめに | フォーマルな記事では使用を避ける |
| ？（疑問符） | ○ 許容 | 疑問文の文末に使用可 |

- 学術的記事では使用を控える
- チュートリアル等では読みやすさのため許容

### 2.7 三点リーダー

**統一ルール**: `…`（全角三点リーダー）を使用

```markdown
<!-- ❌ 避ける -->
続きは...  （半角ピリオド3つ）
続きは・・・（中黒3つ）

<!-- ✅ 推奨 -->
続きは…（全角三点リーダー）
```

- 三点リーダーは**2つ続けて**使用するのが正式: `……`
- ただし、本プロジェクトでは1つでも許容

## 3. 表記の統一

### 3.1 漢字とひらがなのバランス

**原則**: 漢字30%、ひらがな70%を目安に、読みやすさを重視

#### ひらがなで書く語（ひらく）

| 分類 | ひらがな表記 | 漢字表記（使用しない） |
|------|-------------|----------------------|
| **形式名詞** | こと、もの、とき、ため、ところ、ほう、わけ | 事、物、時、為、所、方、訳 |
| **補助動詞** | 〜ていく、〜てくる、〜てみる、〜ておく、〜てある | 〜て行く、〜て来る、〜て見る、〜て置く、〜て有る |
| **接続詞** | しかし、したがって、および、または、ならびに、ただし | 然し、従って、及び、又は、並びに、但し |
| **副詞** | すでに、まったく、もっとも、ほとんど、おそらく | 既に、全く、最も、殆ど、恐らく |
| **動詞（一部）** | できる、いる、ある、いう、わかる | 出来る、居る、有る/在る、言う、分かる |

#### 漢字で書く語（とじる）

- **実質的な意味を持つ名詞・動詞**: 具体的な意味がある場合は漢字
- **熟語**: 複数の漢字で構成される語

```markdown
<!-- ひらがな（形式名詞） -->
設定を変更することができる。← こと
問題があるときは報告する。← とき

<!-- 漢字（実質的意味） -->
大切な事を忘れていた。← 具体的な事柄
その時、彼は気づいた。← 特定の瞬間
```

### 3.2 送り仮名

**原則**: 内閣告示「送り仮名の付け方」本則に準拠

| ✅ 推奨 | ❌ 非推奨 |
|---------|----------|
| 行う | 行なう |
| 表す | 表わす |
| 現れる | 現われる |
| 断る | 断わる |

### 3.3 外来語のカタカナ表記

**原則**: 語尾の長音（ー）は省略しない

| ✅ 推奨 | ❌ 非推奨 |
|---------|----------|
| サーバー | サーバ |
| コンピューター | コンピュータ |
| ユーザー | ユーザ |
| ブラウザー | ブラウザ |
| フォルダー | フォルダ |

**例外**: 技術用語として定着している表記は許容
- JIS規格や業界慣習に従う場合

### 3.4 数字の表記

**基本ルール**: 横書きでは**半角アラビア数字**を使用

| 場面 | 表記 | 例 |
|------|------|-----|
| **数量・数値** | 半角アラビア数字 | 3個、100人、2024年 |
| **日付・時刻** | 半角アラビア数字 | 2024年12月7日、14:30 |
| **序数** | 半角アラビア数字 | 第1章、1番目 |
| **慣用表現・成句** | 漢数字 | 一石二鳥、四苦八苦、一人ひとり |
| **概数** | 漢数字 | 数十人、数百件 |
| **固有名詞** | 元の表記に従う | 六本木、七夕、三菱 |

**桁区切り**:
- 4桁以上の数字にはカンマを使用: `1,000`、`1,000,000`
- 小数点にはピリオドを使用: `3.14`

### 3.5 英数字の全角・半角

| 種類 | 表記 | 例 |
|------|------|-----|
| **アルファベット** | 半角 | JavaScript, HTML, CSS |
| **数字** | 半角 | 123, 2024 |
| **記号（英文中）** | 半角 | `function()`, `array[0]` |
| **記号（和文中）** | 全角 | 「：」「；」「（）」 |

**例外**:
- 固有名詞のケースは保持: `JavaScript`（`javascript`ではない）
- 商標等の表記は公式に従う: `TypeScript`、`GitHub`

## 4. 文の構造

### 4.1 一文の長さ

**目安**: 60文字以内を推奨、最大でも80文字程度

- 長い文は複数の文に分割する
- 読点が3つ以上ある場合は分割を検討

```markdown
<!-- ❌ 長すぎる -->
このフレームワークは、サーバーサイドレンダリングと静的サイト生成の両方に対応しており、開発者体験を重視した設計になっているため、多くのプロジェクトで採用されている。

<!-- ✅ 分割する -->
このフレームワークは、サーバーサイドレンダリングと静的サイト生成の両方に対応している。
開発者体験を重視した設計のため、多くのプロジェクトで採用されている。
```

### 4.2 主語の明示

- 主語が明確な場合は省略可
- 主語が曖昧になる場合、または誤解を招く場合は明示

### 4.3 能動態と受動態

**原則**: 能動態を優先

```markdown
<!-- 受動態（冗長になりやすい） -->
このメソッドは開発者によって呼び出される。

<!-- 能動態（簡潔） -->
開発者がこのメソッドを呼び出す。
```

**受動態を使う場面**:
- 動作主を明示する必要がない場合
- 動作の対象を強調したい場合

### 4.4 二重否定の回避

```markdown
<!-- ❌ 二重否定 -->
できないわけではない
必要がないとは言えない

<!-- ✅ 肯定文に -->
できる
必要がある（かもしれない）
```

### 4.5 冗長表現の回避

| ❌ 冗長 | ✅ 簡潔 |
|---------|---------|
| することができる | できる |
| することが可能である | できる |
| という | （削除または「の」に置換） |
| において | で |
| に関して | について / の |
| という形で | で |
| といったような | のような / など |

## 5. 専門用語・技術用語

### 5.1 表記の統一

- 同一記事内で表記を統一する
- 用語集を作成し、プロジェクト全体で統一

| 例 | 統一表記 |
|----|----------|
| コンポーネント / Component | 文脈に応じて選択し統一 |
| レンダリング / 描画 | 技術文書では「レンダリング」推奨 |
| インターフェース / Interface | カタカナ推奨 |

### 5.2 初出時の説明

- 略語は初出時に正式名称を併記
- 専門用語は初出時に簡潔な説明を添える

```markdown
SSR（Server-Side Rendering、サーバーサイドレンダリング）は…

クロージャ（関数とその関数が参照する変数の組み合わせ）を使用して…
```

### 5.3 固有名詞の表記

- 公式の表記を尊重する
- 大文字・小文字、ハイフンの有無を正確に

| ✅ 正確 | ❌ 不正確 |
|---------|----------|
| JavaScript | Javascript, JAVASCRIPT |
| TypeScript | Typescript, typescript |
| Next.js | NextJS, Nextjs, next.js |
| GitHub | Github, github |
| macOS | MacOS, MACOS |

## 6. 見出し・タイトル

### 6.1 見出しの文体

**原則**: 体言止め（名詞で終わる）を推奨

```markdown
<!-- ✅ 体言止め -->
## 環境構築
### 依存パッケージのインストール

<!-- △ 許容（状況に応じて） -->
## 環境を構築する
### 依存パッケージをインストールする
```

### 6.2 見出しの長さ

- **目安**: 30文字以内
- 長くなる場合はサブ見出しで補足

### 6.3 句読点の使用

- 見出し内に**句点（。）は使用しない**
- 読点（、）は必要に応じて使用可

## 7. 引用・参照

### 7.1 引用の書式

**短い引用**: かぎ括弧で囲む
```markdown
公式ドキュメントには「サーバーコンポーネントはデフォルトで有効」と記載されている。
```

**長い引用**: 引用ブロックを使用
```markdown
> Reactは、ユーザーインターフェースを構築するためのJavaScriptライブラリです。
> 宣言的なViewを採用することで、コードの予測可能性と保守性を高めます。
```

### 7.2 出典の記載

- 引用・参照には必ず出典を明記
- 書式は統一する

**推奨書式**:
```markdown
参考: [React公式ドキュメント](https://react.dev/)
出典: 『プログラミング言語C』（B.W.カーニハン、D.M.リッチー著）
```

## 8. アクセシビリティ・読みやすさ

### 8.1 難読漢字へのルビ

- 常用漢字外の漢字や難読語にはルビを振る
- `<rp>` タグを含めたフォールバック記述を推奨

**推奨書式（ただし、記事では自動でこの形に変換されます）**:
```html
<ruby>彙<rp>（</rp><rt>い</rt><rp>）</rp></ruby>
```

### 8.2 段落の長さ

- 1段落は3〜5文程度を目安
- 長い説明は適切に段落を分ける

### 8.3 箇条書きの活用

- 3項目以上の列挙は箇条書きに
- 手順の説明は番号付きリストを使用

## 9. コード・技術的内容との混在

### 9.1 コード前後の文章

- コードブロックの前に簡潔な導入文を記述
- 「以下のコードを見てください」のような冗長な表現は避ける

```markdown
<!-- ❌ 冗長 -->
以下のコードを見てください。このコードは…

<!-- ✅ 簡潔 -->
環境変数を設定する:
```

### 9.2 コード内コメントの言語

- **日本語記事内のコード**: 日本語コメント推奨
- コメントはプロジェクトの言語設定に準拠

### 9.3 変数名・関数名の言及

- 本文中でコード要素に言及する際はバッククォートで囲む
- 例: `useState`フック、`handleClick`関数

## 10. 禁止事項・避けるべき表現

### 10.1 差別的表現

- ジェンダー、年齢、国籍、障害などに関する差別的表現を避ける
- 包括的な言葉遣いを心がける

### 10.2 曖昧な表現

| ❌ 避ける | ✅ 推奨 |
|----------|---------|
| 〜とか | 〜や、〜など |
| 〜的な | 〜のような、〜に関する |
| いろいろな | さまざまな、多様な |
| すごく | 非常に、きわめて |

### 10.3 過度な敬語・謙譲語

| ❌ 過度 | ✅ 適切 |
|---------|---------|
| 〜させていただく | 〜する |
| ご確認いただければ幸いです | 確認してください |
| 〜してくださいますよう | 〜してください |

### 10.4 断定の回避しすぎ

- 根拠のある内容は断定して記述
- 過度な「かもしれません」「と思われます」は避ける
