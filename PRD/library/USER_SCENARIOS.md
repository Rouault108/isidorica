# ユーザーシナリオ (User Scenarios)

本ドキュメントでは、Isidoricaの各ユーザータイプごとのユースケースとワークフローを定義する。

**ステータス**: ✅ 完了

---

## 1. 読者 (Reader)

### 1.1 読者モードの概要

Isidoricaでは、読者の習熟度とニーズに応じて3つのモードを提供する。詳細は [READER_ADAPTATION_DESIGN.md](../engine/READER_ADAPTATION_DESIGN.md) を参照。

| モード | 対象 | 提供する体験 |
| :--- | :--- | :--- |
| **Explorer** | 概念を初めて知る読者 | 入り口としての圧縮版。Learnerへの誘引を含む |
| **Learner** | 体系的に学びたい読者（主対象） | Phase 1-6の完全な学習体験 |
| **Scholar** | 深い議論を求める読者 | 出典詳細、未解決問題、Agoraへの導線 |

**理論的根拠**: 
- 専門性逆転効果（LEARNING_FOUNDATIONS.md 5.2.4節）への対応
- 認知的倹約家（USER_BEHAVIOR_MODEL.md 6.1節）への配慮

| UC-R2 | 検索 | キーワードで記事を検索 | - |

### 1.3 ワークフロー

#### UC-R1 & R3: 記事の閲覧と出典確認（ディープ・リーディング）

信頼性を確認しながら、思考の流れを中断せずに読み進めるフロー。

```
読者: 記事ページにアクセス
    ↓
読者: 本文を読む
    ↓
読者: 文中の引用番号 [1] にカーソルを合わせる（モバイルではタップ）
    ↓
システム: 脚注ポップオーバーを表示
  - 出典書誌情報（著者、年、タイトル、ページ）引用句
  - 外部データベース（Google Books/CiNii）へのリンク
  - 「図書館で探す」（カーリル等へのリンク - 将来機能）
    ↓
【検証する場合】
読者: リンクをクリックして外部ソースを確認
    ↓
【読み進める場合】
読者: カーソルを外して本文に戻る（0.1秒で復帰）
    ↓
読者: 読了後、記事末尾の「参考文献一覧（Bibliography）」で全体を俯瞰
```

*   **ポイント**: 「認知的倹約家」である読者に負担をかけないよう、ページ遷移なしで信頼性を担保する。

#### UC-R4: Laboratory体験（能動的学習）

「読む」だけでなく「触って仮説を検証する」ことで、概念の身体的獲得（Enactivism）を促す。

```
読者: 記事内の「モンテカルロ法」の説明を読む
    ↓
読者: Laboratoryブロック（Python/QuickJS）を見る
    ↓
システム: 初期状態のシミュレーション結果（静的プレビュー）を表示
    ↓
読者: 「実行」ボタンをクリック
    ↓
システム: WASMランタイムをバックグラウンドでロード & 実行
    ↓
システム: 動的なグラフが描画される
    ↓
読者: コード内のパラメータ（試行回数 N=100 → N=10000）を書き換える
    ↓
システム: 再実行・再描画（精度が劇的に向上する様を目撃）
    ↓
読者: 「予測誤差（驚き）」を感じ、概念を直感的に理解する
```

*   **ポイント**: エラーさえも「発見」として扱う（失敗の正規化）。

#### UC-R2: 探索と発見（セレンディピティ）

目的の記事だけでなく、関連する体系的な知識へと導く。

```
読者: 検索バー（Cmd+K）を開き "Nash" と入力
    ↓
システム: 候補を表示
  - 記事: ナッシュ均衡
  - 人物: ジョン・ナッシュ
  - Topic: ゲーム理論
    ↓
読者: Topic「ゲーム理論」を選択
    ↓
システム: Topicページを表示
  - 構造図: ゲーム理論の木構造（協力/非協力、静的/動的）
  - Learning Path: 「初学者のためのゲーム理論パス」
    ↓
読者: パスに従って最初の記事「囚人のジレンマ」へ移動
```

#### UC-R6: 言語切り替えと原典確認

```
読者（母語: 日本語）: 英語の記事「Categorical Imperative（定言命法）」にアクセス
    ↓
読者: 内容が難しいと感じ、言語スイッチャーで「日本語」を選択
    ↓
システム: 日本語版記事を表示
  - 出典リストも自動的に日本語翻訳版（カント『道徳形而上学の基礎づけ』）を優先表示
    ↓
読者: 翻訳のニュアンスを確認するため、特定の引用の「原典ページ数」を確認
    ↓
システム: ポップオーバーに `カント 1960, p.45 (原著 p.42)` と併記
```

### 1.4 モード別ワークフロー

#### UC-R10: Explorerモードでの学習体験

**目的**: 概念の概要を把握し、興味を持った場合はLearnerへ移行する。
**理論的根拠**: READER_ADAPTATION_DESIGN.md 4.1節、USER_BEHAVIOR_MODEL.md 6.1節（認知的倹約家）

```
読者: 検索エンジンから記事にアクセス
    ↓
システム: デフォルトでLearnerモードを表示
    ↓
読者: 「内容が多い」と感じ、モードトグルで「Explorer」を選択
    ↓
システム: Explorerモードに切り替え
  - Phase 1-2の圧縮版を表示
  - Laboratoryはプレビュー表示（🎮「これを体験できます」）
  - Phase 3-6の魅力を提示（「驚きの発見があります」）
    ↓
読者: 圧縮版を読み、概念の概要を把握
    ↓
システム: 記事末尾に理解度チェックQuiz（任意）を表示
    ↓
【Quizに挑戦する場合】
  読者: Quizに回答
    ↓
  【正解の場合】
    システム: 「素晴らしい直感です！Learnerでさらに深く学べます」
    ↓
  【不正解の場合】
    システム: 「Learnerモードで学ぶと解けるようになります」（励ましトーン）
    ↓
【Learnerへ移行する場合】
  読者: モードトグルで「Learner」を選択
    ↓
  システム: Learnerモードに切り替え、Phase 1から開始
    ↓
【離脱する場合】
  読者: 「概要は把握した」と判断し、離脱
  ※ 流暢性の錯覚のリスクあり。Quizによる較正が有効。
```

#### UC-R11: Learnerモードでの学習体験（Phase 1-6）

**目的**: Phase 1-6を通じて概念を体系的に理解し、転移可能な知識を獲得する。
**理論的根拠**: LEARNING_FOUNDATIONS.md 4章、USER_BEHAVIOR_MODEL.md 11章

```
読者: Learnerモードで記事にアクセス
    ↓
【Phase 1: 予測誤差の喚起】
  システム: 読者の素朴な予測を引き出す問いを提示
  読者: 自分の予測を考える
    ↓
  システム: Laboratoryで予測を覆す結果を表示
  読者: 「予測誤差（驚き）」を体験 → 学習動機が喚起
    ↓
【Phase 2: 定義】
  システム: 概念の正式な定義を提示
  ※ 離脱リスク高（抽象的で退屈）
  対策: Phase 1の「驚き」を再利用した具体例を提示
    ↓
【Phase 3: 機序】
  システム: 概念の動作原理を段階的に説明
  ※ 離脱リスク高（複雑で混乱）
  対策: 足場かけ、コンテキスト依存ヒント
    ↓
  読者: Laboratoryでパラメータを操作し、機序を体験
    ↓
【Phase 4: 確認問題】
  システム: Quizを提示
  読者: 回答
    ↓
  【失敗の場合】
    システム: 「発見しました！この部分を復習しましょう」（失敗の正規化）
    ↓
  【成功の場合】
    システム: 「正解です！次のPhaseに進みましょう」
    ↓
【Phase 5: 文脈・歴史】
  システム: 「なぜこの概念が重要か」を冒頭で明示
  読者: 概念の歴史的・社会的意義を理解
    ↓
【Phase 6: 応用・転移】
  システム: 異なる文脈への転移例を提示
  読者: Laboratoryで新しい問題を解く
    ↓
システム: 次の学習パスへの導線、Agoraへの招待を表示
読者: 達成感を得て、次のステップを検討
```

#### UC-R12 & R14: Scholarモードでの学習体験

**目的**: 深い議論にアクセスし、Agoraでの共同探究に参加する。
**理論的根拠**: USER_BEHAVIOR_MODEL.md 7.6節（興味の発達モデル）、READER_ADAPTATION_DESIGN.md 4.3節

```
読者: Learnerモードで記事を学習完了
    ↓
読者: さらに深い議論を求め、モードトグルで「Scholar」を選択
    ↓
システム: Scholarモードに切り替え
  - 出典詳細を強調表示（原典ページ番号、翻訳との対照）
  - 「未解決問題」セクションを表示
  - 「学術的論争」セクションを表示
  - Agoraへの導線を表示
    ↓
読者: 未解決問題を確認し、知的好奇心を刺激される
    ↓
読者: 「この概念についてAgoraで議論する」ボタンをクリック
    ↓
システム: Agoraの該当Discussion（または新規作成画面）へ遷移
    ↓
読者: 他の読者と議論に参加（状況興味→個人興味への発達）
```

### 1.5 学習パスシナリオ

#### UC-R7: Learning Pathを辿る（将来機能）

**目的**: 複数の記事を体系的に学習し、特定の分野を習得する。
**理論的根拠**: USER_BEHAVIOR_MODEL.md 11.4節（変奏曲的演習）、11.5節（未来への準備）

```
読者: 「ゲーム理論入門」Learning Pathページにアクセス
    ↓
システム: パス全体の構造を表示
  - 記事1: 囚人のジレンマ
  - 記事2: ナッシュ均衡
  - 記事3: 混合戦略
  - 記事4: 繰り返しゲーム
  - ...
    ↓
読者: 「開始」ボタンをクリック
    ↓
システム: 記事1（囚人のジレンマ）をLearnerモードで表示
  - ナビゲーションに「Learning Path: 1/10」を表示
  - 「次の記事へ」ボタンを追加
    ↓
読者: Phase 1-6を学習
    ↓
読者: 「次の記事へ」をクリック
    ↓
システム: 記事2（ナッシュ均衡）を表示
  - 前の記事との関連性を冒頭で明示
    ↓
【パス完遂時】
  システム: 「パス完了！」メッセージと達成感を演出
  システム: 関連する上級パスへの導線を表示
```

### 1.6 離脱防止シナリオ

各Phaseでの離脱リスクと、システムの介入。

| Phase | 離脱リスク | 典型的なユーザー心理 | システムの介入 |
| :--- | :--- | :--- | :--- |
| Phase 1 → 2 | 中 | 「驚きは面白かったが、定義は退屈そう」 | Phase 1の具体例を使った導入 |
| Phase 2 → 3 | 高 | 「定義は分かったが、詳細は難しそう」 | 足場かけ、段階的な説明 |
| Phase 3 → 4 | 中 | 「理解したつもりだが、Quizは怖い」 | 失敗の正規化メッセージ |
| Phase 4 → 5 | 低 | 「Quizに合格したから十分」 | 「なぜ重要か」の明示 |
| Phase 5 → 6 | 低 | 「歴史は興味ない」 | 価値の明示（「これができるようになります」） |
| Phase 6完了後 | - | 達成感 | 次のパス、Agoraへの導線 |

## 2. 執筆者 (Author)

- **執筆者 (Author)**: 記事の執筆・修正を行うユーザー。GitHubアカウントを持つ「登録執筆者」と、持たずにAgora経由で提案する「匿名ゲスト（Future）」の2種類が存在する。

### 2.1 ユースケース

| ID | ユースケース | 説明 |
| :--- | :--- | :--- |
| UC-A1 | 既存出典を使用 | bibliographyに既にある出典を参照 |
| UC-A2 | 新規出典を追加（書籍・論文） | ISBN/DOI/タイトル・著者で検索、外部APIから取得 |
| UC-A3 | 新規手動入力出典を追加 | 外部APIで取得できない資料（Webページ、法令、判例等）を手動登録 |
| UC-A4 | OSSコードを引用 | インラインで出典記載（コメント形式） |
| UC-A5 | 直接Markdown編集 | エディタ統合なしでMarkdownを編集（Web検索ページで出典ID取得） |
| UC-A6 | 新規記事を作成 | テンプレートから新しい記事ファイルを生成 |
| UC-A7 | 画像を追加 | 許諾取得し、PRに証拠を添付 |

### 2.2 ワークフロー

#### UC-A1: 既存出典を使用

```
執筆者: 執筆支援ツールで出典を検索
    ↓
ツール: 並列検索
  - bibliography.json
  - 外部API（Google Books, CiNii, CrossRef等）
    ↓
ツール: 結果をマージして表示
  - 既存出典（優先表示）
  - 外部API結果（「新規追加」マーク付き）
    ↓
執筆者: 既存出典を選択
    ↓
ツール: [@source-id] をクリップボードにコピー
    ↓
執筆者: Markdownに貼り付け
```

#### UC-A2: 新規出典を追加（書籍・論文）

```
執筆者: 執筆支援ツールで出典を検索
  - タイトル・著者名で検索（汎用）
  - ISBNで検索（書籍）
  - DOIで検索（論文）
    ↓
ツール: 並列検索
  - bibliography.json
  - 外部API（Google Books, CiNii, CrossRef等）
    ↓
ツール: 結果を表示
  - 既存出典: 該当なし（「データベースに存在しません」）
  - 外部API結果（「新規追加」マーク付き）
    ↓
執筆者: 外部API結果を選択
    ↓
ツール: 
  1. 出典IDを生成（衝突チェック）
  2. [@source-id] をクリップボードにコピー
  3. 新規出典データをローカルにキャッシュ
    ↓
執筆者: Markdownに貼り付け
    ↓
PR作成
    ↓
CI（PR作成時）: 
  1. 新規出典を検出
  2. PRコメントにレポート（レビュワーに注意を促すため）
  3. 衝突チェック（早期フィードバック、警告のみ）
    ↓
レビュワー: 出典確認・承認
    ↓
CI（マージ前）:
  1. 最終衝突チェック（競合状態への対応）
  2. 衝突があればマージをブロック
    ↓
マージ: bibliography.json に追加（status: curated）
```

*   **検索オプション**: タイトル・著者（汎用）、ISBN（書籍）、DOI（論文）の3種類を提供。
*   **外部API**: Google Books（ISBN）、CrossRef（DOI）、CiNii（タイトル/DOI）を併用。

#### UC-A3: 新規手動入力出典を追加

外部API（Google Books, CiNii, CrossRef等）で取得できない資料を手動入力フォームで登録する場合のワークフロー。

| **判例** | 事件名, 裁判所, 判決日 | 事件番号, 出典 |
| **個人所蔵・アーカイブ資料** | 資料名, 所蔵者/所蔵機関 | 参照番号, 年代, 所在地 |
| **データセット** | タイトル, 提供者, 公開年 | DOI/URL |

##### ワークフロー

```

    ↓
ツール: 資料タイプ選択画面を表示

  - 法令
  - 判例
  - 個人所蔵・アーカイブ資料
  - データセット
    ↓
執筆者: 資料タイプを選択
    ↓
ツール: 選択したタイプに応じた入力フォームを表示
    ↓
執筆者: 必須フィールドを入力
    ↓
ツール: 
  1. 出典IDを生成（衝突チェック）
  2. [@source-id] をクリップボードにコピー
  3. 新規出典データをローカルにキャッシュ
    ↓
執筆者: Markdownに貼り付け
    ↓
PR作成
    ↓
CI（PR作成時）: 
  1. 新規出典を検出
  2. PRコメントにレポート（レビュワーに注意を促すため）
  3. 衝突チェック（早期フィードバック、警告のみ）
    ↓
レビュワー: 出典確認・承認
    ↓
CI（マージ前）:
  1. 最終衝突チェック（競合状態への対応）
  2. 衝突があればマージをブロック
    ↓
マージ: bibliography.json に追加（status: curated）
```

*   **同じURLは同一出典**: Webページの場合、アクセス日が異なっても重複登録しない。
*   **詳細なフィールド定義**: [GOVERNANCE.md](./GOVERNANCE.md) セクション1.6、[FORMS_DESIGN.md](./FORMS_DESIGN.md) セクション2を参照。

#### UC-A4: OSSコードを引用

```
執筆者: コードスニペットをMarkdownに記載
    ↓
執筆者: コメントで出典を記載

```python
# Source: https://github.com/example/repo/blob/abc123/src/main.py#L10-L20
# License: MIT
def hello():
    print("Hello, World!")
```
    ↓
PR作成
    ↓
レビュワー: ライセンス確認
```

*   **注意**: OSSコードは `bibliography.json` に登録せず、インラインで管理。

#### UC-A5: 直接Markdown編集（エディタ統合なし）

Vim、Emacs、その他のエディタを使用し、執筆支援ツールの統合機能なしで直接Markdownを編集する場合のワークフロー。

##### 既存出典を使用する場合

```
執筆者: Web検索ページで出典を検索
    ↓
Web検索ページ: 
  - bibliography.json から既存出典を検索
  - 外部API（Google Books, CiNii, CrossRef等）も並列検索
    ↓
執筆者: 既存出典を選択 → IDをクリップボードにコピー
    ↓
執筆者: エディタに戻り、[@source-id] を貼り付け
    ↓
PR作成
    ↓
CI（PR作成時）: 
  1. 出典IDの存在確認
  2. 問題があればPRコメントで警告
    ↓
レビュワー: 確認・承認
    ↓
マージ
```

##### 新規出典を使用する場合

```
執筆者: Web検索ページで出典を検索
    ↓
Web検索ページ: 
  - bibliography.json: 該当なし
  - 外部API結果を表示（「新規追加」マーク付き）
    ↓
執筆者: 外部API結果を選択
    ↓
Web検索ページ: 
  1. 出典IDを自動生成（衝突チェック）
  2. IDをクリップボードにコピー
  3. 新規出典データをサーバーに一時保存（またはローカルストレージ）
    ↓
執筆者: エディタに戻り、[@source-id] を貼り付け
    ↓
PR作成
    ↓
CI（PR作成時）: 
  1. 新規出典を検出
  2. PRコメントにレポート
  3. 衝突チェック（早期フィードバック、警告のみ）
    ↓
レビュワー: 出典確認・承認
    ↓
CI（マージ前）:
  1. 最終衝突チェック（競合状態への対応）
  2. 衝突があればマージをブロック
    ↓
マージ: bibliography.json に追加（status: curated）
```

*   **ポイント**: Web検索ページは執筆支援ツールの検索機能と同一のUIを提供する。違いはエディタ統合の有無のみ。
*   **将来の拡張**: CLIツールやVSCode拡張が実装された場合も、同じAPIを使用するため、フローは同一。

#### UC-A6: 新規記事を作成

テンプレート機能を使用して新規記事を作成する。

##### 方法1: Web執筆支援ツール

```
執筆者: 「新規記事」ボタンをクリック
    ↓
ツール: カテゴリ選択画面を表示
  - 哲学
  - 経済学
  - プログラミング
  - 等
    ↓
執筆者: カテゴリを選択（例：哲学）
    ↓
ツール: フォーム形式でフロントマター入力画面を表示
  - タイトル（必須）
  - slug（自動生成または手動入力）
  - 説明（必須）
  - Topics（選択式）
  - 等
    ↓
執筆者: 各項目を入力
    ↓
ツール: テンプレートから記事構造を自動生成
  - フロントマター（ユーザー入力値を反映）
  - 記事本文（カテゴリに応じた見出しとプレースホルダー）
    ↓
執筆者: 生成された記事を編集・執筆開始
```

##### 方法2: CLIツール

```bash
# カテゴリ指定で生成
isidorica new article --category=philosophy --title="囚人のジレンマ"

# 対話形式で生成
isidorica new article --interactive
> カテゴリを選択してください: philosophy
> タイトルを入力してください: 囚人のジレンマ
> 説明を入力してください: ゲーム理論における...
> slug（自動生成: prisoner-dilemma）: [Enter]
```

生成されるファイル例:

```markdown
---
title: 囚人のジレンマ
description: ゲーム理論における...
slug: prisoner-dilemma
topics: []
original_language: ja
draft: true
---

## 概要

この概念の概要を記述してください。

## 歴史的背景

この概念の歴史的背景を記述してください。

## 主要な議論

主要な議論や論点を記述してください。

## 参考文献

<!-- 出典は自動生成されます -->
```

*   **テンプレート定義**: 詳細は [GOVERNANCE.md](./GOVERNANCE.md) セクション9を参照。

## 3. 翻訳者 (Translator)

### 3.1 ユースケース

| ID | ユースケース | 説明 |
| :--- | :--- | :--- |
| UC-T1 | 記事を翻訳 | オリジナル版を翻訳 |
| UC-T2 | ページ番号を追加 | 翻訳版の出典にページ番号を追加 |
| UC-T3 | 補足説明を追加 | 「この出典は日本語のみ」等の注記 |
| UC-T4 | 出典追加を提案 | オリジナル版への出典追加をDiscussionで提案 |

### 3.2 ワークフロー

#### UC-T1: 記事を翻訳

```
翻訳者: オリジナル版をコピー
    ↓
翻訳者: テキストを翻訳
    ↓
翻訳者: 出典IDはそのまま維持（変更禁止）
    ↓
PR作成
    ↓
CI: 出典リストがオリジナル版と一致しているか確認
    ↓
レビュワー: 言語レビュー
```

#### UC-T4: 出典追加を提案

翻訳者が独自に出典を追加・削除することは**禁止**。追加が必要な場合は、オリジナル版への提案を行う。

```
翻訳者: 翻訳作業中に出典追加の必要性を発見
    ↓
翻訳者: 該当記事のDiscussionページもしくはGitHub Discussionsにて直接相談
  - カテゴリ: 記事の出典
  - タイトル: [{slug}] 出典追加の提案: {出典名}
  - 内容: 追加理由、出典情報（ISBN/DOI/URL等）
    ↓
【承認された場合】
  オリジナル版の執筆者/レビュワー: オリジナル版にPRを作成
    ↓
  マージ後、翻訳版に自動反映
    ↓
【却下された場合】
  翻訳者: Discussionページの結論を確認し、対応を検討
```

*   **理由**: 翻訳版と原著版の出典リストの一貫性を保つため。
*   **Discussion連携**: GitHub Discussionsはリポジトリと連携しており、Issueと同様にトラッキング可能。詳細は [DISCUSSIONS_DESIGN.md](./DISCUSSIONS_DESIGN.md) を参照。

## 5. 報告者 (Reporter)

### 5.1 ユースケース

| ID | ユースケース | 説明 |
| :--- | :--- | :--- |
| UC-P1 | 誤りを報告 | 誤字脱字、出典の誤り |
| UC-P2 | 翻訳の問題を報告 | 翻訳品質の問題 |
| UC-P3 | リンク切れを報告 | URLが無効 |
| UC-P4 | 内容の問題を報告 | 中立性、正確性 |
| UC-P5 | 出典の削除を提案 | 不適切・重複・存在しない出典の削除 |

### 5.2 ワークフロー

#### 共通フロー（UC-P1〜P4）

```
報告者: 問題を発見
    ↓
【GitHub利用者】
  報告者: 直接GitHub Issueを作成
    ↓
【非GitHub利用者】
  報告者: 報告フォームから送信
    ↓
  ボット（isidorica-bot）: Issue自動作成
    ↓
レビュワー: Issueを確認
    ↓
【対応が必要】
  執筆者/レビュワー: 修正PRを作成 → レビュー → マージ
    ↓
【対応不要】
  レビュワー: 理由を説明してIssueをクローズ
```

*   **報告フォーム詳細**: [FORMS_DESIGN.md](./FORMS_DESIGN.md) を参照。
*   **スパム対策**: hCaptcha等のプライバシー重視ソリューションを使用。

#### UC-P5: 出典の削除を提案

出典の削除が必要と思われる場合のワークフロー。

##### 削除が必要なケース

| ケース | 説明 |
| :--- | :--- |
| **不適切な出典** | 主張と無関係、信頼性のない出典 |
| **存在しない資料** | 実在しない資料（手動入力時のレビュー漏れ） |
| **重複** | 同じ出典が複数IDで登録されている |

##### ワークフロー

```
報告者: Issue/フォームで削除を提案
  - 対象出典ID
  - 削除理由（不適切/存在しない/重複等）
  - 重複の場合：正しいIDを提示
    ↓
レビュワー: 確認
  - 本当に削除が必要か
  - 他の記事での使用状況（usage-index.json で確認）
    ↓
【判断】
├─ 記事から外すだけでOK
│   → 該当記事を修正するPRを作成
│
├─ bibliography.json から削除が必要
│   → deprecated ステータスに変更するPRを作成
│   → マージ後、定期チェック（月1回）で未使用と判定されれば自動的に物理削除
│
└─ 削除しない
    → 理由を説明してIssueをクローズ
```

*   **deprecatedステータス**: 新規使用は禁止されるが、既存参照は維持される。
*   **物理削除**: 月1回の定期チェックで未使用と判定されれば自動PR作成→レビュワーマージで削除。詳細は [GOVERNANCE.md](./GOVERNANCE.md) セクション6.4を参照。
