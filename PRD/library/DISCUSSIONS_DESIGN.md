# ディスカッションページ設計 (Discussions Design)

本ドキュメントでは、Isidoricaのディスカッションページの設計を定義する。

**ステータス**: ドラフト（Phase 4以降で実装予定）

---

## 1. 概要

### 1.1 目的

*   記事の内容、出典選択、翻訳方針などについて執筆者間で議論する場を提供する（**Authors' Table**）。
*   学習者が疑問（問い）や気づきを共有し、水平的な対話を行う場を提供する（**Learners' Agora**）。

### 1.2 基本方針

| 項目 | 決定 |
| :--- | :--- |
| **データ管理** | GitHub Discussions をバックエンドに使用 |
| **表示** | Isidorica内にディスカッションページを作成 |
| **紐付け** | 記事ページからディスカッションへリンク（存在する場合のみ） |
| **Frontmatter** | 汚さない（紐付けは記事slugで自動） |

### 1.3 Discussions と Issue の使い分け

GitHub Discussions と GitHub Issue は役割が異なる。適切に使い分けることで、管理が効率化され、コントリビューターにとっても分かりやすくなる。

#### 基本原則

| ツール | 性質 | 完了状態 |
| :--- | :--- | :--- |
| **GitHub Discussions** | 会話指向（議論・意見交換） | 継続的（明確な終点がない） |
| **GitHub Issue** | タスク指向（何かを完了させる） | オープン/クローズ（完了可能） |

#### 用途別ガイドライン

| 用途 | 使うべきツール | 例 |
| :--- | :--- | :--- |
| 記事の方針検討 | **Discussions** | 「認知バイアスの記事をどう構成すべきか」 |
| 出典の選択 | **Discussions** | 「この主題にはどの文献を参照すべきか」 |
| 翻訳方針 | **Discussions** | 「この専門用語は日本語でどう訳すべきか」 |
| 学習パスの設計 | **Discussions** | 「哲学入門パスにはどの記事を含めるべきか」 |
| 一般的なフィードバック | **Discussions** | 「この記事は分かりにくかった」 |
| 質問 | **Discussions** | 「なぜこの解釈を採用しているのか」 |
| 記事更新依頼 | **Issue** | 「game-theory の出典が古い」 |
| 翻訳依頼 | **Issue** | 「kant-ethics の英語版が必要」 |
| 誤り報告 | **Issue** | 「数式に間違いがある」 |
| 見直しリマインダー | **Issue** | Bot による自動生成 |
| 具体的な修正提案 | **Issue** | 「この段落を〇〇に変更すべき」 |

#### フロー

```
読者・コントリビューター
     │
     ├─ 明確な問題がある → Issue 作成
     │      └─ 対応者がPRを作成 → クローズ
     │
     └─ 議論が必要 → Discussions 投稿
            └─ 議論の結果、アクションが決まる
                   └─ 必要なら Issue を作成
```

#### 判断に迷った場合

| 質問 | 回答に応じた選択 |
| :--- | :--- |
| 「これは誰かがすぐにPRで解決できるか？」 | はい → Issue / いいえ → Discussions |
| 「他の人の意見を聞きたいか？」 | はい → Discussions / いいえ → Issue |
| 「複数の選択肢があり、どれを選ぶべきか議論が必要か？」 | はい → Discussions |

### 3.3 Agora UI設計（非競争的デザイン）

LEARNING_FOUNDATIONS.mdに基づき、Agoraは「能力の競争」ではなく「経験の共有」を促す。

*   **No Ranking**: 「いいね数」や「投稿数」によるランキング表示は行わない。
*   **Insight First**: 「ベストアンサー（正解）」という概念は使用せず、良質な問いや気づきをハイライトする。
*   **Categories**:
    *   **Inquiry**: 「分からないこと」を恥とせず、共有すべき「問い」として扱う。
    *   **Insight**: 「正解」ではなく「個人的な発見」を歓迎する。

### 3.4 行動規範の適用

Agoraにおける対話は [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md) に従う。

#### 非競争的対話の原則

| 推奨される行動 | 避けるべき行動 |
| :--- | :--- |
| つまずきや疑問を共有する | 能力を誇示する |
| 「別の見方もあるか？」と問いかける | 「間違っている」と論破しようとする |
| 発見や気づきを共有する | 他者を見下す |
| 出典に基づいて建設的に議論する | 資格や経験を問題にする |

#### 違反報告

Agoraでの行動規範違反は以下の方法で報告できる：

| 方法 | 説明 |
| :--- | :--- |
| **GitHub Discussion内で報告** | `@isidorica-bot report` とコメント（将来実装） |
| **GitHub Issue** | `conduct` ラベルを付与してIssue作成 |
| **直接連絡** | Core Maintainerへダイレクトメッセージ |

> **参照**: 行動規範の詳細は [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md) を参照。

## 4. GitHub Discussions 連携

### 4.1 カテゴリ設計

GitHub Discussionsに以下のカテゴリを設定:

| カテゴリ | 用途 |
| :--- | :--- |
| `authors-sources` | **[執筆者]** 出典に関する議論 |
| `authors-translation` | **[執筆者]** 翻訳に関する議論 |
| `authors-content` | **[執筆者]** 内容・構成に関する議論 |
| `learners-inquiry` | **[Agora]** 疑問・問いの共有（Q&Aではない探究） |
| `learners-insight` | **[Agora]** 気づき・発見の共有（Aha体験） |
| `learners-connections` | **[Agora]** 他分野との関連性の発見（転移） |
| `general` | その他 |

### 4.2 タイトル規約

```
[{slug}] {議論のタイトル}
```

例:
*   `[prisoner-dilemma] ハイデガー初版使用の理由`
*   `[kant-ethics] 英語版の用語選択について`

### 4.3 紐付けの仕組み

1.  **ビルド時**: GitHub APIで全Discussionsを取得
2.  **マッチング**: タイトルの `[{slug}]` 部分で記事とマッチ
3.  **件数取得**: 記事ごとのDiscussion件数を集計
4.  **表示**: 記事ページに件数とリンクを表示

## 8. 多言語対応戦略

### 8.1 基本方針

Agoraは**言語別に分離せず、共通URL**で運用する。

| 項目 | 仕様 |
| :--- | :--- |
| **URL** | `/agora/{slug}`（言語プレフィックスなし） |
| **投稿** | 任意の言語で投稿可能（原文を保存） |
| **表示** | 閲覧者の言語設定に基づいて翻訳表示 |

### 8.2 設計根拠

| 観点 | 言語分離の問題 | 共通URLのメリット |
| :--- | :--- | :--- |
| **議論の統一** | 同じ記事でも言語ごとに議論が分断 | 全世界の議論が1つの場に |
| **知識共有** | 他言語の洞察が見えない | 言語を超えて学び合える |
| **公正（Equity）** | N/A | 母語で投稿・閲覧可能 |
| **包括性** | N/A | 英語強制なし |

### 8.3 翻訳表示の仕組み

#### フロー

```
ユーザーが /agora/game-theory にアクセス
  ↓
ユーザーの言語設定を取得（例: 日本語）
  ↓
各投稿の原文言語をチェック
  ├─ 原文が日本語 → そのまま表示
  └─ 原文が他言語 → 日本語に翻訳して表示
```

#### UI要件

| 要素 | 要件 |
| :--- | :--- |
| **翻訳インジケータ** | 翻訳されたコンテンツには `🌐 翻訳済み` アイコンを表示 |
| **「原文を見る」ボタン** | ワンクリックで原文に切り替え可能 |
| **原文言語の表示** | 投稿者の使用言語を表示（例: `原文: English`） |

### 8.4 投稿時の言語処理

| 項目 | 仕様 |
| :--- | :--- |
| **言語検出** | 投稿内容から自動検出（またはユーザーが選択） |
| **原文保存** | 常に原文を保存（翻訳は表示時に生成） |
| **翻訳キャッシュ** | 同じ言語への翻訳はキャッシュ（コスト削減） |

### 8.5 翻訳品質への対策

| 懸念 | 対策 |
| :--- | :--- |
| **誤訳** | 「原文を見る」機能で確認可能 |
| **ニュアンスの損失** | 翻訳であることを明示（アイコン） |
| **専門用語** | `terms.yml` との連携（将来機能） |
| **翻訳問題の報告** | 「翻訳を報告」ボタンでフィードバック収集 |

## 9. 将来の検討事項

*   **ボット経由投稿**: 非GitHubユーザーがフォームから投稿
*   **通知機能**: 記事の執筆者・翻訳者への通知
*   **RSS/Atom**: ディスカッションのフィード配信
*   **検索機能**: ディスカッション内の全文検索
*   **用語辞書連携**: `terms.yml` を翻訳時に参照して専門用語の翻訳精度向上
*   **スパム対策**: GitHub認証で基本的に抑制されるが、ボットアカウント対策として追加の検証（投稿頻度制限、レピュテーションシステム）を検討
*   **翻訳コスト管理**: 大量投稿発生時のLLM翻訳APIコスト増大への対策（キャッシュ戦略、翻訳リクエスト制限、オンデマンド翻訳方式の検討）
