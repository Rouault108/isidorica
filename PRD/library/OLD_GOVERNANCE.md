# コンテンツガバナンス (Content Governance)

本ドキュメントでは、Isidoricaにおけるコンテンツの執筆、レビュー、出典管理に関するガバナンスルールを定義する。

> **翻訳関連**: 多言語対応・翻訳ワークフローについては [TRANSLATION_DESIGN.md](./TRANSLATION_DESIGN.md) を参照。理論的基盤は [TRANSLATION_FOUNDATIONS.md](../shared/TRANSLATION_FOUNDATIONS.md) を参照。

---

## 概要

### 基本原則

| 原則 | 内容 |
| 原則 | 内容 |
| :--- | :--- |
| **出典至上主義** | 全ての主張に検証可能な出典を要求 |
| **AI証言の制約** | AIは「真実を述べる意思（誠実性）」を持たないため、知識源として使用してはならない（[CORE_FOUNDATIONS.md §2.1](../shared/CORE_FOUNDATIONS.md) 証言の認識論） |
| **独自研究の排除** | 教育的な再構成（Synthesis）は推奨するが、独自の結論や因果関係を断定してはならない |
| **執筆者に負担を与えない** | 複雑な処理は内部で自動化 |
| **概念レベルで共有** | 出典は言語版ごとではなく、概念に紐づく |

### 執筆者の匿名性

執筆者は原則として匿名とする。

| 項目 | 決定 |
| :--- | :--- |
| **公開情報** | 著者名なし、最終更新日のみ表示 |
| **内部管理** | GitHubアカウントで追跡可能 |

#### 理由

*   **出典至上主義との整合**: 「誰が言ったか」ではなく「何を根拠にしているか」で信頼性を判断。
*   **ハロー効果の排除**: 有名人だから正しい、という誤謬を防ぐ。
*   **参入障壁の低下**: 学位や肩書きがなくても貢献可能。

#### リスク対策

| 対策 | 説明 |
| :--- | :--- |
| **レビュー必須** | PRマージ前にレビュワーが確認 |
| **出典必須** | 検証可能な出典がなければ公開しない |
| **GitHub認証** | 荒らし行為の防止 |
| **多くの目（Linus's Law）** | 読者は誰でも誤りを発見・報告・修正提案可能 |
| **編集の機会** | 記事ページから「編集を提案」リンクを提供 |

#### 共同執筆

*   記事は**コミュニティ全体の成果物**であり、特定の個人に帰属しない。
*   複数の執筆者が貢献することを推奨する。
*   責任はコミュニティ全体で担い、問題発生時は記事単位で対応（個人を特定して責任追及しない）。
*   技術的にはGitの履歴で追跡可能（内部管理のみ）。

#### 利益相反

執筆者は匿名であり、利益相反の明示的な開示は不要とする。

| 対策 | 説明 |
| :--- | :--- |
| **出典の多様性** | 単一の情報源に依存しない |
| **レビューでの検出** | 偏った記述をレビュワーが指摘 |
| **中立性の原則** | 特定の立場に偏らない記述を求める |

*   **自著引用**: 禁止はしないが、単一出典への依存はレビューで指摘。
*   **広告的内容**: 特定の製品・サービスを過度に推奨する記述は禁止。

#### 記事履歴の非公開

読者向けに記事の編集履歴UIは**提供しない**。

| 項目 | 決定 |
| :--- | :--- |
| **読者向け表示** | 「最終更新日」のみ表示。履歴へのリンクは提供しない |
| **内部管理** | Gitの履歴で追跡可能（管理者・執筆者のみ） |

##### 理由

| 観点 | 説明 |
| :--- | :--- |
| **出典至上主義との整合** | 記事の信頼性は「出典」で担保される。履歴は「誰が」「いつ」という人物情報であり、出典至上主義とは本質的に無関係 |
| **匿名原則の保護** | 履歴を公開すると、GitHubハンドルや編集パターンから執筆者が特定されるリスクがある |
| **認知的倹約家への配慮** | 大半の読者は履歴を必要としない。不要な情報を排除することで認知負荷を軽減 |

##### 透明性への対応

*   **出典の透明性**で検証可能性を担保する（履歴ではなく）。
*   問題が発覚した場合は、記事単位で対応し、個人を特定して責任追及しない（共同執筆の原則）。


### 役割

| 役割 | 責務 |
| :--- | :--- |
| **執筆者** | 記事を執筆し、出典を執筆支援ツールで検索・挿入 |
| **レビュワー** | 記事の内容・出典をレビュー、翻訳関係の設定（必要な場合のみ） |
| **Core Maintainer** | 最終決定権、レビュワーの任命、緊急対応（セクション4.6参照） |
| **CI** | 新規出典の自動検出・レポート、ビルド時の出典処理 |

#### レビュワーの権限

| 権限 | 説明 |
| :--- | :--- |
| **PRのレビュー** | 記事の内容・出典を確認し、Approve/Request Changes |
| **PRのマージ** | 承認後、mainブランチへマージ |
| **出典の検証** | 新規出典の存在確認、内容確認 |
| **翻訳関係の設定** | 原著と翻訳版の紐付け |

#### レビュワーの選任基準

| 基準 | 説明 |
| :--- | :--- |
| **貢献実績** | Isidoricaへの一定量の貢献（具体的な数値は未決定） |
| **出典遵守** | 自身の執筆において出典ルールを遵守してきた実績 |
| **コミュニケーション** | 建設的なフィードバックができる |
| **分野専門性**（オプション） | 特定分野のレビュワーとして、その分野の知識 |

> **TODO**: 貢献実績の具体的な基準（記事数、PR数等）は運用開始後に決定。

#### レビュワーの選任プロセス

| フェーズ | 方法 |
| :--- | :--- |
| **現状（Phase 5）** | Core Maintainerによる任命 |
| **将来（コミュニティ拡大後）** | 候補者の推薦 → Core Maintainerの承認 |

**選任フロー（将来）**:

```
候補者がレビュワー就任を希望 or 既存レビュワー/Core Maintainerが推薦
    ↓
Core Maintainerが選任基準を確認
    ↓
【基準充足】
    ├─ 候補者に打診
    ├─ 同意を得たら任命
    └─ GitHub Team への追加、権限付与
    ↓
【基準未充足】
    └─ 理由を説明し、今後の貢献を奨励
```

#### レビュワーの解任基準

| 基準 | 説明 |
| :--- | :--- |
| **長期非活動** | 6ヶ月以上レビュー活動がない |
| **品質問題** | レビューの質が著しく低い（出典確認漏れが繰り返される等） |
| **行動規範違反** | 他のコントリビューターへの攻撃的な言動 |
| **自主退任** | レビュワー本人の希望 |

#### レビュワーの解任プロセス

| ケース | プロセス |
| :--- | :--- |
| **長期非活動** | Core Maintainerから連絡 → 2週間以内に応答なし → 解任通知 → 権限削除 |
| **品質問題** | Core Maintainerから改善依頼 → 改善なし → 解任通知 → 権限削除 |
| **行動規範違反** | Core Maintainerが判断 → 即時解任も可 |
| **自主退任** | レビュワーから申し出 → 引き継ぎ（必要に応じて） → 権限削除 |

#### レビュワーの種類（将来拡張）

| 種類 | 説明 |
| :--- | :--- |
| **General Reviewer** | 全分野のレビューが可能 |
| **Domain Reviewer** | 特定分野（哲学、経済学等）のレビューを担当 |
| **Language Reviewer** | 特定言語の翻訳レビューを担当 |

#### 専門家レジストリ

KPIの「専門家レビュー率」を測定するため、専門的知識を持つレビュワーを登録・管理する仕組み。

> **参照**: [KPI_FRAMEWORK.md §5.5](../shared/KPI_FRAMEWORK.md) 専門家レビュー率

##### 定義

**専門家**とは、以下のいずれかを満たすレビュワーを指す：

| 基準 | 説明 |
| :--- | :--- |
| **学術的資格** | 当該分野での博士号、または同等の専門教育 |
| **職業的実績** | 当該分野での5年以上の実務経験 |
| **学術的貢献** | 査読論文、著書、または同等の学術的成果 |

##### レジストリ管理

| 項目 | 内容 |
| :--- | :--- |
| **ファイル** | `config/expert-reviewers.yml` |
| **編集権限** | Core Maintainerのみ（CODEOWNERS設定で保護） |
| **透明性** | レジストリ自体は公開（GitHubリポジトリ上） |

##### 認定プロセス

```
候補者が専門家レビュワーとして登録を希望
   ↓
Core Maintainerが資格を確認（CV、実績等）
   ↓
【承認】
   ├─ config/expert-reviewers.yml に追加
   └─ GitHub Teamへの追加は不要（既存レビュワー権限で十分）
【非承認】
   └─ 理由を説明し、必要に応じて通常レビュワーとして活動を奨励
```

##### 測定の仕組み

専門家がPRを承認すると、GitHub Actionsワークフローが自動的に記録を作成する。

> **実装詳細**: [CI_CD_PIPELINE.md §9.2](../engine/technical/CI_CD_PIPELINE.md) 専門家レビュー追跡ワークフロー

| **選任・解任の記録** | GitHub Discussionsで公開（理由は概要のみ） |

### 学習理論に基づく品質基準

コンテンツの品質は、単なる「正確さ」だけでなく、以下の学習理論的基準を満たす必要がある。

#### 1. 望ましい困難 (Desirable Difficulties)
> **参照**: [LEARNING_FOUNDATIONS.md §10.3](../shared/LEARNING_FOUNDATIONS.md)

*   **過度な単純化の回避**: わかりやすさを優先して、概念の本質的な複雑さを隠蔽してはならない。
*   **不快な流暢さの排除**: 読者が「わかった気」になるだけの、摩擦のない流暢な説明は避ける。適度な認知的負荷（立ち止まって考える機会）を含めること。

#### 2. 転移適切性処理 (TAP)
> **参照**: [LEARNING_FOUNDATIONS.md §2.5](../shared/LEARNING_FOUNDATIONS.md)

*   **実践的文脈の提示**: 概念の定義だけでなく、「その概念を使って現実の問題をどう解決するか」という適用例（Laboratory課題等）を含めること。

#### 3. 注意資源の管理 (Attention Restoration)
> **参照**: [LEARNING_FOUNDATIONS.md §2.4](../shared/LEARNING_FOUNDATIONS.md)

*   **Soft Fascinationへの配慮**: 記事内で使用するメディア（画像、動画、Laboratoryの演出）は、過剰な刺激（Hard Fascination）を避け、穏やかな注意回復を促すものを選定する。

#### 4. ナッジの倫理的制約
> **参照**: [USER_BEHAVIOR_MODEL.md §4.4](../shared/USER_BEHAVIOR_MODEL.md)

*   **ユーザー利益優先**: 執筆者およびレビュワーは、記事内の誘導（「次へ」「Laboratoryへ」等）がユーザーの長期的学習利益に資するかを確認する。プラットフォームの滞在時間やエンゲージメントのみを目的とした誘導は禁止する。

#### 出典確認の多層防御

品質を最優先しつつ、運用可能な負担に抑えるため、多層的な確認体制を構築する。

##### 執筆者の責任（一次確認）

| 責任 | 内容 |
| :--- | :--- |
| **全出典の確認** | AIが提示した出典を含む全ての出典を一次情報で確認 |
| **存在確認** | 出典が実在することを確認（ISBN/DOI検索、図書館検索等） |
| **内容確認** | 出典が主張を支持していることを確認（ページ番号レベルで） |

##### 自動チェック（CI）

| チェック項目 | 方法 | 検出できるもの |
| :--- | :--- | :--- |
| **ISBN存在確認** | Google Books API / OpenLibrary API | 存在しない書籍 |
| **DOI存在確認** | CrossRef API | 存在しない論文 |
| **URL疎通確認** | HTTP HEAD リクエスト | リンク切れ |

> **限界**: 自動チェックは「存在」を確認できるが、「内容が主張を支持しているか」は確認できない。人間による確認は依然として必要。

##### レビュワーの責任（検証）

| 責任 | 内容 |
| :--- | :--- |
| **書誌情報の妥当性確認** | 著者名、出版年、タイトル等が妥当か目視確認 |
| **不審な出典の精査** | 知らない出典、曖昧な書誌情報は一次情報で重点確認 |
| **新規出典の全件確認** | 初めて使用される出典（auto status）は全件確認を目標とする |

##### 既存出典の扱い

| 出典ステータス | 確認レベル |
| :--- | :--- |
| **新規出典（auto）** | 全件確認 |
| **既存出典（curated）** | 確認済みのためスキップ可 |

> **理由**: 一度curatedされた出典は、別の記事で使用される際に再確認は不要。

##### 公開後の継続的品質管理

| 仕組み | 内容 |
| :--- | :--- |
| **読者報告** | 報告フォームで出典の誤りを受け付け |
| **定期リンクチェック** | 月1回、全URL出典の疎通確認 |
| **迅速な修正フロー** | 誤り発見時、24時間以内に対応開始 |

#### 執筆者向けガイダンス

執筆者向けドキュメントには以下を明記する。

```
## AIの使用について

AIツール（ChatGPT、Claude、Gemini等）を執筆支援に使用することは禁止も推奨もしません。

ただし、**AIが提示した出典は必ず人間が一次情報で確認してください**。
AIは存在しない出典を自信満々に提示することがあります（ハルシネーション）。

確認事項：
- 出典が実在すること（ISBN/DOI検索、図書館検索等）
- 出典が主張を支持していること（ページ番号レベルで確認）

存在しない出典を引用した記事は、発見次第修正または削除されます。
```

## 1. 出典管理 (Source Management)

### 1.1 技術基盤

| 項目 | 技術 |
| :--- | :--- |
| **書誌データベース** | `sources/bibliography.json`（Git管理） |
| **引用形式** | CSL (Citation Style Language) |
| **引用形式の選択** | ハイブリッド方式（デフォルト + 分野別オーバーライド） |
| **外部API** | Google Books, CiNii, CrossRef（書誌情報の自動取得） |

#### 引用形式のハイブリッド方式

Isidoricaは複数の学問分野を横断するため、**デフォルト形式**と**分野別オーバーライド**を組み合わせたハイブリッド方式を採用する。

##### 設計原則

| 原則 | 説明 |
| :--- | :--- |
| **出典至上主義との整合** | 各分野の引用慣習を尊重することで、出典の「品質」を担保 |
| **読者の期待値** | 心理学記事を読む人はAPA形式を期待する。慣れ親しんだ形式は信頼性を高める |
| **執筆者の負担軽減** | `primary_discipline` は Optional。未指定時はデフォルト適用 |

##### マッピング定義

ディレクトリ構造と連動し、親ディレクトリから引用形式を自動推論する。

| 分野 (`discipline`) | CSLスタイル | 引用形式名 |
| :--- | :--- | :--- |
| **（未指定 / 空 / 未定義値）** | `harvard-cite-them-right` | **Harvard**（デフォルト） |
| `psychology` | `apa-7th-edition` | APA 7th Edition |
| `education` | `apa-7th-edition` | APA 7th Edition |
| `sociology` | `apa-7th-edition` | APA 7th Edition |
| `linguistics` | `apa-7th-edition` | APA 7th Edition |
| `medicine` | `vancouver` | Vancouver |
| `biology` | `vancouver` | Vancouver |
| `law` | `oscola` | OSCOLA |
| `philosophy` | `chicago-fullnote-bibliography` | Chicago (Notes-Bibliography) |
| `history` | `chicago-fullnote-bibliography` | Chicago (Notes-Bibliography) |
| `art` | `chicago-fullnote-bibliography` | Chicago (Notes-Bibliography) |
| `religion` | `chicago-fullnote-bibliography` | Chicago (Notes-Bibliography) |
| `economics` | `chicago-author-date` | Chicago (Author-Date) |
| `politics` | `chicago-author-date` | Chicago (Author-Date) |
| `natural-science` | `nature` | Nature Style |
| `physics` | `nature` | Nature Style |
| `chemistry` | `nature` | Nature Style |
| `mathematics` | `american-mathematical-society-numeric` | AMS |
| `literature` | `modern-language-association` | MLA 9th Edition |
| `engineering` | `ieee` | IEEE |
| `computer-science` | `ieee` | IEEE |

##### Harvardをデフォルトとする理由

| 理由 | 説明 |
| :--- | :--- |
| **汎用性** | 分野を問わず使える「無色透明」な形式 |
| **読みやすさ** | `(田中, 2020)` という著者-日付形式はWebで読みやすい |
| **国際標準** | イギリス・オーストラリア・日本で広く採用 |
| **シンプルさ** | APAほど厳格なフォーマット規則がない |

##### 執筆者支援ツールでのガイダンス

Web執筆支援ツールでは、`topics` に基づいて適切な `primary_discipline` を提案する。

```
💡 ヒント: この記事は「心理学」タグを含んでいます。
   心理学分野では APA 7th Edition が標準です。
   primary_discipline: psychology を指定すると適用されます。
```

##### 読者への表示

引用形式は**表示しない**ことを原則とする。

| 理由 | 説明 |
| :--- | :--- |
| 研究者は見れば分かる | フォーマットを見れば判断できる |
| 一般読者には不要 | 混乱を招く可能性がある |
| ミニマリズム | デザイン哲学「洗練されたミニマリズム」との整合 |

##### 分野リストの管理

`primary_discipline` に使用可能な値は、設定ファイル `config/disciplines.yml` で一元管理する。

**設計方針**:

| 方針 | 説明 |
| :--- | :--- |
| **独立した設定ファイル** | topics（発見可能性）とは別に、disciplines（引用形式決定）を管理 |
| **Single Source of Truth** | 分野リストとCSLマッピングを1ファイルで定義 |
| **拡張可能** | 多言語名称、関連トピック、説明文等を将来追加可能 |

**スキーマ案**:

```yaml
# config/disciplines.yml
default:
  citation_style: harvard-cite-them-right
  citation_name: Harvard

disciplines:
  psychology:
    name: 心理学
    name_en: Psychology
    citation_style: apa-7th-edition
    citation_name: APA 7th Edition
  
  medicine:
    name: 医学
    name_en: Medicine
    citation_style: vancouver
    citation_name: Vancouver
  
  # ... 他の分野
```

> **実装**: 設定ファイルの作成はPhase 4.3で行う。詳細は [PROJECT_ROADMAP.md](../shared/PROJECT_ROADMAP.md) を参照。

##### 記事内での引用形式の一貫性

**原則**: 1つの記事内では1つの引用形式を使用する。記事途中での形式変更は**実装しない**。

| 設計判断 | 理由 |
| :--- | :--- |
| **1記事1形式** | 学術慣習に従う。読者体験の向上。実装の単純化 |
| **途中変更なし** | 形式の混在は読みにくく、散漫な印象を与える |

**学際的（Interdisciplinary）な記事の場合**:

複数分野にまたがる記事（例: 認知科学 = 心理学 + 神経科学 + 哲学）では、以下の優先順位で `primary_discipline` を決定する。

| 優先度 | 方法 | 例 |
| :--- | :--- | :--- |
| **1** | 記事の主たるアプローチ/切り口に基づく分野 | 認知心理学的アプローチなら `psychology` |
| **2** | 想定する主要読者層の慣習に合わせる | 心理学者向けなら `psychology` |
| **3** | 最も頻繁に引用する分野 | 心理学論文が多いなら `psychology` |
| **4** | デフォルト（Harvard）を使用 | 判断できない場合 |

Harvard形式は「無色透明」なため、どの分野の読者にも受け入れられやすい。

> **参照**: Frontmatterの `primary_discipline` フィールド仕様は [MARKDOWN_SPEC.md](./technical/MARKDOWN_SPEC.md) を参照。

| API | 料金 | 制限 | 備考 |
| :--- | :--- | :--- | :--- |

### 1.3 引用記法

Pandoc標準の引用記法を採用する。

#### 出典ID形式

出典IDは以下の形式で自動生成される。

```
{第一著者姓}-{キーワード}-{年}
```

| 例 | ID |
| :--- | :--- |
| 田中太郎『ゲーム理論入門』2020 | `tanaka-game-theory-2020` |
| 田中・山田『経済学概論』2019 | `tanaka-economics-2019` |
| Heidegger "Being and Time" 1962 | `heidegger-being-time-1962` |
| 厚生労働省『白書』2023 | `mhlw-whitepaper-2023` |

*   **複数著者**: 第一著者のみIDに含む。共著情報は `bibliography.json` に記録。
*   **同著者複数著作**: キーワードで区別。
*   **機関著者**: 略称を使用（例: `mhlw` = 厚生労働省）。
*   **自動生成**: 執筆支援ツールがAPIを通じて自動生成。執筆者はコピーして使用。

#### ID衝突時の処理

| レイヤー | タイミング | 処理 |
| :--- | :--- | :--- |
| **API（POST /api/sources）** | ID生成時 | 既存IDと照合。衝突時は名のイニシャル→連番の順で付与。 |
| **CI（早期フィードバック）** | PR作成時 | 追加された全IDをチェック。衝突があれば警告（マージは可能）。 |
| **CI（最終防衛線）** | マージ前 | 追加された全IDをチェック。衝突があればマージをブロック。 |
*   **競合状態への対応**: 複数のPRが同時に同じIDを追加しようとした場合、先にマージされたPRが優先され、後続のPRはマージ時にエラーとなる。この場合、後続のPR作成者はIDを修正する必要がある。
*   **衝突対策1**: 名のイニシャルを追加（例: `wang-m-game-theory-2020`）
*   **衝突対策2**: 連番を追加（例: `wang-m-game-theory-2020-2`）
*   **API設計詳細**: [API_DESIGN.md](../engine/technical/API_DESIGN.md) を参照。

#### 基本記法

| 記法 | 意味 | 例 |
| :--- | :--- | :--- |
| `[@source-id]` | 出典全体を参照 | `[@heidegger-being-time]` |
| `[@source-id, p. 45]` | 特定ページを参照 | `[@heidegger-being-time-ja, p. 45]` |
| `[@source-id, pp. 45-50]` | ページ範囲を参照 | `[@heidegger-being-time-ja, pp. 45-50]` |
| `[@source-id, ch. 3]` | 章を参照 | `[@heidegger-being-time-ja, ch. 3]` |
| `[@source-id, sec. 2.1]` | セクションを参照 | `[@heidegger-being-time-ja, sec. 2.1]` |

#### 原典ページの併記

執筆者が翻訳版と原典のページ対応を確認した場合、両方を記載可能。

| 記法 | 意味 |
| :--- | :--- |
| `[@id-ja, p. 45; orig. p. 42]` | 日本語版p.45、原典p.42 |
| `[@id-ja, pp. 45-50; orig. pp. 40-45]` | ページ範囲も同様 |

#### 自動選択ルール

| ケース | 処理 |
| :--- | :--- |
| `[@source-id]`（ページなし） | 言語版に応じて翻訳版を自動選択 |
| `[@source-id-ja, p. 45]`（ページあり） | 指定された版をそのまま使用（自動選択しない） |
| `[@source-id-ja, p. 45; orig. p. 42]`（両方） | 言語版に応じて主従を切り替え |

#### 表示例

| 記法 | 日本語版での表示 | 英語版での表示 |
| :--- | :--- | :--- |
| `[@id-ja, p. 45]` | ハイデガー 2003, p.45 | Heidegger (ja trans. 2003), p.45 |
| `[@id, p. 42]` | Heidegger 1927, p.42 | Heidegger 1927, p.42 |
| `[@id-ja, p. 45; orig. p. 42]` | ハイデガー 2003, p.45（原著 p.42） | Heidegger 1927, p.42 (ja trans. p.45) |

#### ページ番号要件

| 引用タイプ | ページ番号 | CI検証 |
| :--- | :--- | :--- |
| **直接引用**（「」で囲む、blockquote） | **必須** | エラー（マージ不可） |
| **パラフレーズ**（特定箇所の言い換え） | **強く推奨** | 警告（レビュワーが判断） |
| **全体参照**（著作全体のアイデア） | 任意 | 警告なし |

*   **出典至上主義**: 検証可能性を担保するため、可能な限りページ番号を記載することを推奨。
*   **引用形式との関係**: MLA、Chicagoなど厳格な形式ではパラフレーズでもページ番号が求められる。
*   **レビュワーの役割**: CI警告が出た場合、レビュワーが文脈を確認し最終判断。詳細な判断基準は運用を通じて策定。

### 1.4 出典の共有モデル

*   出典は「**概念**」レベルで管理される。言語版ごとではなく、同一概念の全言語版で出典リストを共有する。
*   出典の追加は任意の言語版から可能だが、レビューを経て全言語版に反映される。
*   **注意事項**: CSLにアクセシビリティ用の項目（`(In Japanese)` 等の言語注記）を含める方法について要調査。

### 1.5 言語圏で確認困難な出典

*   原著の言語を明記する（例: `(In Japanese)`）。
*   可能な限り、国際的に入手可能な出典を併記することを推奨する。

### 1.6 外部APIでカバーできない資料

外部APIで取得できない資料は、執筆者が手動で入力する。

| 資料タイプ | 入力フィールド | CSL Type |
| :--- | :--- | :--- |
| **Webページ** | URL, タイトル, アクセス日, 著者（任意）, 公開日（任意） | webpage |
| **個人所蔵・未出版資料** | タイトル, 所蔵者, 資料形態, 年代, 所在地 | manuscript |
| **考古・アーカイブ資料** | 資料名, 所蔵機関, 参照番号, 出土地/年代 | article |
| **法令** | 法律名, 法律番号, 施行日, 国 | legislation |
| **判例** | 事件名, 裁判所, 判決日, 事件番号 | legal_case |
| **データセット** | タイトル, 提供者, 公開年, DOI/URL | dataset |

#### Webページの引用に関する注意

*   **必須**: URL, タイトル, アクセス日
*   **同じURLは同一出典**: アクセス日が異なっても、同じURLは同一の出典エントリとして管理。
    *   執筆支援ツールは既存のURLを検索し、存在すれば既存の出典IDを返す。
    *   個別のアクセス日は記事ごとに管理（ビルド時に自動記録）。
*   **Wayback Machine連携**: **積極的に推奨しない**
    *   日本著作権法にはフェアユースがなく、第三者がアーカイブに登録する行為は著作権侵害のリスクがある。
    *   Isidoricaとして法的グレーゾーンの行為は避ける。
*   **リンク切れ対策**:
    *   アクセス日を必ず記録し、リンク切れ時は「元のURLとアクセス日」を表示。
    *   既にアーカイブされているURLがあれば、補助的に記載可能（自ら登録しない）。
    *   重要な情報は権利者に許諾を得て保存を検討。

#### 手動入力資料の翻訳関係

国際条約、他国法令、古典文献など、公式翻訳や定訳が存在する資料は翻訳関係を設定可能。

| ケース | 翻訳関係設定 | 例 |
| :--- | :--- | :--- |
| 公式翻訳がある | 設定する | 国連憲章（英語原文 ↔ 日本語公定訳） |
| 学術的な定訳がある | 設定する | プラトン『国家』（ギリシャ語 ↔ 各国語訳） |
| 非公式翻訳のみ | 設定しない | 原典を使用、言語注記を追加 |
| 翻訳がない | 設定しない | 原典のみ表示 |

*   執筆者は**原典のID**を使用（例: `[@un-charter]`）。
*   翻訳関係が設定されていれば、ビルド時に言語版に応じて自動選択。
*   入力フォームで翻訳版を追加登録可能。

### 1.7 出典消滅時の対応

Webページ削除、URL変更、ドメイン失効等により出典にアクセスできなくなった場合の対応。

#### 検出

| 方法 | 説明 |
| :--- | :--- |
| **定期自動チェック** | CIで月1回、全WebページURLをチェック（404/5xx検出） |
| **読者からの報告** | 報告フォーム経由でリンク切れを報告 |

#### 対応手順

| 優先度 | 対応 | 説明 |
| :--- | :--- | :--- |
| 1 | **Wayback Machineを補助表示** | 既にアーカイブがあれば `archive_url` に記録し表示 |
| 2 | **代替URLを探す** | 移転先、ミラーサイトを探して `url` を更新 |
| 3 | **代替出典を探す** | 同内容の別資料を `alternative` に記録 |
| 4 | **状態を明示** | 「この出典は現在アクセスできません」と表示 |
| 5 | **主張を削除/修正** | 出典なしでは成り立たない場合、記事を修正 |

#### データ構造

```json
{
  "source-id": {
    "url": "https://example.com/article",
    "accessed": "2025-12-13",
    "status": "dead",
    "dead_since": "2025-11-01",
    "archive_url": "https://web.archive.org/web/...",
    "alternative": "alternative-source-id"
  }
}
```

*   **status**: `alive`（アクセス可能）, `dead`（アクセス不可）, `unknown`（未確認）

### 1.8 同じ著作の異なる版

初版・改訂版など、同じ著作でも版が異なる場合は別々のエントリとして管理する。

#### データ構造

```json
{
  "sample-book-1st": {
    "title": "サンプル書籍",
    "edition": "初版",
    "year": 2010
  },
  "sample-book-2nd": {
    "title": "サンプル書籍（改訂版）",
    "edition": "第2版",
    "year": 2015,
    "replaces": "sample-book-1st"
  }
}
```

*   **版ごとに別ID**: ページ番号が変わるため、正確な参照には版の区別が必要。
*   **`replaces` フィールド**: 旧版との関係を記録。

#### 版の選択ガイドライン

| ケース | 推奨 |
| :--- | :--- |
| 一般的な参照 | 最新版を推奨 |
| 削除された内容を引用 | 旧版を使用（引用箇所が新版に存在しない旨を明記） |
| ページ番号が重要 | 使用した版を正確に記載 |
| 版間の違いを論じる | 両方を引用 |

#### 表示

*   版情報を表示に含める（例: `著者 2015（第2版）, p.50`）。
*   執筆支援ツールでは最新版を上位表示し、版の関係を明示。

### 1.9 出典の信頼性階層

出典の信頼性を階層（Tier）で分類し、使用に関するガイドラインを定める。

> **参照**: Tier分類、Tier別使用条件、分野別考慮事項については [CONTENT_POLICY.md §3](../shared/CONTENT_POLICY.md) を参照。

#### レビュワーの責任

| 責任 | 内容 |
| :--- | :--- |
| **Tier確認** | 使用されている出典のTierを確認 |
| **Tier 4の注記確認** | プレプリント使用時に注記があるか確認 |
| **Tier 5の排除** | 原則禁止の出典が使用されていないか確認 |
| **分野適合性** | 分野に適した出典が使用されているか確認 |

### 1.10 データセットの出典管理

Isidoricaの記事やLaboratoryで使用する**例示用データセット**についても、文献と同様に信頼性と透明性を担保する。

#### 背景

統計リテラシー等の記事では、「人口推移」「気温データ」「経済指標」などのデータセットを例示に使用する。これらの信頼性が低い場合、「統計リテラシーを説く記事自体がバイアスを持つ」というブーメラン批判を受けるリスクがある。

#### データソースのTier分類

| Tier | 説明 | 例 |
| :--- | :--- | :--- |
| **Tier 1** | 公的統計機関の公式データ | 総務省統計局、国連、WHO、OECD、世界銀行 |
| **Tier 2** | 学術機関のオープンデータ | UCI Machine Learning Repository、Kaggle（学術系コンペ） |
| **Tier 3** | 政府・自治体のオープンデータ | data.go.jp、e-Stat、各国データポータル |
| **Tier 4** | 民間のオープンデータ（明確なライセンス） | GitHub上のCC0/MIT/CC BYデータ |
| **Tier 5（非推奨）** | 出典不明・ライセンス不明のデータ | 個人ブログ添付CSV、出典未記載のスプレッドシート |

#### 選定原則

| 原則 | 説明 |
| :--- | :--- |
| **Tier 1-3を優先** | 公的機関・学術機関のデータを第一選択とする |
| **ライセンスの明示** | 使用するデータのライセンスを記事内に明記する |
| **永続性の確保** | DOIやアーカイブURLを使用。リンク切れリスクの高いデータは避けるか、ローカルコピーを保持 |
| **バイアスの回避** | 特定の政治的・商業的立場を支持するために意図的にデータを選ばない |
| **更新日の明記** | 使用したデータの取得日・バージョンを記録する |

#### 合成データ（Synthetic Data）の扱い

教育目的で**架空のデータ**を使用する場合のルール：

| 状況 | 許容/禁止 | 例 |
| :--- | :--- | :--- |
| 説明用の架空データ | ✅ 許容 | 「Aさん、Bさん…のテスト結果」 |
| 明確な注記付き | ✅ 必須 | 「これは説明用の架空データです」 |
| 実データの傾向を模したシミュレーション | ✅ 許容 | 正規分布に従う乱数生成 |
| 実データと誤解させる提示 | ❌ 禁止 | 架空データを実在のものとして提示 |
| 特定結論に誘導する操作データ | ❌ 禁止 | 意図的にバイアスを埋め込んだデータ |

#### データセットの引用形式

データセットを使用する場合、以下の情報を記載する：

```markdown
**使用データ**: 
- 出典: 総務省統計局「国勢調査」（2020年版）
- URL: https://www.stat.go.jp/data/kokusei/
- 取得日: 2025-01-15
- ライセンス: CC BY 4.0
```

#### レビュワーの責任

| 責任 | 内容 |
| :--- | :--- |
| **データTier確認** | 使用されているデータセットのTierを確認 |
| **ライセンス確認** | 再利用可能なライセンスか確認 |
| **合成データの注記確認** | 架空データの場合、明確な注記があるか確認 |
| **バイアスチェック** | データ選択に意図的な偏りがないか確認 |

## 2. 執筆支援ツール (Authoring Tools)

執筆者の環境を問わず、出典検索・挿入を簡単にするため、以下のツールを提供する。

### 2.1 エディタ非依存の方針

**推奨エディタは定めない。** 執筆者は好みのエディタ（Vim, Emacs, VS Code, Cursor, その他）を自由に使用できる。

| 方針 | 説明 |
| :--- | :--- |
| **エディタ非依存** | 特定のエディタがないと執筆できない、という状況を避ける |
| **Web検索ページが基本** | 全ての執筆者はWeb検索ページで出典IDを取得可能 |
| **拡張機能はオプション** | VSCode拡張やCLIは利便性向上のために提供、必須ではない |

### 2.2 提供ツール

| ツール | 対象ユーザー | 機能 |
| :--- | :--- | :--- |
| **Web検索ページ** | 全ての執筆者 | ブラウザで出典検索、IDをコピー |
| **Webエディタ** | 非技術者、ブラウザ完結派 | 執筆 + 出典検索・挿入 一体型 |
| **VSCode拡張** | VSCode/Cursor ユーザー | エディタ内で出典検索・挿入 |
| **CLIツール** | Vim/Emacs/ターミナル派 | コマンドで検索、IDを取得 |

### 2.3 検索対象

| データソース | 説明 |
| :--- | :--- |
| **蓄積済み出典** | 過去に使われた出典（`bibliography.json`） |
| **外部API** | Google Books, CiNii, CrossRef から書誌情報を取得 |

### 2.4 執筆者の体験

1.  **検索**: 「ハイデガー 存在と時間」と入力
2.  **結果表示**: 蓄積済み or 外部APIから候補が表示
3.  **選択**: 正しい版を選択
4.  **挿入**: `[@heidegger-being-time, p.45]` が挿入される
5.  **自動蓄積**: 新規出典なら `bibliography.json` に自動追加（ステータス: auto）

### 2.5 実装優先順位

1.  **出典検索API**（全ツールの基盤）
2.  **Web検索ページ**（全ての執筆者をカバー、最小コストで最大効果）
3.  **Webエディタ統合**
4.  **VSCode拡張**（需要に応じて）
5.  **CLIツール**（需要に応じて）

## 4. ブランチ戦略 (Branch Strategy)

本セクションでは `isidorica-library`（コンテンツリポジトリ）のブランチ戦略を定義する。
`isidorica-engine`（システムリポジトリ）のブランチ戦略は別途定義する。

### 4.1 採用戦略

**GitHub Flow（シンプル版）** を採用する。

```
main（本番・公開中のコンテンツ）
  │
  └── feature/xxx または web-edit/xxx（作業ブランチ）
        ↓
      PR作成 → レビュー → mainにマージ
```

*   **Git-flowを採用しない理由**: コンテンツリポジトリには develop, release, hotfix 等の複雑なブランチが不要。シンプルさを優先。
*   **mainブランチ**: 常にデプロイ可能な状態を維持。直接pushは禁止。

### 4.2 ブランチ命名規則

| ユーザータイプ | プレフィックス | 命名例 |
| :--- | :--- | :--- |
| **GitHub利用者** | `feature/` | `feature/prisoner-dilemma-fix-typo` |
| **非GitHub利用者（Web執筆支援ツール経由）** | `web-edit/` | `web-edit/prisoner-dilemma-1702486570` |

#### GitHub利用者

*   推奨パターン: `feature/{article-slug}-{作業内容}`
*   例: `feature/game-theory-add-sources`, `feature/heidegger-fix-citation`
*   強制ではないが、わかりやすい命名を推奨。

#### 非GitHub利用者（Web執筆支援ツール経由）

*   ツール（bot）が自動生成: `web-edit/{article-slug}-{timestamp}`
*   タイムスタンプまたはランダムIDで一意性を保証。
*   ユーザーはブランチ名を意識する必要なし。

### 4.3 PRタイトルの重要性

ブランチ名ではなく、**PRタイトルで作業内容を明確に説明**する。

| 良い例 | 悪い例 |
| :--- | :--- |
| `[prisoner-dilemma] 出典を3件追加` | `修正` |
| `[game-theory] 誤字修正（ナッシュ均衡の説明）` | `update` |

*   レビュワーはPRタイトルを見て内容を判断する。
*   ブランチ名が `web-edit/xxx-1702486570` のように自動生成でも、PRタイトルが明確であれば問題ない。

### 4.4 プレビュー環境

#### PRごとのプレビュー環境（自動生成）

PR作成時に、ホスティングサービス（Cloudflare Pages等）がプレビュー環境を自動デプロイする。

```
PR作成
  ↓
Cloudflare Pages: プレビュー環境を自動デプロイ
  - URL例: https://pr-123.isidorica.pages.dev
  ↓
レビュワー: プレビューURLで実際の表示を確認
  ↓
マージ → 本番デプロイ
```

*   **目的**: レビュワーがローカル環境なしで、実際のサイト表示（CSS、レイアウト、出典表示等）を確認できる。
*   **非技術者にも優しい**: GitHub操作ができなくても、URLをクリックするだけで確認可能。

#### Web執筆支援ツールでのプレビュー

Web執筆支援ツール（エディタ）内でリアルタイムプレビューを提供する。

| 機能 | 内容 |
| :--- | :--- |
| **マークダウンレンダリング** | 見出し、リスト、表等の表示確認 |
| **出典表示** | `[@source-id]` の展開形式を確認 |
| **完全なサイトプレビュー** | 提供しない（PRプレビュー環境で確認） |

#### ローカルプレビュー（GitHub利用者向け）

Engine と Library は**完全に独立したリポジトリ**として管理されている。GitHub利用者は両方をクローンし、ローカルで完全なプレビューを実行できる。

```bash
# isidorica-engine をクローン
git clone https://github.com/isidorica/isidorica-engine.git
cd isidorica-engine

# isidorica-library を content/ にクローン
git clone https://github.com/isidorica/isidorica-library.git content

# 開発サーバー起動
pnpm install
pnpm dev
```

*   `content/` は `.gitignore` に含まれており、Engine リポジトリには含まれない。
*   **詳細**: [SYSTEM_ARCHITECTURE.md](../engine/technical/SYSTEM_ARCHITECTURE.md) を参照。

### 4.5 ロールバック戦略

デプロイ後に問題が発覚した場合のロールバック手順を定義する。

#### 緊急度に応じた対応

| 緊急度 | 対応方法 | 実行者 |
| :--- | :--- | :--- |
| **緊急（即時対応）** | Cloudflare Pages ダッシュボードでロールバック | レビュワー |
| **非緊急（恒久対応）** | Git revert でロールバック、PRを作成 | 執筆者/レビュワー |

#### Cloudflare Pages ダッシュボードからロールバック（即時対応）

```
問題発覚
  ↓
Cloudflare Pages ダッシュボードにアクセス
  ↓
デプロイ履歴から過去の正常なデプロイを選択
  ↓
「ロールバック」をクリック → 即座に反映（数秒〜1分）
```

*   **メリット**: Git操作不要、ダッシュボードのみで完結、非技術者でも実行可能。
*   **用途**: セキュリティ問題、重大な表示崩れ、誤った情報の公開等。

#### Git revert でロールバック（恒久対応）

```
問題発覚
  ↓
問題のコミットを特定
  ↓
git revert <commit-hash> でrevertコミット作成
  ↓
PR作成 → レビュー → マージ
  ↓
自動デプロイ
```

*   **メリット**: 履歴が残る（なぜロールバックしたかが記録される）、正式な手順として追跡可能。
*   **用途**: コンテンツの誤り、出典の問題等。

#### ロールバック後の対応

| ステップ | 内容 |
| :--- | :--- |
| **1. 原因特定** | 問題の原因を調査 |
| **2. 修正PR作成** | 問題を修正したPRを作成 |
| **3. 正式ロールバック記録** | ダッシュボードでロールバックした場合も、後からGit revertで正式に記録 |

### 4.6 緊急削除手順

法的要件やセキュリティ上の理由で、通常のPRプロセスを経ずにコンテンツを削除する必要がある場合の手順を定義する。

> **参照**: 著作権侵害申告への法的対応フローは [LEGAL_REQUIREMENTS.md](./LEGAL_REQUIREMENTS.md) セクション5を参照。

#### 削除が必要なケースの分類

| 緊急度 | ケース | 例 | 対応時間 |
| :--- | :--- | :--- | :--- |
| **🔴 即時対応** | 法的リスク | 著作権侵害申告（明白なケース）、名誉毀損、個人情報漏洩 | 24時間以内 |
| **🔴 即時対応** | セキュリティ | 悪意のあるコード、フィッシングリンク | 数時間以内 |
| **🟡 迅速対応** | 重大な誤情報 | 生命・健康に関わる誤情報、詐欺的な内容 | 48時間以内 |
| **⚪ 通常対応** | 品質問題 | 出典の誤り、軽微な誤情報 | 通常のPRプロセス |

#### 緊急削除の権限

| 緊急度 | 削除権限者 | 事前承認 |
| :--- | :--- | :--- |
| **🔴 即時対応** | Core Maintainer（単独） | 不要（事後報告） |
| **🟡 迅速対応** | Core Maintainer | 可能な限り事前確認 |
| **⚪ 通常対応** | レビュワー | 通常のPRプロセス |

#### 緊急削除フロー

```
問題の発見
    ↓
緊急度の判断（Core Maintainer）
    ↓
【🔴 即時対応】
    ├─ Cloudflare Pagesダッシュボードでロールバック（数秒〜1分）
    ├─ または: Git revert + 強制デプロイ
    ├─ 記録（Issue作成：削除理由、対象、日時）
    └─ 事後報告（発信者への通知、必要に応じてステークホルダーへ）
    ↓
【🟡 迅速対応】
    ├─ 対象記事の非公開化（Frontmatterで draft: true に変更）
    ├─ 発信者への確認（可能であれば）
    └─ 48時間以内に対応完了
    ↓
【⚪ 通常対応】
    └─ 通常のPRプロセスで修正
```

#### LEGAL_REQUIREMENTS.mdとの連携

| 法的要件 | ガバナンス上の対応 |
| :--- | :--- |
| **著作権侵害申告（DMCA等）** | LEGAL_REQUIREMENTS.md セクション5のフローに従い、明白な侵害は即時対応 |
| **DSAの対応期限** | EU DSAの対応期限（24時間〜）を考慮した即時対応体制 |
| **異議申立て期間** | 削除後も復元の可能性を考慮し、コンテンツをバックアップ |

#### 記録と透明性

| 項目 | 内容 |
| :--- | :--- |
| **削除記録** | 全ての緊急削除をGitHub Issueで記録（理由、対象、日時、対応者） |
| **透明性レポート** | 年次で緊急削除件数・理由を公開（DSA対応、LEGAL_REQUIREMENTS.md セクション5.8参照） |
| **復元記録** | 異議申立てによる復元も記録 |
| **バックアップ保持** | 削除コンテンツは1年間バックアップを保持（異議申立て対応のため） |

### 4.7 放棄されたPR/Issueの扱い

長期間更新がないPRやIssueの扱いを定義する。

#### 放棄されたPR

##### ドラフトPR vs 通常PR

| 種別 | 意味 | 期待される状態 |
| :--- | :--- | :--- |
| **ドラフトPR** | 作業中、レビュー不要 | 執筆者が積極的に作業している |
| **通常PR** | レビュー準備完了 | レビュワーの対応を待っている |

##### 放棄判定の期間

| 種別 | リマインダー | Staleラベル | クローズ |
| :--- | :--- | :--- | :--- |
| **ドラフトPR** | 60日 | 90日 | 120日 |
| **通常PR** | 30日 | 45日 | 60日 |

##### 対応フロー

```
【ドラフトPR】
60日更新なし → Bot がリマインダー
90日更新なし → `stale` ラベル付与
120日更新なし → クローズ（再開可能）

【通常PR】
30日更新なし → Bot がリマインダー
45日更新なし → `stale` ラベル付与
60日更新なし → クローズ（再開可能）
```

##### 例外

| 条件 | 対応 |
| :--- | :--- |
| `wip` または `on-hold` ラベル | Stale処理をスキップ |
| クローズ後の再開希望 | コメントで依頼 → 再オープン |
| レビュワー側の放置 | 通常PRがレビュワー待ちで14日経過 → Core Maintainerに通知 |

#### 放棄されたIssue（報告）

報告されたが対応者がいないIssue。

| 期間 | 対応 |
| :--- | :--- |
| **14日間** | `needs-triage` ラベル維持 |
| **30日間** | Core Maintainerがレビュー、優先度を判断 |
| **90日間** | 対応不要と判断された場合 `wontfix` でクローズ |

> **注意**: セキュリティや法的問題のIssueは即時対応（セクション4.6参照）

#### 放棄されたIssue（対応中）

アサインされた対応者が長期間活動しない場合。

| 期間 | 対応 |
| :--- | :--- |
| **30日間更新なし** | Bot がアサイニーにリマインダー |
| **45日間更新なし** | 他の対応者を募集（コメント） |
| **60日間更新なし** | アサインを解除、`help-wanted` ラベル付与 |

#### 自動化

| 機能 | 方法 |
| :--- | :--- |
| **Stalebot** | GitHub Actions または `actions/stale` で自動化 |
| **リマインダー** | 定期実行（週1回）でチェック |
| **ラベル管理** | Bot が自動付与・削除 |

#### ラベル設計

| ラベル | 意味 |
| :--- | :--- |
| `stale` | 長期間更新なし |
| `wip` | 作業中（stale処理をスキップ） |
| `on-hold` | 意図的に保留（stale処理をスキップ） |
| `needs-triage` | 対応者・優先度未決定 |
| `help-wanted` | 対応者を募集中 |
| `wontfix` | 対応しないと判断 |

## 5. レビュープロセス (Review Process)

### 5.1 新規記事のワークフロー

詳細なワークフローは [USER_SCENARIOS.md](./USER_SCENARIOS.md) の執筆者セクション（UC-A1〜A6）を参照。

#### 概要

1.  執筆者が記事を執筆し、出典を執筆支援ツールで検索・挿入
2.  PRを作成
3.  CIが新規出典を検出、衝突チェック（早期フィードバック）
4.  レビュワーが記事内容・出典をレビュー
5.  CIがマージ前に最終衝突チェック
6.  マージ時に出典ステータスが auto → curated に更新

### 5.1.1 Laboratoryを含む記事のレビュー

Laboratoryを含む記事は、記事本文に加えて以下の追加レビューが必要。

| 観点 | 確認事項 |
| :--- | :--- |
| **セキュリティ** | LAB-CODEに悪意のあるコードが含まれていないか |
| **教育的価値** | 記事の学習目標にLaboratoryが寄与しているか |
| **アクセシビリティ** | キーボード操作、スクリーンリーダー対応が適切か |
| **正確性** | シミュレーション結果が正確か |

> **詳細**: [LABORATORY_DESIGN.md](../engine/LABORATORY_DESIGN.md) セクション5.5「品質管理」を参照。

### 5.2 翻訳版のワークフロー

翻訳版のワークフロー、機械翻訳の使用ポリシー、文化適応については [TRANSLATION_DESIGN.md](./TRANSLATION_DESIGN.md) セクション2〜3を参照。

### 5.3 編集衝突の対応

複数の執筆者が同じ記事を同時に編集した場合の対応。

| 項目 | 対応 |
| :--- | :--- |

## 7. 出典の誤り発見・修正 (Source Error Correction)

### 7.1 報告チャネル

報告は**GitHub Issue に一元化**する。

| 報告者 | 方法 |
| :--- | :--- |
| GitHub利用者 | 直接Issueを作成 |
| 非GitHub利用者 | 報告フォーム → ボット（`isidorica-bot`）がIssue自動作成 |

*   報告フォームの設計・実装詳細は別途定義（フォーム設計ドキュメント）。
*   スパム対策にはhCaptcha等のプライバシー重視ソリューションを使用。

### 7.2 誤りの種類と修正フロー

| 誤りの種類 | 例 | 修正方法 |
| :--- | :--- | :--- |
| **出典の選択ミス** | ソレルと間違えてベンヤミンを引用 | 記事内のIDを正しいものに変更 |
| **ページ番号の誤り** | p.45のつもりがp.54 | 記事内のページ番号を修正 |
| **APIデータの誤り** | 著者名の誤字、出版年の誤り | `bibliography.json` を手動修正 |
| **書誌情報の欠落** | 翻訳者名がない | `bibliography.json` に情報を追加 |

### 7.3 bibliography.json の手動修正

APIデータに誤りがある場合、手動で修正する。

*   **ステータス**: `corrected`（修正済み）に変更。
*   **修正履歴**: 何を、いつ、なぜ修正したかを `corrections` フィールドに記録。
*   **API元への報告**: 可能であればAPI提供元にも誤りを報告（長期的な改善のため）。

```json
{
  "source-id": {
    "status": "corrected",
    "corrections": [
      {
        "date": "2025-12-12",
        "field": "translator",
        "original": null,
        "corrected": [{"family": "原", "given": "佑"}],
        "reason": "APIデータに翻訳者情報が欠落していた"
      }
    ]
  }
}
```

### 7.4 出典の削除

#### 削除が必要なケース

| ケース | 説明 |
| :--- | :--- |
| **不適切な出典** | 主張と無関係、信頼性のない出典 |
| **存在しない資料** | 実在しない資料（手動入力時のレビュー漏れ） |
| **重複** | 同じ出典が複数IDで登録されている |

#### 削除フロー

詳細なワークフローは [USER_SCENARIOS.md](./USER_SCENARIOS.md) の報告者セクション（UC-P5）を参照。

**概要**:
1.  報告者がIssue/フォームで削除を提案
2.  レビュワーが確認（削除の必要性、他記事での使用状況）
3.  判断: 記事から外す / deprecatedに / 削除しない

#### 記事からの削除と bibliography.json からの削除

| 操作 | 影響範囲 | 対応 |
| :--- | :--- | :--- |
| 記事から出典を外す | その記事のみ | 通常の記事修正として対応 |
| bibliography.json から削除 | 全記事に影響 | 慎重に対応、まず deprecated に |

#### deprecated ステータス

物理削除ではなく「非推奨化」を行う。

```json
{
  "source-id": {
    "status": "deprecated",
    "deprecated_reason": "重複 → source-id-2 を使用してください",
    "deprecated_date": "2025-12-12"
  }
}
```

*   **新規使用禁止**: deprecated の出典は執筆支援ツールで非表示/警告。
*   **既存参照は維持**: deprecated化しても、既存の記事からの参照は維持される。

#### 物理削除

deprecated化された出典がどの記事からも参照されなくなった場合、物理削除を行う。

##### 使用状況の確認方法

ビルド時に全記事の出典使用状況を `sources/usage-index.json` に自動生成する。

```json
{
  "sources": {
    "heidegger-being-time": ["philosophy/existentialism", "philosophy/phenomenology"],
    "tanaka-game-theory": ["economics/game-theory"]
  },
  "generated_at": "2025-12-13T12:00:00Z"
}
```

*   **ビルド時に自動生成**: remark/rehypeプラグインが各記事の出典IDを抽出し、インデックスを生成。
*   **追加コストなし**: ビルド処理の一部として実行されるため、別途スキャンは不要。

##### 物理削除の手順

| ステップ | 内容 |
| :--- | :--- |
| **1. 定期チェック** | 月1回のCIジョブで `usage-index.json` と `bibliography.json` を照合 |
| **2. 削除候補抽出** | `deprecated` かつ `usage-index` に存在しない出典をリストアップ |
| **3. 自動PR作成** | 削除候補を `bibliography.json` から削除するPRを自動作成 |
| **4. レビュワー確認** | PRをレビュー・マージ |

*   **猶予期間なし**: deprecated化の時点で削除判断は完了しているため、未使用であれば即削除。
*   **再追加は容易**: 誤って削除した場合も、外部APIまたは手動入力で再追加可能。

#### 構造的な問題（過剰出典）

*   一文ごとに同じ出典を繰り返すなど、明らかに冗長な出典記載。
*   出典の削除ではなく**記事の編集**として対応。
*   レビュワーが判断し、適切な出典配置に修正。

## 8. 記事の更新・削除ポリシー (Update & Deletion Policy)

### 8.1 更新

*   内容の更新は原則としてオリジナル言語版で行い、翻訳版に反映する。
*   誤字脱字の修正等の軽微な修正は翻訳版で直接行ってよい。
*   翻訳版独自の修正（翻訳改善等）は許可されるが、内容の変更はオリジナル版に提案する。

### 8.2 削除

*   記事の削除は慎重に行う。原則として「非公開」ステータスへの移行を優先する。
*   削除理由を記録し、削除済み記事の一覧を管理する。
*   外部からのリンク切れを最小化するため、301リダイレクトを設定する。

### 8.3 定期更新ポリシー（見直し推奨ポリシー）

コンテンツの陳腐化を防ぐため、定期的な見直しを促進する仕組みを導入する。

#### 基本方針

| 項目 | 内容 |
| :--- | :--- |
| **トリガー** | 時間ベース（最終更新からの経過期間） |
| **データソース** | Git履歴（`git log` の最終コミット日） |
| **自動化** | GitHub Actions + Bot による自動Issue作成 |
| **Frontmatter追加** | 不要（執筆者の負担なし） |

#### コンテンツタイプと陳腐化速度

| コンテンツタイプ | 陳腐化速度 | 例 |
| :--- | :--- | :--- |
| **テクノロジー** | 非常に速い（1-2年） | フレームワーク、API仕様 |
| **法制度・規制** | 中程度（法改正時） | 著作権法、GDPR |
| **科学の最前線** | 中程度 | AI研究、量子コンピュータ |
| **確立された科学概念** | 遅い（ほぼ不変） | ニュートン力学、微積分 |
| **哲学・思想史** | 非常に遅い | カントの哲学、プラトン |
| **歴史的事実** | ほぼ不変 | 第二次世界大戦の経緯 |

#### 分野別閾値

閾値は `primary_discipline` または `topics` に基づいて決定する。

| 分野 | 閾値 | 説明 |
| :--- | :--- | :--- |
| **technology** | 1年 | テクノロジー分野は変化が速い |
| **law** | 2年 | 法改正に注意が必要 |
| **psychology** | 3年 | 研究の進展に応じて更新 |
| **natural-science** | 3年 | 確立された概念は安定 |
| **philosophy** | 5年 | 哲学・思想史は安定 |
| **history** | 8年〜10年 | 歴史的事実は陳腐化しない |
| **（デフォルト）** | 3年 | 上記に該当しない分野 |

#### 自動Issue作成の仕組み

```

     │

     │

     │
     ├─ 4. 既存Issueがなければ自動作成
     │

```

##### 見直し優先度の計算ロジック

見直しIssue作成時に、Bot（GitHub Actions）が各記事の優先度スコアを自動計算し、対応する優先度ラベルを自動付与する。人間が手動でラベルを付ける必要はない（必要に応じて後からオーバーライドは可能）。

週次サマリーおよび個別Issueの優先度は、以下のロジックで計算する。

| 要因 | 最大点 | 計算方法 |
| :--- | :--- | :--- |
| **学習パス含有** | 30点 | `learning-paths.yml` に含まれる回数 × 10（上限30） |
| **被依存数** | 20点 | `concept-relations.yml` で `type: requires` される回数 × 5（上限20） |
| **アクセス数** | 20点 | 月間PV / 500（上限20）※将来実装 |

**計算式**:
```
Review Priority Score = 
  min(学習パス数 × 10, 30) +
  min(被依存数 × 5, 20) +
  min(月間PV / 500, 20)  ← 将来実装
```

> **注意**: 翻訳優先度とは異なり、「安定性」は含まない。見直し対象は定義上「古い」記事であるため。

##### 優先度ラベルへのマッピング

| スコア | ラベル | 意味 |
| :--- | :--- | :--- |
| **60-100** | `priority:critical` | 学習パスの中核記事。最優先で見直し |
| **40-59** | `priority:high` | 重要な記事。優先的に見直し |
| **20-39** | `priority:medium` | 通常の記事 |
| **0-19** | `priority:low` | 単独記事。時間があるときに見直し |

##### 優先度ラベルの用途

| 用途 | 説明 |
| :--- | :--- |
| **フィルタリング** | Issue一覧で `label:priority:critical` でフィルタ |
| **対応順序** | 優先度の高いものから対応 |
| **ダッシュボード** | GitHub Projects で優先度ごとにグルーピング |

> **注意**: 個別Issue化までの期間は優先度によらず4週で統一。優先度は対応順序の参考として使用。

#### Issue重複防止ロジック

無駄なIssueの作成を防ぐため、以下のロジックで重複を検出する。

##### ラベル設計

| ラベル | 用途 | 付与者 |
| :--- | :--- | :--- |
| `review-needed` | 見直しが必要 | Bot（自動） |
| `auto-generated` | Botによる自動生成 | Bot（自動） |
| `article:<slug>` | 記事を特定（例: `article:react-hooks`） | Bot（自動） |

##### シナリオ別対応

| シナリオ | 状況 | 対応 |
| :--- | :--- | :--- |
| **A** | 同じ記事に対してオープンな見直しIssueが存在 | Issue作成をスキップ |
| **B** | 記事を更新してのIssueをクローズ | Git履歴が更新されるので自動解決 |
| **C** | 「更新不要」と判断してのIssueをクローズ | Issueクローズ日から閾値を再計算 |
| **D** | 該当記事を変更するPRが進行中 | Issue作成をスキップ |

##### シナリオA: オープンIssueの判定

手動で作成されたIssue（誤字報告等）と区別するため、Bot専用のラベルで判定する。

```
判定条件:
- state: open
- labels に `review-needed` AND `auto-generated` を含む
- labels に `article:<slug>` を含む
→ 条件を満たすIssueが存在すれば作成をスキップ
```

##### シナリオC: 「更新不要」クローズからの閾値再計算

記事を更新せずに「確認済み」としてIssueをクローズした場合、クローズ日を基準日とする。

```
基準日の計算:
1. Git log から記事の最終更新日を取得
2. `article:<slug>` + `review-needed` ラベルを持つ
   最新のクローズ済みIssueを検索
3. 基準日 = max(Git最終更新日, Issueクローズ日)
4. 現在日 - 基準日 > 閾値 → Issue作成
```

##### シナリオD: PR進行中の判定

記事の更新PRが作成されているがまだマージされていない場合、Issue作成をスキップする。

```
判定条件:
1. GitHub API で全オープンPRを取得（ドラフトPRも含む）
2. 各PRの変更ファイルリストを取得
3. 該当記事ファイル（例: content/articles/**/react-hooks.md）が
   変更対象に含まれるかチェック
→ 含まれていればIssue作成をスキップ
```

##### 統合チェックフロー

```
毎月1日: GitHub Actions 実行
     │
     ├─ 1. 全記事をリストアップ
     │
     ├─ 2. オープンPRの変更ファイルリストを一括取得（キャッシュ）
     │
     ├─ 3. 各記事について:
     │      │
     │      ├─ 3a. オープンPRに含まれるか → 含まれればスキップ
     │      │
     │      ├─ 3b. `article:<slug>` ラベルを持つIssueを検索
     │      │       - オープン → スキップ
     │      │       - クローズ → クローズ日を取得
     │      │
     │      ├─ 3c. Git log から最終更新日を取得
     │      │
     │      ├─ 3d. 基準日 = max(Git更新日, Issueクローズ日)
     │      │
     │      ├─ 3e. 閾値超過判定
     │      │       - 超過 → Issue作成
     │      │       - 未超過 → スキップ
     │      │
     └─ 4. 結果サマリーをログ出力
```

#### 対応フロー

| ステップ | 担当 | 内容 |
| :--- | :--- | :--- |
| **1. Issue確認** | レビュワー/執筆者 | 自動作成されたIssueを確認 |
| **2. 内容レビュー** | 執筆者 | 記事の内容が現在も正確かを確認 |
| **3a. 更新必要** | 執筆者 | PRを作成して更新、Issueにリンク |
| **3b. 更新不要** | 執筆者 | 理由をコメントしてIssueをクローズ |
| **4. Issue解決** | - | PRマージまたはクローズでIssue解決 |

#### 読者向け表示（オプション）

閾値を超過した記事には、記事ヘッダーに控えめな警告を表示することを検討する。

```
⏳ この記事は2年以上更新されていません。最新情報をご確認ください。
   最終更新: 2022年3月15日
```

| 表示方針 | 説明 |
| :--- | :--- |
| **控えめなデザイン** | 学術的な信頼性を損なわない範囲で表示 |
| **リンク提供** | 「最新版を確認」「問題を報告」等のリンク |
| **閾値未超過は表示なし** | 通常の記事には表示しない |

#### 翻訳優先度

オリジナル記事が更新された際、どの記事から優先的に翻訳を進めるべきかを判断するためのスコアを自動計算する。

##### 優先度スコアの計算ロジック

| 要因 | 最大点 | 計算方法 |
| :--- | :--- | :--- |
| **学習パス含有** | 30点 | `learning-paths.yml` に含まれる回数 × 10（上限30） |
| **被依存数** | 20点 | `concept-relations.yml` で `type: requires` される回数 × 5（上限20） |
| **安定性** | 10点 | 6ヶ月以上更新なし → 10点、3ヶ月以上 → 5点 |
| **アクセス数** | 20点 | 月間PV / 500（上限20）※将来実装 |

**計算式**:
```
Translation Priority Score = 
  min(学習パス数 × 10, 30) +
  min(被依存数 × 5, 20) +
  安定性ボーナス +
  min(月間PV / 500, 20)  ← 将来実装
```

##### 優先度ラベルへのマッピング

| スコア | ラベル | AI翻訳フォールバック | 説明 |
| :--- | :--- | :--- | :--- |
| **80-100** | `priority:critical` | 30日 | 学習パスの中核、多数の記事から依存 |
| **50-79** | `priority:high` | 45日 | 重要な記事 |
| **20-49** | `priority:medium` | 60日 | 通常の記事 |
| **0-19** | `priority:low` | 90日 | 単独記事、依存少 |

##### 手動オーバーライド（PRチェックボックス）

執筆者が記事更新PRを作成する際、翻訳優先度を明示的に指定できる。Frontmatter には記載せず、PRの情報として管理することで、記事を汚染しない。

PRテンプレートにはチェックボックスが用意されており、執筆者が `[x]` をチェックすると GitHub Actions がこれを読み取り、優先度を設定する。

| チェックボックス | 効果 | AI翻訳フォールバック |
| :--- | :--- | :--- |
| 🚨 `critical` | 優先度スコアを100に設定 | 30日後 |
| ⬆️ `high` | 優先度スコアを80に設定 | 45日後 |
| ⬇️ `low` | 優先度スコアを20に設定 | 90日後 |
| 🚫 `none` | 翻訳Issueを作成しない | - |
| 未チェック | 自動計算 | スコアに応じて決定 |

> **参照**: PRテンプレートの詳細は [GITHUB_TEMPLATES.md](../shared/GITHUB_TEMPLATES.md) を参照。

##### Web 執筆支援ツールでの指定

Web 執筆支援ツールでも同様の選択肢を提供する。

| 選択肢 | 効果 |
| :--- | :--- |
| 自動計算（推奨） | 学習パス含有数・被依存数等から自動計算 |
| 🚨 緊急（critical） | 優先度スコア100、30日後にAI翻訳 |
| ⬆️ 高（high） | 優先度スコア80、45日後にAI翻訳 |

| 🚫 対象外（none） | 翻訳Issueを作成しない |

#### 翻訳リソース制約

翻訳のスケールと品質を両立するため、リソース制約を考慮した段階的な方式を採用する。

##### 現状の制約

| リソース | 状況 |
| :--- | :--- |
| **レビュー可能者** | 管理者1名（日本語・英語、限定的にドイツ語・フランス語） |
| **専任翻訳者** | なし |
| **AI翻訳** | 利用可能（GPT-4, Claude等） |

##### 方式の選択

| フェーズ | 方式 | 説明 |
| :--- | :--- | :--- |
| **Phase 5-6** | AI翻訳 + 必須レビュー | 管理者が対応可能な言語（英語）のみ。AI翻訳を活用するが、人間レビューを必須とする |
| **将来（コントリビューター確保後）** | コミュニティ翻訳 | 言語ごとにレビュワーを任命し、翻訳者コミュニティを形成 |

##### 原則

| 原則 | 説明 |
| :--- | :--- |
| **レビュー必須** | AI翻訳を使用する場合でも、人間によるレビューが必須 |
| **言語制限** | レビュー可能者がいない言語への翻訳は行わない |
| **段階的拡大** | 言語追加はリソース確保が前提、品質を維持しながら段階的に拡大 |

##### 翻訳レビュワーの役割

| 役割 | 責任 |
| :--- | :--- |
| **言語レビュワー** | 特定言語の翻訳品質を保証。翻訳PRのレビューを担当 |
| **最終承認者** | PRのマージ権限。品質が基準を満たしていることを確認 |

##### 翻訳レビュワーの要件

| 要件 | 説明 |
| :--- | :--- |
| **言語能力** | 翻訳対象言語のネイティブまたは上級者（C1以上） |
| **原文理解** | オリジナル言語（日本語もしくは英語）の読解能力 |
| **技術スキル** | GitHubの基本操作（PR、レビュー、コメント） |
| **コミットメント** | 定期的なレビュー対応（週1回程度） |

> **参照**: 多言語対応ロードマップは [PROJECT_ROADMAP.md](../shared/PROJECT_ROADMAP.md) のPhase 5を参照。

#### 翻訳版への波及

オリジナル記事が更新された場合、翻訳版も見直しが必要になる。

##### シナリオ

| シナリオ | 状況 | 対応 |
| :--- | :--- | :--- |
| **E** | オリジナル版が更新された | 翻訳版に自動でIssue作成 + 読者向け警告表示 |
| **F** | 翻訳Issueが作成されたが対応者なし | 閾値超過後、AI翻訳でドラフトPR作成 |

##### オリジナル更新時の自動Issue作成（シナリオE）

オリジナル記事を更新するPRがマージされたとき、翻訳版に対してIssueを自動作成する。

```
PR マージ（オリジナル記事を更新）
     │
     ├─ 1. GitHub Actions がトリガー
     │
     ├─ 2. マージされたPRの変更ファイルを取得
     │
     ├─ 3. 記事ファイルかどうかを判定
     │
     ├─ 4. 同じslugを持つ翻訳版を特定
     │      （例: ja/philosophy/kant.md → en/philosophy/kant.md）
     │
     ├─ 5. 各翻訳版に対してIssueを作成
     │      - ラベル: `translation-needed`, `auto-generated`, `article:<slug>`
     │
     └─ 6. 翻訳版に「オリジナルが更新されました」警告フラグを設定
```

**翻訳版の特定方法**:

ディレクトリ構造（言語ベース + 主題サブディレクトリ）を活用する。

```
content/articles/
├── ja/philosophy/kant-ethics.md  ← オリジナル
└── en/philosophy/kant-ethics.md  ← 翻訳版（同じslug）
```

> **参照**: ディレクトリ構造の詳細は [URL_AND_TAXONOMY_DESIGN.md](./technical/URL_AND_TAXONOMY_DESIGN.md) を参照。

**自動生成されるIssueの内容**:

```markdown
## 🌐 翻訳版の更新が必要です

| 項目 | 値 |
|:-----|:---|
| **記事** | [react-hooks](/en/react-hooks) |
| **言語** | English |
| **オリジナル更新日** | 2024-03-15 |
| **オリジナル更新PR** | #456 |
| **翻訳版最終更新** | 2023-06-01 |

## アクション

- [ ] 翻訳版を確認し、オリジナルの変更を反映
- [ ] PRを作成して更新
- [ ] 翻訳が不要な場合（軽微な変更等）は理由をコメントしてクローズ
```

##### AI翻訳によるフォールバック（シナリオF）

翻訳Issueが一定期間（閾値）対応されない場合、AIによる自動翻訳でドラフトPRを作成する。

**段階的エスカレーション**:

| 段階 | 経過期間 | 対応 |
| :--- | :--- | :--- |
| **1. Issue作成** | 0日 | 翻訳更新Issueを作成 |
| **2. 人間対応待ち** | 0〜30日 | 人間の翻訳者を待つ |
| **3. リマインダー** | 30日 | Issueにコメント「対応をお願いします」 |
| **4. AI翻訳トリガー** | 60〜90日 | Bot がAI翻訳を実行、ドラフトPR作成 |
| **5. レビュー** | - | 人間がレビュー、必要に応じて修正 |
| **6. マージ** | - | レビュー完了後マージ |

**AI翻訳ワークフロー**:

```
翻訳Issueが閾値（例: 60日）を超過
     │
     ├─ 1. Bot がオリジナル記事の差分を取得
     │      （前回翻訳時点からの変更）
     │
     ├─ 2. AI（GPT-4, Claude等）で翻訳を生成
     │
     ├─ 3. ドラフトPRを自動作成
     │      - タイトル: [AI Translation] react-hooks (en)
     │      - ラベル: `ai-generated`, `needs-review`
     │
     ├─ 4. IssueにPRリンクをコメント
     │
     └─ 5. レビュワーに通知
```

**品質担保**:

| 対策 | 説明 |
| :--- | :--- |
| **ドラフトPR** | 自動マージはしない、必ずレビューを経る |
| **AIラベル** | `ai-generated` ラベルで明示 |
| **差分ベース翻訳** | 全文翻訳ではなく、変更部分のみ翻訳（コスト削減、精度向上） |
| **人間レビュー必須** | レビュワーが承認しなければマージされない |

**翻訳方法の透明性**:

Frontmatterで翻訳方法を記録し、読者に明示する。

```yaml
translation_method: ai-assisted  # human, ai-assisted, ai-only
translation_reviewed_by: username
```

| 値 | 意味 |
| :--- | :--- |
| `human` | 人間による翻訳 |
| `ai-assisted` | AI翻訳 + 人間レビュー |
| `ai-only` | AI翻訳のみ（非推奨、緊急時のみ） |

> **実装**: 設定ファイル（`config/review-policy.yml`）とGitHub Actionsワークフローの作成はPhase 4で行う。

## 9. 記事テンプレート (Article Templates)

### 9.1 概要

新規記事作成時に、カテゴリごとに一貫した構造とフロントマターを提供するため、記事テンプレート機能を提供する。

| 目的 | 説明 |
| :--- | :--- |
| **一貫性の確保** | カテゴリごとに統一された記事構造を維持 |
| **参入障壁の低下** | フロントマター構文を知らなくても記事作成可能 |
| **エラー防止** | 必須項目の欠落、typoを防止 |

### 9.2 テンプレート提供方法

| ツール | 提供方法 |
| :--- | :--- |
| **Web執筆支援ツール** | フォーム形式でフロントマター入力、カテゴリ選択で構造を自動生成 |
| **CLIツール** | `isidorica new article --category=philosophy` でテンプレートからファイル生成 |

### 9.3 テンプレート定義

テンプレートは `templates/` ディレクトリにMarkdown形式で保存する。テンプレート自体が記事のサンプルとして機能する。

#### 定義例（哲学記事）

```markdown
<!-- templates/philosophy.md -->
---
title: [タイトルを入力]
description: [説明を入力]
topics: []
original_language: ja
draft: true
---

## 概要

この概念の概要を記述してください。

## 歴史的背景

この概念の歴史的背景を記述してください。

## 主要な議論

主要な議論や論点を記述してください。

## 参考文献

<!-- 出典は自動生成されます -->
```

*   **メリット**: テンプレートがそのまま記事のサンプルとして機能。執筆者に馴染みやすい。
*   **保存場所**: `isidorica-library/templates/`

### 9.4 テンプレートのカスタマイズ

*   テンプレートの追加・変更はPRで管理
*   詳細なワークフローは [USER_SCENARIOS.md](./USER_SCENARIOS.md) を参照

### 9.5 未決定事項

以下の項目はフロントマター仕様の確定後に決定する。

| 項目 | 状況 | 備考 |
| :--- | :--- | :--- |
| **カテゴリとテンプレートの関係** | 未決定 | 1:1なのか、カテゴリごとに複数テンプレートを持てるのか |
| **フロントマター必須/任意項目の管理** | 未決定 | テンプレート内のコメントで管理？別ファイルで管理？ |
| **共有方法** | 未決定 | Web/CLIが同じテンプレートを参照する技術的な仕組み |

## 10. 品質基準 (Quality Standards)

### 10.1 必須要件

| 要件 | 内容 |
| :--- | :--- |
| **出典** | 全ての主張に検証可能な出典を付与 |
| **正確性** | 事実に基づく記述、誤解を招く表現の排除 |
| **中立性** | 特定の思想・立場に偏らない |
| **アクセシビリティ** | WCAG 2.2 AA準拠、スクリーンリーダー対応 |

### 10.2 推奨事項

| 推奨 | 内容 |
| :--- | :--- |
| **平易な表現** | 専門用語には説明を付与 |
| **構造化** | 見出し、リスト、表を活用した読みやすい構造 |
| **国際的な出典** | 可能な限り複数言語で入手可能な出典を優先 |

### 10.3 画像・図表の出典

| ケース | 出典要件 |
| :--- | :--- |
| **自作の図表** | 出典不要（ただし元データがあれば明記） |
| **外部からの引用** | 出典必須 + 許諾取得 |
| **パブリックドメイン** | 出典明記（許諾不要） |
| **クリエイティブ・コモンズ** | ライセンス条件に従う |

#### 許諾の責任

| 役割 | 責任 |
| :--- | :--- |
| **執筆者** | 許諾を取得し、証拠をPRに添付 |
| **レビュワー** | 許諾の証拠を確認 |

*   **出典記法**: [MARKDOWN_SPEC.md](./technical/MARKDOWN_SPEC.md) を参照。
*   **許諾記録の保存方法**: [LEGAL_REQUIREMENTS.md](./LEGAL_REQUIREMENTS.md) を参照。

### 10.4 動画・音声の引用

#### 基本方針

| 項目 | 決定 |
| :--- | :--- |
| **記事本文への埋め込み** | **禁止** |
| **外部リンク** | 許可（タイムスタンプ付き） |
| **音声** | 同様に埋め込み禁止、リンクのみ |

#### 理由

*   **洗練されたミニマリズム**: 外部動画はIsidoricaのデザイン言語と統一できない。
*   **質のコントロール**: 動画の品質・広告・UIをIsidoricaが保証できない。
*   **読書体験の維持**: テキストの流れを中断しない。
*   **Laboratory**: インタラクティブ体験はLaboratoryで独自に提供。

#### 出典としての参照

動画・音声を出典として参照する場合はテキストリンクを使用する。

```markdown
この演説の詳細は映像で確認できる[@kennedy-inaugural, t=3:45]。
```

| 記法 | 意味 |
| :--- | :--- |
| `t=3:45` | タイムスタンプ（3分45秒） |
| `t=1:23:45` | 長時間動画（1時間23分45秒） |

#### 歴史的映像の例外（検討中）

歴史的映像（パブリックドメイン、一次資料）は**別ページ**で提供する。

| 方式 | 説明 |
| :--- | :--- |
| **記事本文** | テキストリンクのみ（ミニマリズム維持） |
| **アーカイブページ** | `/archives/videos/` で映像を集約 |
| **内容** | 映像埋め込み + 書き起こし + 文脈説明 |

**書き起こしツール（検討中）**:
*   品質統一のため、Whisper（OpenAI）ベースの書き起こしツールを検討中。
*   詳細は [PROJECT_ROADMAP.md](../shared/PROJECT_ROADMAP.md) の将来の検討事項を参照。

### 10.5 コードスニペットの出典

#### 自作コード

*   **出典不要**: 記事のために執筆者が作成したコード。
*   Isidoricaのライセンスで公開される。

#### 引用コード（出典元別の管理）

| 出典元 | 管理方法 | 記法 |
| :--- | :--- | :--- |
| **書籍・論文** | `bibliography.json` | `[@source-id, p. 45]` |
| **公式ドキュメント** | `bibliography.json`（Webページとして） | `[@mdn-fetch]` |
| **OSS・GitHub** | インライン（コメント） | `# Source: URL` + `# License:` |
| **Stack Overflow** | インライン（コメント） | `# Source: URL` + `# CC BY-SA` |

#### OSS・GitHubからの引用

```python
# Source: https://github.com/example/repo/blob/abc123/src/main.py#L10-L20
# License: MIT
def hello():
    print("Hello, World!")
```

*   **コミットハッシュ**: 特定バージョンを固定するため、コミットハッシュを含める。
*   **ライセンス**: ライセンス条件を明記。

#### Stack Overflowからの引用

```javascript
// Source: https://stackoverflow.com/a/12345678
// Author: username
// License: CC BY-SA 4.0
function example() { }
```

*   **CC BY-SA**: Stack Overflowの回答はCC BY-SA 4.0でライセンスされている。

## 11. 紛争解決 (Dispute Resolution)

### 11.1 根本原則

紛争解決は**「誰が正しいか」ではなく「何が根拠か」**で判断する。

| 原則 | 説明 |
| :--- | :--- |
| **出典ベースの議論** | 「私はこう思う」ではなく「この出典によれば」で議論する |
| **人格への攻撃禁止** | 匿名原則の延長。執筆者個人への批判は禁止 |
| **記事単位での対応** | 問題が発覚しても個人を特定して責任追及しない（共同執筆の原則） |

### 11.2 紛争の種類

| 種類 | 例 |
| :--- | :--- |
| **事実の正確性** | 「この数値は誤りではないか」「この事件の年代が違う」 |
| **解釈の妥当性** | 「この出典はこの主張を支持していない」 |
| **中立性** | 「この記述は特定の立場に偏っている」 |
| **出典の信頼性** | 「この情報源は信頼に足るか」 |
| **ガバナンスルールの解釈** | 「このケースはルールXに該当するか否か」 |

### 11.3 解決フロー（段階的エスカレーション）

| 段階 | 場所 | 対応者 | 期間目安 |
| :--- | :--- | :--- | :--- |
| **1. PRコメントでの議論** | GitHub PR | 当事者（執筆者、レビュワー） | 〜1週間 |
| **2. 第三者レビュワーの招聘** | GitHub PR | 別のレビュワー | 〜1週間 |
| **3. Agora（Discussion）** | GitHub Discussions | コミュニティ全体 | 〜2週間 |
| **4. Core Maintainer裁定** | GitHub Issue（クローズド） | Core Maintainer | 〜1週間 |

#### 段階1: PRコメントでの議論

*   最初のステップ。出典を提示しながら議論する。
*   多くの紛争はここで解決する想定。

#### 段階2: 第三者レビュワーの招聘

*   当事者だけで解決しない場合、別のレビュワーに意見を求める。
*   PRコメントで `@` メンションして招聘。

#### 段階3: Agora（Discussion）への昇格

*   より広いコミュニティの意見を求める場合。
*   中立性や解釈に関する議論で有効。
*   **注意**: 投票ではなく、出典に基づく議論を継続する。

#### 段階4: Core Maintainer裁定

*   それでも解決しない場合、最終決定者が判断する。
*   理由を明記した上で裁定を下す。

##### 再審請求

裁定は原則として最終決定だが、**新しい出典に基づく場合に限り**再審請求を認める。

| 条件 | 説明 |
| :--- | :--- |
| **新しい出典の提示** | 裁定時に存在しなかった、または提示されなかった出典が必要 |
| **同じ論点の蒸し返し禁止** | 同じ出典・同じ論拠での異議は不可 |
| **冷却期間** | 裁定から最低2週間経過後に再審請求可能 |

> **回数制限なし**: 有効な新しい出典がある限り、再審請求は何度でも認める。出典至上主義において、正当な証拠を無視することは原則に反する。

**乱用防止策**:

| 対策 | 説明 |
| :--- | :--- |
| **有効性の厳格な判断** | 「新しい」かつ「裁定を覆すに足る」出典のみ受理 |
| **却下理由の明記** | 却下時は理由を明記し、同じ理由での再申請を防ぐ |
| **悪意のある乱用への対応** | 明らかに無関係な出典での繰り返し申請は、紛争解決プロセスの濫用としてアカウント対応を検討 |

**再審フロー**:

```
裁定（最終決定）
    ↓
2週間の冷却期間
    ↓
【新しい出典を発見】
    ↓
再審請求（GitHub Issueで申請）
  - 新しい出典を明示
  - なぜこの出典が裁定を覆すか説明
    ↓
Core Maintainerが再審の可否を判断
  - 新しい出典が有効 → 再審（再度裁定）
  - 新しい出典が無効/条件未達 → 却下（理由を明記）
```

> **理由**: 出典至上主義において、新しい証拠の発見は裁定を覆す正当な理由となりうる。一方、感情的な異議や蒸し返しは認めない。

#### 利益相反のある紛争

紛争当事者が全員レビュワー、またはCore Maintainerが当事者である場合の対応を定義する。

##### ケース1: 当事者がレビュワーだが、Core Maintainerは非当事者

| 対応 | 説明 |
| :--- | :--- |
| 段階2をスキップ | 第三者レビュワーがいないため |
| 段階3（Agora）を活用 | コミュニティに広く意見を求める |
| 段階4でCore Maintainerが裁定 | 利益相反がないため通常通り |

##### ケース2: Core Maintainerも当事者である（現状）

| 対応 | 説明 |
| :--- | :--- |
| **Agoraで透明性を確保** | 利益相反があることをDiscussionで明示 |
| **出典に基づく議論を徹底** | 人ではなく証拠で判断 |
| **コミュニティの意見を尊重** | 明確な出典に基づく反論があれば、Core Maintainerは自らの立場を再考 |
| **最終判断** | 出典に基づき、透明性を確保した上でCore Maintainerが判断 |

> **注意**: このケースでは、Core Maintainerは自らの利益相反を認識し、より慎重かつ透明な判断を行う必要がある。

##### ケース3: Core Maintainerも当事者である（将来）

コミュニティ拡大後は以下の対応を検討する。

| 対応 | 説明 |
| :--- | :--- |
| **他のCore Maintainerが裁定** | 当事者以外のCore Maintainerが判断 |
| **外部専門家の招聘** | 高度に専門的な紛争の場合 |

#### 解釈が分かれる分野の紛争

哲学的解釈など、学術的に複数の立場が存在する分野での紛争対応を定義する。

##### 原則

| 原則 | 説明 |
| :--- | :--- |
| **複数解釈の併記** | 対立する学術的立場を公平に紹介 |
| **事実記述** | 「Aは〜と主張した」「Bはこれを〜と批判した」 |
| **出典の明示** | 各立場について専門家による出典を明示 |
| **Isidoricaとしての立場を取らない** | 読者に判断を委ねる |

##### 判断基準

Core Maintainerは**「どちらの解釈が正しいか」を裁定しない**。代わりに以下を判断する。

| 基準 | 説明 |
| :--- | :--- |
| **公平性** | 複数の立場が公平に紹介されているか |
| **出典の質** | 各立場に学術的な出典があるか |
| **事実と意見の分離** | 「〜は主張した」と「〜である」が混同されていないか |
| **中立性** | 特定の立場を「正解」として提示していないか |

##### 紛争の解決方法

| ケース | 対応 |
| :--- | :--- |
| 一方の立場のみ記載 | 他方の立場を追記するよう修正を求める |
| 出典のない解釈 | 学術的出典の追加を求める |
| 「〜である」と断定 | 「〜と主張されている」に修正を求める |
| 両論が公平に併記 | 紛争ではなく、現状維持 |

### 11.4 現状の運用体制

| 項目 | 現状（Phase 5） | 将来（コミュニティ拡大後） |
| :--- | :--- | :--- |
| **Core Maintainer** | 管理者1名（事実上のBDFL） | 複数名によるコア・メンテナー・グループ |
| **レビュワー** | 管理者のみ | コミュニティから任命 |
| **紛争頻度** | 低い（コンテンツ・コントリビューターが少ない） | 増加を想定 |

> **BDFL**: Benevolent Dictator for Life（慈悲深い終身独裁者）。オープンソースプロジェクトにおいて、最終決定権を持つ創設者を指す慣用表現。

### 11.5 将来の検討事項

| 論点 | 状況 | 備考 |
| :--- | :--- | :--- |
| **Core Maintainerグループの設立基準** | 未決定 | コミュニティ規模に応じて検討 |

## 12. 定期レビュー対象文書 (Document Review Policy)

技術選定やガバナンスポリシー等、定期的な見直しが必要な文書を管理する。

### 12.1 レビュー対象文書一覧

| 文書 | レビュー周期 | 担当 | 備考 |
| :--- | :--- | :--- | :--- |
| **ARCHITECTURE_DECISION_RECORD.md** | 6ヶ月 | Core Maintainer | 技術選定の妥当性、新選択肢の検討 |
| **GOVERNANCE.md** | 12ヶ月 | Core Maintainer | ガバナンスルールの見直し |
| **CONTENT_POLICY.md** | 12ヶ月 | Core Maintainer | コンテンツポリシーの妥当性 |
| **IMPLEMENTATION_STRATEGY.md** | 12ヶ月 | Core Maintainer | 戦略の妥当性、環境変化への対応 |
| **NON_FUNCTIONAL_REQUIREMENTS.md** | 6ヶ月 | Core Maintainer | パフォーマンス基準、テスト戦略 |
| **RISK_ANALYSIS.md** | 6ヶ月 | Core Maintainer | リスク評価の見直し |
| **LEGAL_REQUIREMENTS.md** | 12ヶ月 | Core Maintainer | 法的要件の変更確認 |
| **CI_CD_PIPELINE.md** | 6ヶ月 | Core Maintainer | ツール・依存関係の更新 |

### 12.2 レビュー周期の基準

| 周期 | 対象 | 理由 |
| :--- | :--- | :--- |
| **6ヶ月** | 技術選定、CI/CD、非機能要件 | 技術の進歩が速い。依存ライブラリのEOL、新選択肢の登場 |
| **12ヶ月** | ガバナンス、戦略、法的要件 | 変更頻度が低い。安定性を優先 |

### 12.3 レビュー項目

各文書のレビュー時に確認すべき項目：

#### 技術選定（ARCHITECTURE_DECISION_RECORD.md）

- [ ] 採用技術のサポート状況（Deprecated、EOL、セキュリティ問題）
- [ ] 新しい選択肢の登場（競合技術、標準化の進展）
- [ ] 依存ライブラリのメジャーバージョンアップ
- [ ] 却下した選択肢の再評価（状況変化により妥当になった可能性）
- [ ] コスト変動（Cloudflare、Algolia等の料金体系変更）

#### ガバナンス・ポリシー

- [ ] 運用上の問題点（ルールが機能していない箇所）
- [ ] コミュニティからのフィードバック
- [ ] 法的環境の変化（著作権法改正、GDPR等）
- [ ] 他プロジェクトのベストプラクティス

#### 非機能要件・CI/CD

- [ ] パフォーマンス基準の達成状況
- [ ] テストカバレッジの推移
- [ ] CI/CDツールのバージョン・機能変更
- [ ] 新しいテスト手法・ツールの検討

### 12.4 レビュープロセス

```
1. CIが期限30日前にIssueを自動作成（review-reminderラベル）
   ↓
2. Core MaintainerがIssueをアサイン
   ↓
3. 担当者がレビューを実施し、変更点をPRで提案
   ↓
4. PRレビュー・マージ
   ↓
5. 文書内の「定期レビュー」日付を次回に更新
   ↓
6. Issueをクローズ
```

### 12.5 レビュー結果の記録

各レビュー完了時、以下の形式でIssueにコメントを残す：

```markdown
## レビュー結果

**レビュー日**: 2026-07-01
**担当者**: @username

### 確認事項
- [x] 採用技術のサポート状況
- [x] 新しい選択肢の検討
- [x] 依存ライブラリの状態

### 変更点
- ADR-003: Algolia有料プラン移行を決定（DocSearch審査落ち）
- ADR-021: CSPポリシー更新

### 次回レビュー
2027年1月
```
