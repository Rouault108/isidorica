## 1. 基本原則

Isidoricaは「人類の知識をすべての人が理解できる形で提供する」公共図書館です。

| 原則 | 説明 |
| :--- | :--- |
| **公正** | 文化的障壁を取り除く |
| **構造** | 明確な構造で読者を導く |
| **能動的学習** | 読者が能動的に概念を獲得できるよう設計 |

### 対象読者の基準

**義務教育を修了した読者**を想定する。ただし、専門用語の定義要否は絶対的な基準ではなく、**その記事の主題からの認知的距離**で判断する（詳細はセクション3「専門用語」を参照）。難しい概念は具体例や比喩を用いて説明する。

## 2. 文体（最低限のルール）

> **詳細**: [JAPANESE_WRITING_GUIDE.md](./JAPANESE_WRITING_GUIDE.md) を参照。textlintで自動チェックされます。

### 必須ルール

| ルール | 理由 | 例 |
| :--- | :--- | :--- |
| **「です・ます」調で統一** | 文体混在は読者の認知負荷を高める | ✅「〜です」 ❌「〜である」 |
| **一文60文字以内** | スクリーンリーダーでの聞き取りやすさ、認知負荷軽減 | 長文は分割する |
| **「あなた」「私」は使わない** | 客観性の担保、多様な読者への配慮 | ✅「開発者は〜」 ❌「あなたは〜」 |

### 避けるべき表現

| 表現 | 理由 | 代替 |
| :--- | :--- | :--- |
| 「簡単に言えば」 | 読者を見下す印象 | 直接説明 |
| 「常識的に」「当然」 | 前提知識を強制 | 根拠を明示 |
| 「〜すべき」 | 義務を課す印象 | 「〜が推奨されます」 |
| 「することができる」 | 冗長 | 「できる」 |

## 3. 専門用語・ルビ

### 専門用語の定義基準：「認知的距離」モデル

用語に定義が必要かどうかは、**その記事の主題からの認知的距離**で判断する。

| 距離 | 対象 | 対応 | 例（「ナッシュ均衡」の記事） |
| :--- | :--- | :--- | :--- |
| **距離0** | 記事の主題そのもの | `:::definition` で**必ず**定義 | ナッシュ均衡 |
| **距離1** | 主題を理解するための直接的前提 | `:::definition` で定義 | 戦略、利得、プレイヤー、ゲーム |
| **距離2** | 周辺知識（あれば理解が深まるが必須ではない） | `[[Term]]` でToggletip | 期待値、最適化、パレート効率 |
| **距離3+** | 一般教養レベル | 定義不要 | 数学、経済学、意思決定 |

#### 判断の指針

- **距離0-1**:「この概念を知らなければ記事の内容が理解できない」→ 定義必須
- **距離2**:「知っていれば理解が深まるが、知らなくても記事は読める」→ `[[Term]]`
- **距離3+**:「義務教育を修了した読者なら概ね知っている」→ 定義不要

#### 前提記事がある場合

Frontmatterで `requires` を設定している場合、**前提記事で定義済みの用語は再定義不要**。

```yaml
# Frontmatter例
requires:
  - game-theory-basics
```

この場合、`game-theory-basics` 記事で定義済みの「ゲーム理論」「プレイヤー」などは、本記事では `[[Term]]` でリンクするのみでよい。

#### 読者モードと用語説明の関係

| 読者モード | 用語の扱い |
| :--- | :--- |
| **Explorer** | `:::explorer` 内で平易な言葉（限定コード）を使い、専門用語を避けるか、その場で自然に説明する |
| **Learner** | `:::learner` 内で専門用語を正式に定義（`:::definition`）し、体系的に説明する |
| **Scholar** | `:::scholar` 内で専門用語を前提として使用し、学術的補足を提供する |

この設計により：
- **読書体験がスムーズ**になる
- **文章自体が読者に最適化**される

### `[[Term]]` の役割

`[[Term]]` は**全モード共通**の「用語へのオンデマンドリンク」として機能する。

| 用途 | 説明 |
| :--- | :--- |
| **用語リンク** | クリック/タップで `terms.yml` の定義を Toggletip 表示 |
| **読者主導** | 読者が「この用語について詳しく知りたい」と思ったときに使用 |
| **モード差異** | **なし**（全モードで同じ控えめなマーキング） |

### 専門用語の記法まとめ

| 状況 | 記法 | 用途 |
| :--- | :--- | :--- |
| **距離0-1（定義必須）** | `:::definition` | Learner向け文章内で詳細に定義 |
| **距離2（周辺知識）** | `[[Term]]` | 読者がオンデマンドで参照 |
| **略語** | 初出時に正式名称を併記 | SSR（Server-Side Rendering） |
| **Explorer向け** | 平易な言葉で自然に説明 | `:::explorer` 内で使用 |

### ルビ

**基本は自動付与**。何も書かなくてOK。

自動ルビが不正確な場合のみ手動指定：

```markdown
{東海林|しょうじ}さんは{今日|きょう}も元気です。
```

## 4. 引用・出典

### 基本ルール

**すべての主張には出典が必要**（自明な事実・論理的帰結を除く）

### 記法

```markdown
ゲーム理論の基礎は1944年に確立された [@neumann-1944, p. 45]。
複数の出典 [@nash-1950; @selten-1965]。
```

### 出典ID命名規則

| パターン | 例 |
| :--- | :--- |
| 単著 | `@neumann-1944` |
| 共著（2人） | `@neumann-morgenstern-1944` |
| 共著（3人以上） | `@smith-et-al-2020` |

### 参考文献リスト

記事末尾への参考文献リストの記載は**不要**です。引用記法 `[@id]` を使用すれば、システムが `bibliography.json` から自動的にリンクを生成します。

## 5. 画像・図表

### 画像の配置

画像は記事ファイル（`index.md`）と同じ階層の **`images/`** ディレクトリに配置する。

```
articles/
└── computer-science/
    └── algorithm/
        ├── index.md
        └── images/
            └── flowchart.svg
```

### 画像の記法

すべての画像に **altテキスト必須**（アクセシビリティ）。

```markdown
::: figure {id="fig:diagram" src="./images/diagram.svg" alt="概念図の説明"}
キャプション

出典: [@tanaka-2020, p. 45]
:::
```

### 相互参照

```markdown
:ref:[fig:diagram] に示すように...  → 「図 1」と表示
:ref:[table:payoff] を参照。        → 「表 1」と表示
:ref:[eq:gauss] より...              → 「式 (1)」と表示
```

## 6. 記事構造（6Phaseモデル）

Learner向け記事は以下の構造に従う。

| Phase | 目的 | 推奨記法 |
| :--- | :--- | :--- |
| **1** | 予測誤差の喚起 | `:::intuition` |
| **2** | 概念定義 | `:::definition` |
| **3** | 詳細解説 | `:::laboratory`, `:::steps` |
| **4** | 確認問題 | `:::quiz` |
| **5** | 要約 | 通常のMarkdown |
| **6** | 発展 | `[[Term]]`, リンク |

> **参照**: [LEARNING_FOUNDATIONS.md](../shared/LEARNING_FOUNDATIONS.md)

## 7. 読者適応Directive

読者モードに応じてコンテンツを切り替える。詳細は [READER_ADAPTATION_DESIGN.md](../engine/READER_ADAPTATION_DESIGN.md) を参照。

### 各モードの概要

| Directive | 役割 | 前提知識 | 用語の扱い |
| :--- | :--- | :--- | :--- |
| `:::explorer` | 興味喚起・自己完結した概要 | **不要** | 平易な言葉（限定コード）で説明。専門用語を避ける |
| `:::learner` | 体系的学習 | **前提記事を読んでいる** | 前提記事で定義済みなら再定義不要。`[[Term]]`でリンク |
| `:::scholar` | 学術的補足 | **専門知識あり** | 専門用語を前提として使用。定義なし |

### Explorerモードの執筆指針

**Explorerは「やさしい辞書」ではない。**

| 指針 | 説明 |
| :--- | :--- |
| **定義しない** | 「〇〇とは〜です」で始めない |
| **問いかける** | 日常的な疑問から始める（「なぜコンビニは同じ場所に集まるのか？」） |
| **日常と接続** | 抽象概念を具体例に結びつける |
| **誘導先は学習パスの最初** | 「この記事のLearnerモード」ではなく「学習パスの最初の記事」へ誘導 |

> **重要**: Explorerからの誘導先は「この記事のLearnerモード」ではありません。Learnerは順序依存（前提記事を読んでいることを前提）のため、途中の記事のLearnerに誘導しても理解できません。

### Learnerモードの執筆指針

**Learnerは「学習パスを順番に辿ること」を前提とする。**

| 指針 | 説明 |
| :--- | :--- |
| **前提記事の用語は再定義不要** | `requires` で指定した記事で定義済みの用語は `[[Term]]` でリンクのみ |
| **距離0-1の用語は `:::definition` で定義** | 主題と直接的前提は必ず定義 |
| **6Phase構造に従う** | Phase 1-6の完全な学習体験を提供 |

### Scholarモードの執筆指針

| 指針 | 説明 |
| :--- | :--- |
| **専門用語を前提** | 定義は省略 |
| **原典・議論の余地に言及** | 学術的な補足情報を提供 |
| **Agoraへの導線** | 「この概念についてAgoraで議論する」リンクを設置 |

### 執筆例

```markdown
::: explorer
なぜコンビニは競合店のすぐ近くに出店するのでしょうか？
ライバル同士なら離れた方が良さそうですが、実は「誰も動きたくなくなる」不思議な理由があります。

→ [体系的に学ぶには「ゲーム理論入門」パスへ]
:::

::: learner
::: definition {title="ナッシュ均衡"}
各プレイヤーが互いに最適な戦略をとっており、誰も単独で戦略を変更する動機がない状態。
:::

この概念は、非協力ゲームにおける解の概念として重要です...
:::

::: scholar
> **学術的補足**: Nash (1950) の原論文では、不動点定理を用いた存在証明が示されている。
> 計算複雑性の観点からはPPAD完全であり、効率的なアルゴリズムの存在は未解決問題である。
:::
```

## 8. よく使うMarkdown記法

### コールアウト

```markdown
:::note {title="補足"}
補足情報
:::

:::warning
注意点
:::

:::definition {title="ナッシュ均衡"}
各プレイヤーが互いに最適な戦略をとっている状態。
:::
```

### Quiz

```markdown
::: quiz {id="quiz-1"}
### Q1: ナッシュ均衡とは？

- [ ] すべてのプレイヤーが最大利得を得ている
- [x] 誰も一方的に戦略を変えたくない状態

::: feedback {correct}
正解！
:::
:::
```

### インライン・グロッサリー

```markdown
[[ナッシュ均衡]]は重要な概念です。
```

### コードブロック（行ハイライト）

````markdown
```python {3,5-7}
def example():
    x = 1
    y = 2  # ハイライト
```
````

## 9. サポートしない記法

以下はIsidoricaの理念に基づきサポートしない。

| 記法 | 理由 | 代替 |
| :--- | :--- | :--- |
| 動画/音声 | アクセシビリティ | Laboratory, ダイアグラム |
| iframe | セキュリティ | Laboratory |
| 生のHTML | 構造の一貫性 | Directive記法 |
| インラインCSS | アクセシビリティ | デザイントークン |

## 10. レビュープロセス

### PRを作成する前のセルフチェック

- [ ] 文体（です・ます調）が統一されているか
- [ ] すべての主張に出典があるか
- [ ] すべての画像にaltがあるか
- [ ] 6Phase構造に沿っているか
- [ ] `pnpm textlint articles/` でエラーがないか

### 判断に迷ったら

| 状況 | 対応 |
| :--- | :--- |
| textlintで解決できないエラー | GitHub Issueで相談、または `#writing` チャンネルで質問 |
| 文体・表現の判断に迷う | [JAPANESE_WRITING_GUIDE.md](./JAPANESE_WRITING_GUIDE.md) を確認、それでも不明なら相談 |
| 専門用語の定義レベルに迷う | 「認知的距離」モデル（セクション3）を参照。距離0-1なら定義必須、距離2なら `[[Term]]` |
| 出典の信頼性に迷う | [GOVERNANCE.md](./GOVERNANCE.md) の出典ポリシーを確認 |

> **詳細**: [GOVERNANCE.md](./GOVERNANCE.md)
