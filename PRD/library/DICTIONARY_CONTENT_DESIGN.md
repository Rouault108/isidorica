## 1. ビジョン：理論統合による辞書像

### 1.1 ビジョン・ステートメント

> **Isidorica辞書は、語の意味を「列挙」するのではなく「生成」の仕組みとして捉え、読者が概念を構造的・関係的・体験的に理解できる、動的な知識アクセスポイントを提供する。**

---

### 1.2 既存の辞書の限界

Isidorica辞書が必要な理由を、既存の辞書の限界から明確にする。

| 辞書タイプ | 達成していること | 達成できていないこと |
| :--- | :--- | :--- |
| **伝統的辞書**（大辞林、OED） | 意味の網羅的記述 | **なぜその意味になるか**の説明。「良いナイフ」が「切れ味が良い」を意味する理由を説明できない |
| **学習者辞典**（LDOCE、COBUILD） | 平易な定義、例文 | **概念の体系的理解**。語彙レベルで完結し、知識ネットワークへ接続しない |
| **Wikipedia** | 概念の詳細な解説 | **語の意味構造・用法**。「ナッシュ均衡とは何か」は詳しいが「語の使い方」はない |
| **Wiktionary** | 語源、発音、語形変化 | **意味の生成的構造**。静的な意味列挙にとどまる |
| **専門用語辞典** | 厳密な定義 | **初学者への橋渡し**。限定コード話者に閉ざされている |

---

### 1.3 理論統合マップ

Isidoricaは、[CONTENT_FOUNDATIONS.md](../shared/CONTENT_FOUNDATIONS.md)で整理した理論を統合して辞書を形成する。

| 理論層 | 理論 | 辞書への貢献 | 従来の辞書が欠いていたもの |
| :--- | :--- | :--- | :--- |
| **Tier 1** | **GL（生成語彙論）** | 意味の内部構造を提供。「何であるか」「何のためか」「どう生じたか」「何から構成されるか」 | 意味列挙から**意味生成**へ |
| **Tier 1** | **TCL（型強制論）** | 文脈依存の意味変化メカニズムを記述 | 多義語の**なぜ**を説明 |
| **Tier 2** | **フレーム意味論** | 語が喚起する知識フレームを明示 | 語を孤立させず**場面**と共に提示 |
| **Tier 2** | **認知意味論** | 身体経験・メタファーに基づく理解 | 抽象概念を**具体的イメージ**で接地 |
| **Tier 2** | **分布意味論** | 意味的近傍を提示 | 類義語だけでなく**分布的類似**を網羅 |
| **Tier 2** | **語用論** | 使い方を記述（使用域、コロケーション） | 「正しい定義」だけでなく**自然な使い方** |
| **Tier 2** | **意味変化論** | 語の意味がどう変遷したかを追跡 | 現在の意味だけでなく**歴史的理解** |
| **メタ理論** | **言語コード論** | 限定コード（日常）と精密コード（学術）の両方を提供 | 専門家だけでなく**初学者にも開かれる** |
| **LEARNING_FOUNDATIONS** | **Laboratory** | 概念の体験的グラウンディング | 定義を読むだけでなく**操作して理解** |

---

## 2. 設計哲学：メタ理論からの導出

### 2.1 辞書の3つの機能（辞書機能論）

**Bergenholtz & Tarp (2003)** の辞書機能論に基づき、Isidoricaの辞書が満たすべき機能を定義する。

| 機能 | 定義 | ユーザーの状況 |
| :--- | :--- | :--- |
| **認知機能（Cognitive）** | 新しい知識を獲得する | 「概念を理解したい」 |
| **伝達機能（Communicative）** | テキストの理解・産出を支援する | 「読解中」「執筆中」 |
| **操作機能（Operative）** | 特定のタスクを遂行する | 「翻訳」「コマンド実行」「手順確認」 |

---

#### 認知機能の詳細（Isidoricaの主戦場）

ユーザーが**新しい知識を獲得**したい場合に対応する。Isidoricaはこの機能を最も重視する。

| サブ機能 | ユーザーの状況 | Isidoricaでの対応 |
| :--- | :--- | :--- |
| **一般的知識獲得** | 世界について知りたい | 記事（Phase 1-6）、概念ネットワーク |
| **言語知識獲得** | 言語体系を知りたい | `form.morphology`、`syntax`、`pragmatics` |
| **専門知識獲得** | 分野固有の知識を知りたい | `qualia`（クオリア構造）、`ecology.frame` |

**Isidoricaの特徴**:

- **Phase 1-6**: 予測誤差から始まる体系的学習
- **Laboratory**: 概念の体験的グラウンディング
- **概念ネットワーク**: 関連概念のブラウジング

---

#### 伝達機能の詳細

ユーザーが**テキストの理解・産出**を行う場合に対応する。従来の辞書はこの機能を最も重視してきた。

| サブ機能 | ユーザーの状況 | Isidoricaでの対応 |
| :--- | :--- | :--- |
| **テキスト受容（読解）** | 読んでいる語の意味を調べたい | `display.definition`、Toggletip |
| **テキスト産出（執筆）** | 書くときの用法を確認したい | `examples[]`、`pragmatics.collocation`、`pragmatics.register` |
| **翻訳** | 翻訳等価語を調べたい | `translations[]`（等価タイプ付き） |

**Isidoricaの特徴**:

- **Toggletip**: 記事内での即時参照（文脈を離れずに確認）
- **コロケーション**: 共起表現の提示
- **使用域（Register）**: フォーマル/インフォーマルの区別

---

#### 操作機能の詳細

ユーザーが**特定のタスクを遂行**する場合に対応する。従来は辞書の範囲外とされることが多かったが、Isidoricaは支援可能である。

| サブ機能 | ユーザーの状況 | Isidoricaでの対応 |
| :--- | :--- | :--- |
| **技術的操作** | コマンド・APIの構文を確認したい | `syntax.construction`、`examples[]`、Laboratory |
| **規範的判断** | 規格・標準を確認したい | `qualia.formal.constraints`、`meta.sources` |
| **翻訳作業** | 翻訳等価語と文化的差異を調べたい | `translations[]`、`ecology.cultural_context` |

**Isidoricaの特徴**:

- **Laboratory**: コードを実行して結果を確認
- **制約情報**: 有効な値域、前提条件の明示
- **出典明記**: 「誰がこう定めたか」の追跡可能性

---

**設計原則 P1**: Isidoricaは**認知機能を最優先**しつつ、伝達機能・操作機能も同時に満たす。

> これは従来の辞書（伝達機能重視）やWikipedia（認知機能のみ）との差別化点である。

---

### 2.2 辞書と記事の関係

LEARNING_FOUNDATIONS.mdの6段階螺旋（Phase 1-6）との整合性を保つため、辞書と記事を**別リソース**として分離する。

| リソース | URL | 主な機能 | 構造 |
| :--- | :--- | :--- | :--- |
| **記事ページ** | `/concepts/{slug}` | 認知機能（体系的学習） | Phase 1（予測誤差）から始まる |
| **辞書ページ** | `/dictionary/{term}` | 伝達機能 + 認知機能への橋渡し | 表示層（Display）から始まる |

---

#### 記事と辞書の対応関係

記事と辞書は**1:1対応ではない**。以下のケースが存在する。

| ケース | 例 | 対応関係 |
| :--- | :--- | :--- |
| **1:1** | 「ナッシュ均衡」 | 記事1つ : 辞書エントリー1つ |
| **1:多** | 「ゲーム理論」 | 記事1つ : 辞書エントリー複数（関連用語） |
| **多:1** | 「均衡」（多義語） | 辞書エントリー1つ : 記事複数（各均衡概念） |
| **辞書のみ** | 「プレイヤー」（一般語） | 辞書エントリーあり : 記事なし |
| **記事のみ** | 学習パス導入記事 | 記事あり : 辞書エントリーなし |

---

#### ID/slugの管理

記事slugと辞書term-idは**同じ名前空間を共有**する。同一IDであれば自動的に関連付けられる。

| 原則 | 説明 |
| :--- | :--- |
| **共通ID** | 記事slug = 辞書term-id（同じ概念を指す） |
| **自動リンク** | 同一IDの記事が存在すれば「この概念を学ぶ →」を表示 |
| **1:1でない場合** | 記事のみ、辞書のみ、両方存在のケースがある |

> **詳細**: 記事と辞書は同一IDを持つファイルをビルド時に自動検出して連携する。

---

**設計原則 P2**: 辞書ページは**認知機能の入口**であり、完全な認知機能は記事ページが担う。

```
辞書ページ（即時参照）
    ↓ 「この概念を学ぶ →」（同名の記事が存在する場合）
記事ページ（Phase 1-6）
    ↓
Laboratory（体験的グラウンディング）
```

### 2.3 百科事典的意味論の採用

**Langacker (1987)** の百科事典的意味論に基づき、Isidoricaは以下の立場をとる。

#### 知識ネットワークとしての設計

| 伝統的問い | Isidoricaの回答 |
| :--- | :--- |
| 辞書を作るのか、百科事典を作るのか？ | **知識ネットワーク**を構築する |
| 語の意味をどこまで記述するか？ | 語は**アクセスポイント**。深さは目的による |

**設計原則 P3**: 辞書エントリーは知識ネットワークへの**アクセスポイント**である。

---

#### 辞書と記事の目的の違い

辞書と記事は「深さの違い」ではなく、**目的の違い**。

| 観点 | 辞書 | 記事 |
| :--- | :--- | :--- |
| **目的** | 即時参照（伝達機能） | 体系的学習（認知機能） |
| **構造** | 構造化データ | 叙述的テキスト（Phase 1-6） |
| **入口** | 検索、Toggletip | 学習パス、概念ネットワーク |

**情報の重複は許容**される。異なる目的のために同じ情報を異なる形式で提供してよい。

| 例 | 辞書での提供 | 記事での提供 |
| :--- | :--- | :--- |
| 定義 | `display.definition`（構造化） | Phase 2（叙述的） |
| 例文 | `examples[]`（代表例） | 本文中（文脈付き） |
| 歴史 | `history`（要約） | 専用セクション（詳細） |

### 2.4 辞書の対象範囲

Isidoricaのミッション「人類の知識を、すべての人が理解できる形で提供する」（PHILOSOPHY_DISCUSSION.md）に基づき、辞書の対象範囲を定義する。

| 理念 | 辞書への含意 |
| :--- | :--- |
| **「人類の知識」** | 専門用語だけでなく、言語自体も知識の一部 |
| **「公共図書館」** | 国語辞典、人名辞典、地理辞典も公共図書館の基本 |
| **「すべての人」** | 限定コード話者への配慮。一般語も対象とすべき |

**設計原則 P4**: 辞書は**専門用語・固有名詞・一般語**のすべてを対象とする。

## 4. エントリータイプ定義

### 4.1 エントリータイプ一覧

Wikipedia/百科事典のインフォボックス分析に基づき、以下のタイプを定義する。

##### Tier 1: Phase 4-6で実装（記事連動）

| タイプ | 対象 | 例 | 理由 |
| :--- | :--- | :--- | :--- |
| **term** | 専門用語・学術概念 | ナッシュ均衡、エントロピー、認知負荷 | 記事との1:1対応。中核タイプ |
| **person** | 人物 | アインシュタイン、ソクラテス | 人物記事連動、引用・出典で必須 |
| **place** | 地名・地理的実体 | パリ、富士山、アマゾン川 | 地理関連記事 |
| **work** | 作品（著作、芸術、ソフトウェア） | 源氏物語、モナ・リザ、Linux | 文献・作品への言及 |
| **organization** | 組織・団体 | UNESCO, 東京大学, Google | 学術機関、企業への言及 |
| **event** | 歴史的出来事 | 第二次世界大戦、フランス革命 | 歴史記事で必須 |

##### Tier 2: Phase 5-6で実装（専門分野拡張）

| タイプ | 対象 | 例 | 理由 |
| :--- | :--- | :--- | :--- |
| **species** | 生物種 | イヌ、サクラ、大腸菌 | 生物学記事。分類学的階層、学名など固有項目 |
| **journal** | 学術雑誌 | Nature, Science, Cell | 引用管理。出版社、インパクトファクターなど |

##### Tier 3: Phase 7以降で実装（網羅性拡張）

| タイプ | 対象 | 例 | 理由 |
| :--- | :--- | :--- | :--- |
| **word** | 一般語・基本語彙 | 犬、走る、美しい | 言語的障壁の解消。データ管理が複雑 |
| **chemical** | 化学物質 | H₂O, NaCl, エタノール | 化学記事。分子式、モル質量など固有項目 |
| **language** | 言語 | 日本語、ラテン語、手話 | 言語学記事。話者数、言語系統など固有項目 |
| **celestial_body** | 天体 | 太陽、木星、アンドロメダ銀河 | 天文学記事。軌道、質量など（placeとは全く異なる項目） |
| **law** | 法律・条約 | 民法、ウィーン条約 | 法学・政治学記事 |

##### 統合判断の記録

| 候補タイプ | 判断 | 理由 |
| :--- | :--- | :--- |
| **software** | workに統合 | 著作物の一種。プログラミング言語、ライセンスは追加フィールドで対応 |
| **award（賞）** | 独立タイプ不要 | termまたは記事で扱う |
| **unit（単位）** | termに統合 | 専門用語として扱える |
| **food（食品）** | Phase 7以降で検討 | 優先度は低い |

#### 段階的拡張戦略

| 段階 | 実装タイプ | 理由 |
| :--- | :--- | :--- |
| **Phase 4-6** | term, person, place, work, organization, event | 記事との連動があるタイプを優先 |
| **Phase 5-6** | species, journal | 専門分野（生物学、学術引用）の拡張 |
| **Phase 7以降** | word, chemical, language, celestial_body, law | 網羅性の拡張。既存リソースとの連携を検討 |

**スキーマ設計**: 全エントリータイプを定義しておき、実装は必要なタイプから行う。

#### 不採用理由の記録

| 選択肢 | 説明 | 不採用理由 |
| :--- | :--- | :--- |
| **専門用語のみ** | 記事と1:1対応 | 理念「すべての人」に反する。言語的障壁が残る |
| **固有名詞を含めない** | 人名・地名・作品を除外 | 記事で人物・作品に言及する必要がある |
| **天体をplaceに統合** | 地理的実体の拡張として扱う | インフォボックス項目が全く異なる（軌道、質量 vs 緯度/経度、人口） |

#### 実装複雑度の評価

| 観点 | 複雑度 | 備考 |
| :--- | :--- | :--- |
| **スキーマ定義** | 中 | Zodの union type で対応可能 |
| **UI実装** | 中〜高 | タイプ別の表示コンポーネントが必要 |
| **一般語のデータ管理** | 高 | 優先順位、既存リソースとの連携が課題。Phase 7以降で対応 |

## 5. コンテンツ項目設計

### 5.1 項目選択の原則

**設計原則 P5**: ユーザーの状況（認知/伝達/操作）に応じた項目を提供する。

| 機能 | ユーザーのニーズ | 必要な項目 |
| :--- | :--- | :--- |
| **認知（学習）** | 概念を理解したい | 定義、直感、例、図解、Laboratory |
| **伝達（受容）** | 読解中に意味を調べたい | 定義、文脈に合う意味、コロケーション |
| **伝達（産出）** | 書く・話すために語を探す | 例文、コロケーション、類義語、使用域 |
| **操作（翻訳）** | 翻訳等価語を知りたい | 翻訳、偽の友、文化的差異 |

**設計原則 P6**: すべての項目を埋める必要はない。**概念によって適切な項目が異なる**。

---

### 5.2 体験的グラウンディング

**設計原則 P7**: Laboratoryにより言語定義の**循環を断ち切る**外部接地を提供する。

CONTENT_FOUNDATIONS.md（定義の循環性問題）より：
> 「赤」= 血の色 → 「血」= 赤い体液 → （循環）

**解決策**: `laboratory[]`や`ecology.embodiment`を通じて、言語定義の外部にある**経験・現実に接地**する。

### 5.3 デフォルト表示に含める項目（タイプ別）

辞書学・辞書ユーザー研究・主要辞書の知見に基づき、各タイプのデフォルト表示項目を確定。

#### 共通原則

| 原則 | 根拠 |
| :--- | :--- |
| **全タイプ共通**: 直感 + 定義 | 辞書機能論：受容機能の基本 |
| **Laboratoryは存在表示** | 体験への導線を確保（自動実行はしない） |
| **発音**: 外来語・人名由来語のみ | 辞書ユーザー研究：IPA/発音記号は読み飛ばされる |
| **品詞**: 専門用語では非表示 | 専門用語はほぼ名詞。動詞用法がある場合のみ表示 |
| **画像**: タイプ依存 | 具体物・人物・場所・生物には有効 |

#### タイプ別デフォルト表示項目

| タイプ | デフォルト表示項目 |
| :--- | :--- |
| **term** | 直感, 定義, 例, discipline, （発音: 外来語のみ）, Laboratory存在表示 |
| **person** | 直感, 定義, 生没年, 職業, known_for, 画像, （発音: 外来人名のみ）, Laboratory存在表示 |
| **place** | 直感, 定義, place_type, country, 画像（地図 or 写真）, Laboratory存在表示 |
| **work** | 直感, 定義, work_type, creator, 発表年, 画像（表紙等）, Laboratory存在表示 |
| **organization** | 直感, 定義, org_type, 設立年, Laboratory存在表示 |
| **event** | 直感, 定義, event_type, 開始日-終了日, location, （画像: 象徴的なものがある場合）, Laboratory存在表示 |
| **species** | 直感, 定義, 学名, 分類, 画像, Laboratory存在表示 |
| **journal** | 直感, 定義, 出版社, discipline, Laboratory存在表示 |

#### 横断的項目の判断基準

| 項目 | 基準 | 表示形式 |
| :--- | :--- | :--- |
| **発音** | 外来語・人名由来語で読み方が自明でない場合のみ | カタカナ優先、IPAは詳細開示 |
| **品詞** | 動詞・形容詞用法がある場合のみ | 日本語（名詞、動詞、形容詞） |
| **discipline** | termとjournalのみ | 選択式（economics, philosophy等） |
| **画像** | 視覚認識が有効なタイプのみ（person, place, work, species） | パブリックドメイン or CC限定 |
| **Laboratory** | 存在する場合は常に導線を表示 | サムネイル or リンク（自動実行なし） |
| **register（使用域）** | 学術的/日常的の混同リスクがある場合のみ | ラベル（例：「学術的」「法律用語」） |
| **related（関連語）** | 関連概念が存在する場合 | 「関連 →」リンク（件数表示） |
| **sources（出典）** | 常に存在するが... | ❌ デフォルト非表示、「出典を見る」で展開 |

## 6. 学習者適応：LEARNING_FOUNDATIONSからの導出

### 6.1 専門性逆転効果への対応

**Kalyuga (2003, 2007)** の専門性逆転効果に基づき、学習者の熟達度に応じて表示情報を変える。

| 教育手法 | 初心者への効果 | 専門家への効果 |
| :--- | :--- | :--- |
| 詳細な説明 | ✅ 有効 | ❌ 冗長負荷 |
| ステップバイステップ誘導 | ✅ 認知負荷軽減 | ❌ 制約が邪魔 |
| 問題解決のみ | ❌ 過負荷 | ✅ 効率的 |

**設計原則 P8**: 辞書では**段階的開示**により、読者が自分の熟達度に応じて情報量を調整できるようにする。

---

#### 辞書機能論からの設計判断

辞書機能論（Tarp 2008, Fuertes-Olivera & Tarp 2014）では、辞書利用状況を**受容（Reception）**と**産出（Production）**に分類する。この分類に基づき、UI設計の選択肢を検討した。

##### 設計選択肢の比較

| 選択肢 | 説明 | 採用 | 理由 |
| :--- | :--- | :--- | :--- |
| **A: 明示的モード切り替え** | 「読む/書く/学ぶ」ボタンで表示を切り替え | ❌ 不採用 | 認知コストが高い。Lew (2011) のユーザー研究では、明示的なモード選択はユーザーの負担になることが示されている |
| **B: 完全統合** | すべての情報を同時に表示 | ❌ 不採用 | 情報過多。満足化（Simon 1956）により、ユーザーは最初の情報で満足し、産出情報を見ない傾向がある |
| **C: 段階的開示** | 基本情報→詳細→完全の3段階 | ✅ 採用 | 認知負荷を管理しつつ、必要な情報にアクセス可能 |

##### 受容と産出の統合

言語習得研究（Melka 1997, Laufer & Goldstein 2004）によれば、語彙知識は**受容-産出連続体（Receptive-Productive Continuum）**上に存在する。受容と産出は別の能力だが、連続したプロセスとして位置づけられる。

辞書ユーザー研究（Dziemianko 2010）では、**例文**が受容・産出の両方で最も参照されることが示されている。したがって、例文はデフォルト表示に含めることで、受容から産出への移行を支援する。

---

#### 辞書エントリーの段階的開示

| 表示レベル | 対象 | 表示内容 | 支援する機能 |
| :--- | :--- | :--- | :--- |
| **デフォルト（折りたたみなし）** | 全員 | 直感、定義、例文（1〜2文） | 受容 + 産出の核心 |
| **詳細（1段階目）** | 学習者・実践者 | コロケーション、使用域、追加例文 | 産出（中級） |
| **完全（2段階目）** | 上級者・研究者 | クオリア構造、型強制、語源、構文 | 産出（高度）+ 学習 |

```
デフォルト表示: 直感 + 定義 + 例文
  [詳細を見る]
    ↓
詳細表示: コロケーション + 使用域 + 追加例文
  [意味構造を見る]
    ↓
完全表示: クオリア構造 + 型強制 + 語源 + 構文
```

---

### 6.2 認知負荷の管理

**Sweller (1988)** の認知負荷理論に基づき、外在的負荷を最小化する。

| 負荷タイプ | 辞書設計への含意 |
| :--- | :--- |
| **内在的負荷** | 概念の複雑さは変えられない → 分節化（Segmenting）で管理 |
| **外在的負荷** | 読み飛ばし行動を予測 → 情報の階層化、段階的開示 |
| **学習関連負荷** | 能動的関与を誘発 → Laboratory、例文、Quiz |

**設計原則 P9**: **段階的開示**により認知負荷を管理する（セクション4.1で詳述）。

---

### 6.3 言語コードの架橋

**Bernstein (1971)** の言語コード論に基づき、限定コードと精密コードを架橋する。

| コード | 説明 | 辞書での対応 |
| :--- | :--- | :--- |
| **限定コード** | 文脈依存的、日常的 | `display.intuition` |
| **精密コード** | 文脈独立的、学術的 | `display.definition`, `qualia` |

**設計原則 P10**: **両方のコードを提供**し、読者が選択できるようにする。

---

### 6.4 満足化への対応

**Simon (1956)** の満足化理論と辞書ユーザー研究（Tarp, Lew）に基づく。

> ユーザーは「合理的」に認知コストを節約している。

**設計原則 P11**: **最初に見える情報が「正解」**であるべき。

| ユーザー行動 | 設計対応 |
| :--- | :--- |
| 最初に見つかった情報で満足 | デフォルト表示を最適化 |
| 長い定義は読み飛ばす | 短い定義 + 詳細は折りたたみ |
| 例文に直行する | 例文を目立つ位置に配置 |

## 7. 設計原則まとめ

| 原則 | 説明 |
| :--- | :--- |
| **P1: 認知機能の優先** | 概念理解を最優先。伝達機能は副次的 |
| **P2: 記事への橋渡し** | 辞書は即時参照 + 深い学習への入口 |

| **P5: 機能駆動** | ユーザーの状況（認知/伝達/操作）に応じた項目 |
| **P6: 概念依存** | すべての項目を埋める必要はない |
| **P7: 体験的接地** | Laboratoryにより言語定義の循環を断ち切る |

| **P9: 認知負荷の管理** | 段階的開示、分節化、外在的負荷の最小化 |
| **P10: コードの架橋** | 限定コード（intuition）と精密コード（definition）の両立 |
| **P11: 満足化への対応** | 最初に見える情報が「正解」であるべき |
