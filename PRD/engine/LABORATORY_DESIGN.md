# Laboratory設計書 (Laboratory Design Document)

本ドキュメントでは、Isidoricaにおけるインタラクティブコンテンツ「Laboratory」の設計を定義する。

**ステータス**: ドラフト（検討中）

---

## 1. 概要

### 1.1 Laboratoryとは

Laboratoryは、読者が**能動的に操作・体験**することで概念を理解するためのインタラクティブコンテンツである。

### 1.2 理論的位置づけ（LEARNING_FOUNDATIONS.mdとの連携）

LaboratoryはLEARNING_FOUNDATIONS.mdで定義された**二層構造モデル**のTIER 2（認知プロセス層）を具体化する主要な装置である。

| 10層モデルの層 | Laboratoryの役割 | 対応する理論（Section 5） |
| :--- | :--- | :--- |
| **1. 身体的・行為的層** | 操作を通じた意味の産出 | 5.2 エナクティビズム |
| **2. 情動・動機的層** | 驚きと好奇心の喚起 | 5.3 SDT, 5.4 CVT |
| **3. 予測的・生成的層** | 予測誤差の意図的生成 | 5.1 予測処理 |
| **5. 因果的・機構的層** | パラメータ操作による因果理解 | - |
| **6. メタ認知的層** | JOLの較正、流暢性の錯覚への対抗 | 5.7 メタ認知 |
| **10. 構築的・転移的層** | 遠隔転移の支援 | 5.10 遠隔転移 |

### 1.3 認知的目標

| 目標 | 説明 | Laboratoryの役割 | 理論的根拠 |
| :--- | :--- | :--- | :--- |
| **予測誤差の喚起** | 読者の素朴なメンタルモデルを揺さぶる | 「こう思っていたけど違った」という驚きを生成 | Friston (予測処理) |
| **身体化認知** | 操作を通じて意味を産出 | 見るだけでなく「触れる」「動かす」 | Varela (エナクティビズム) |
| **パラメータ操作** | 因果関係の理解 | 「これを変えるとあれが変わる」を体感 | - |
| **流暢性の錯覚への対抗** | 「読んだだけで分かったつもり」を防ぐ | 手を動かして考えさせる | Nelson (メタ認知) |

### 1.4 Laboratoryの3つの性質（LPP理論より）

LEARNING_FOUNDATIONS.md Section 5.9（状況的学習とLPP）に基づき、Laboratoryは以下の3つの性質を持つ。

#### 1.4.1 認知的徒弟制（Cognitive Apprenticeship）

[Collins et al. 1989]

Laboratoryは、専門家の**思考プロセスを可視化**する装置として機能する。

| 要素 | 説明 | Laboratoryでの実装 |
| :--- | :--- | :--- |
| **モデリング（Modeling）** | 専門家の思考を見せる | パラメータの変化と結果の関係を可視化 |
| **足場かけ（Scaffolding）** | 補助を提供する | ヒント、ガイド付きモード、初期設定 |
| **フェーディング（Fading）** | 補助を徐々に外す | 自由探索モードへの移行 |
| **コーチング（Coaching）** | フィードバックを提供 | 操作結果の即時フィードバック |

#### 1.4.2 物象化（Reification）

[Wenger 1998]

Laboratoryは、専門家コミュニティの知恵が**凝縮されたアーティファクト（道具）**である。シミュレータ、数式の可視化、アルゴリズムの動作表示などは、単なる教材ではなく、専門家が実際に使う「道具の使い方」を学ぶ機会を提供する。

| 道具 | 凝縮された知恵 | 例 |
| :--- | :--- | :--- |
| **シミュレータ** | モデル化の作法、パラメータの意味 | 感染症シミュレータ（SIRモデル） |
| **可視化** | 抽象概念の視覚的表現 | フーリエ変換のインタラクティブ可視化 |
| **コード実行環境** | プログラミングの作法、デバッグの考え方 | REPL、Jupyter風インターフェース |

#### 1.4.3 正統なミニチュア（Legitimate Miniatures）

Laboratoryでの課題は、**子供騙しのゲームではなく、専門家が扱う問題を単純化（低リスク化）した「正統なミニチュア」**であるべき。これにより、読者は「ごっこ遊び」ではなく「真正な実践」を行っている感覚（LPP）を得る。

| ✅ 正統なミニチュア | ❌ 子供騙し |
| :--- | :--- |
| 限られた変数での需要供給シミュレーション | 「経済ゲーム」と称した単純なクイズ |
| 実際の遺伝子配列データ（一部）を使った分析 | 架空の「DNA」と称した文字列操作 |
| SQLでの実データ（サンプル）クエリ | 「データベースごっこ」 |

> **設計原則**: Laboratoryは「専門家がやっていることの本質」を保持しつつ、複雑さを調整する。本質を捨てた「易しいだけの活動」は避ける。

### 1.5 教育的公正への貢献（TIER 1への配慮）

LEARNING_FOUNDATIONS.md Section 6-7（教育的公正、限界と射程）に基づき、Laboratoryは以下の点で教育的公正に貢献する。

| 観点 | Laboratoryの役割 | 理論的根拠 |
| :--- | :--- | :--- |
| **精密コードの明示化** | 操作の意味、期待される結果を明示的に説明 | Bernstein（言語コード論） |
| **構造の提供** | ガイド付きモード、段階的な複雑化、明確なゴール | 苅谷（構造は公正のため） |
| **暗黙知の可視化** | 「専門家はこう考える」をテキストと操作で示す | Bourdieu（合理的教育学） |
| **無料アクセス** | 有料シミュレータや教材の代替 | Brown（ペアレントクラシー批判） |
| **身体性の拡張** | 支援技術（スクリーンリーダー等）での操作保証 | 身体化認知の包括的適用 |

### 1.6 6段階螺旋との対応

| Phase | 目的 | Laboratoryの役割 |
| :--- | :--- | :--- |
| **Phase 1** | 予測誤差の喚起 | パラドックス提示、失敗するシミュレーション |
| **Phase 3** | 機序の精緻化 | パラメータ操作、ホワイトボックス化 |
| **Phase 4** | メタ認知と定着 | Quizによる自己評価 |
| **Phase 6** | 構築的転移 | 応用課題、創造的探索 |

> **注記**: Phase 2（定義と形式）とPhase 5（文脈と規範）は主にテキストで対応するが、COMPARE（定義の比較）やTIMELINE（歴史的文脈）で補強可能。

## 2. Laboratoryタイプの分類

### 2.1 分類の視点

Laboratoryタイプは、以下の2軸で分類する。

1. **認知的機能（Cognitive Function）**: 読者のどの認知プロセスを活性化するか
2. **インタラクションパターン（Interaction Pattern）**: 読者がどのように操作するか

### 2.2 認知的機能による大分類

| 大分類 | 説明 | 対応するPhase | 含まれるカテゴリ |
| :--- | :--- | :--- | :--- |
| **探索・発見系** | 既知の情報を探索し、パターンや関係を発見する | Phase 1 | COMPARE, MAP, TIMELINE, NETWORK |
| **操作・実験系** | パラメータを操作し、因果関係を体感する | Phase 3 | PARAM, SIM, BUILD |
| **判断・推論系** | 情報を分析し、判断・意思決定を行う | Phase 4 | DECISION, CASE, STEP |
| **表現・創造系** | 知識を表現・創造に応用する | Phase 6 | CODE, AUDIO, VISUAL |

> **注記**: `LAB-VISUAL` は「表現・創造系」に分類されるが、実際の使用例（分子ビューア、3D解剖学ビューア等）を見ると「探索・発見系」の性質も強い。働きが「可視化による理解促進」なら探索系、「新たな表現の創出」なら創造系として捉える。

### 2.3 抽象カテゴリ一覧

#### A. 探索・発見系 (Exploration & Discovery)

読者が情報を探索し、パターンや関係を自ら発見する。

| カテゴリID | カテゴリ名 | 説明 | 使用頻度 | 認知的目標 |
| :--- | :--- | :--- | :--- | :--- |
| **LAB-COMPARE** | 比較型 | 複数の要素を並べて類似点・相違点を発見 | ★★★ (24) | パターン認識、批判的思考 |
| **LAB-MAP** | 空間探索型 | 地図・空間情報を操作し、分布・関係を発見 | ★★★ (21) | 空間的思考、文脈理解 |
| **LAB-TIMELINE** | 時系列探索型 | 時間軸に沿って変化・因果を追跡 | ★★ (13) | 時間的思考、歴史的理解 |
| **LAB-NETWORK** | ネットワーク探索型 | ノードとエッジの関係を探索・可視化 | ★★ (9) | 関係性思考、システム思考 |

#### B. 操作・実験系 (Manipulation & Experimentation)

読者がパラメータを操作し、システムの振る舞いを観察して因果関係を理解する。

| カテゴリID | カテゴリ名 | 説明 | 使用頻度 | 認知的目標 |
| :--- | :--- | :--- | :--- | :--- |
| **LAB-PARAM** | パラメータ操作型 | スライダー/入力で変数を操作し、即座に結果を確認 | ★★★★ (39) | 因果推論、モデル理解 |
| **LAB-SIM** | シミュレーション型 | システムの動的な振る舞いを時間経過で観察 | ★★★★★ (43) | 動的システム理解、予測 |
| **LAB-BUILD** | 構築型 | 要素を組み合わせて構造物・システムを作成 | ★★★ (24) | 構成的理解、設計思考 |

#### C. 判断・推論系 (Judgment & Reasoning)

読者が情報を分析し、論理的な判断・意思決定を行う。

| カテゴリID | カテゴリ名 | 説明 | 使用頻度 | 認知的目標 |
| :--- | :--- | :--- | :--- | :--- |
| **LAB-DECISION** | 意思決定型 | 選択肢を選び、結果・帰結を比較 | ★★★ (23) | 意思決定、トレードオフ理解 |
| **LAB-CASE** | ケーススタディ型 | 実際の事例を分析し、概念を適用 | ★★★ (19) | 応用力、臨床推論 |
| **LAB-STEP** | ステップ実行型 | プロセスを1ステップずつ追跡・検証 | ★★ (16) | 論理的思考、手続き理解 |

#### D. 表現・創造系 (Expression & Creation)

読者が知識を新たな形で表現・創造する。

| カテゴリID | カテゴリ名 | 説明 | 使用頻度 | 認知的目標 |
| :--- | :--- | :--- | :--- | :--- |
| **LAB-VISUAL** | 可視化型 | 抽象概念を視覚的に表現・操作 | ★★★★ (35) | 抽象化、視覚的理解 |
| **LAB-CODE** | コード実行型 | コードを書いて実行し、即座に結果を確認 | ★ (4) | プログラミング思考、試行錯誤 |
| **LAB-AUDIO** | 音響型 | 音を聴いて・操作して理解 | ★ (3) | 聴覚的理解、パターン認識 |

> **注記**: `LAB-CODE` は計算機科学に限らず、統計学（R/Python）、経済学（計量経済モデル）、生物学（バイオインフォマティクス）など、データ分析やモデリングを伴う分野にも適用可能。

### 2.7 予測誤差喚起のための設計パターン

LEARNING_FOUNDATIONS.mdの Phase 1 では「適度な予測誤差（驚き）」が学習の駆動力とされる。Laboratoryはこの「驚き」を生成する装置として設計すべきである。ただし、予測誤差が大きすぎると理解不能となり学習を阻害するため、学習者が認知的努力で解消可能な範囲（ZPD）に収める必要がある（詳細は LEARNING_FOUNDATIONS.md の3.1節および3.6節を参照）。

#### 設計パターンと具体例

| パターン | 説明 | 具体例 |
| :--- | :--- | :--- |
| **直感に反する結果** | 読者の素朴な予測が外れる体験 | 物理：真空中では羽と鉄球が同時に落ちる |
| | | 物理：重い物体も軽い物体も同じ速度で落下（ガリレオ） |
| | | 化学：水と油は混ざらないが、界面活性剤で乳化 |
| | | 心理：「選択のパラドックス」選択肢が多いと満足度が下がる |
| **失敗するシミュレーション** | 読者の操作が「正しくない」結果を生む | 経済：価格統制が闇市場を生む |
| | | 経済：最低賃金引き上げが雇用減少を招く場合 |
| | | 環境：ダム建設が下流の生態系を破壊 |
| | | 都市計画：道路拡張が渋滞を悪化させる（誘発需要） |
| **隠れた変数の発見** | 見落としていた要因の影響を体感 | 統計：交絡変数による擬似相関（アイス売上と溺死） |
| | | 医学：ランダム化されていない観察研究のバイアス |
| | | 社会学：「シンプソンのパラドックス」集団全体と部分で傾向が逆転 |
| | | 歴史：「生存者バイアス」戻ってきた飛行機だけ見て装甲を判断 |
| **スケールの錯覚** | 直感的なスケール感覚のずれ | 天文：太陽と地球の距離を1mとすると最も近い恒星は270km先 |
| | | 生物：人体の細胞数（37兆個）vs 腸内細菌数（100兆個） |
| | | 経済：国家予算の規模（1兆円 = 1万人に1億円ずつ） |
| | | 物理：原子の大部分は空っぽ（原子核が野球ボールなら電子は数km先） |
| **確率の直感と現実** | 確率的直感の誤りを体験 | 確率：モンティ・ホール問題（ドアを変えると2/3で当たる） |
| | | 確率：誕生日問題（23人で50%以上が同じ誕生日） |
| | | 医学：偽陽性の罠（陽性的中率が検査精度と異なる） |
| | | ギャンブル：「ギャンブラーの誤謬」前回の結果が次に影響しない |
| **フィードバックループ** | 予期しない連鎖反応を観察 | 生態系：捕食者を減らすと被食者も減る（トロフィックカスケード） |
| | | 経済：デフレスパイラル（物価下落→消費控え→さらに下落） |
| | | 気候：氷床融解→反射率低下→さらに温暖化（正のフィードバック） |
| | | 社会：SNSのエコーチェンバー（同質的情報が強化される） |

#### パターン × カテゴリ マッピング

各予測誤差パターンを効果的に実装できるLaboratoryカテゴリを示す。

| パターン | 推奨カテゴリ | 実装のポイント | 最適化のポイント |
| :--- | :--- | :--- | :--- |
| **直感に反する結果** | LAB-SIM, LAB-PARAM | パラメータを変えて「こう思ったのに違った」を体験させる | 最初は極端なケースで驚きを与え、次に「なぜか」の説明へ誘導。段階的に複雑化 |
| **失敗するシミュレーション** | LAB-SIM, LAB-DECISION | 読者の選択が「失敗」につながるシナリオを設計 | 失敗後に「正解」への道筋を示す。何度でもリトライ可能に。失敗理由を明示 |
| **隠れた変数の発見** | LAB-PARAM, LAB-COMPARE | 変数を追加/除去して結果の変化を観察させる | 最初は1変数から始め、徐々に変数を追加。ヒント機能で行き詰まり防止 |
| **スケールの錯覚** | LAB-VISUAL, LAB-COMPARE | 視覚的な比較で直感とのずれを示す | 身近なスケール（例：東京ドーム何個分）から宇宙スケールへ段階的に |
| **確率の直感と現実** | LAB-SIM, LAB-PARAM | 多数回試行で理論値との収束を観察させる | 少数試行→多数試行と段階的に。収束過程をアニメーションで可視化 |
| **フィードバックループ** | LAB-SIM, LAB-NETWORK | 時間経過による連鎖反応を可視化 | 最初は単純な2要素から始め、要素を追加。速度調整（スロー再生）で理解支援 |

#### 実装上の注意

1. **「驚き」の後に「理解」を**: 驚きを与えるだけでなく、なぜそうなるかの説明へ誘導する
2. **予測を明示させる**: 操作前に「どうなると思うか」を問い、予測と結果を比較させる
3. **難易度の調整**: 難しすぎる驚きは「わからない」で終わる。段階的に複雑化する

#### 著者向けチェックリスト

予測誤差を喚起するLaboratoryを設計する際のチェックリスト：

| チェック項目 | 説明 |
| :--- | :--- |
| ☐ **素朴な予測は何か？** | 読者が最初に持つであろう直感的予測を明確にする |
| ☐ **驚きのポイントは何か？** | 予測と結果のギャップを具体的に言語化する |
| ☐ **操作前に予測を促すか？** | 「どうなると思いますか？」と問いかけるUIを設計 |
| ☐ **なぜそうなるか説明するか？** | 驚きの後に理論的説明へ誘導する導線を設計 |
| ☐ **段階的に複雑化するか？** | 最初はシンプルなケースから始め、徐々に変数を追加 |
| ☐ **再試行可能か？** | 読者が条件を変えて何度も試せる設計にする |

#### 最適な予測誤差の設計原則

Laboratory設計における「適度な予測誤差」とは、単に「難しすぎない」ことではなく、**学習進捗（Learning Progress）が最大化される領域**を指す。

##### 理論的背景

| 概念 | 説明 | 出典 |
| :--- | :--- | :--- |
| **学習進捗仮説** | 学習者は「学習が進んでいる」と感じる領域に内発的に惹きつけられる | Oudeyer & Kaplan 2007 |
| **シャノンサプライズ** | 予測モデルと観測のズレの情報量。学習により減少する | Shannon 1948 |
| **Flow理論** | スキルと挑戦のバランスが取れた時に没入状態に入る | Csikszentmihalyi 1990 |

**最適な予測誤差 = サプライズ減少速度（学習進捗）が最大となる領域**

```
    高い
    ↑                    ┌─────────────┐
  予 │                    │  理解不能    │
  測 │                    │  学習進捗=0  │
  誤 │          ┌─────────┴─────────────┘
  差 │          │  最適領域（Goldilocks Zone）
    │          │  学習進捗 = 最大
    │  ┌───────┴─────────┐
    │  │  退屈            │
    │  │  学習進捗≒0     │
    └──┴─────────────────┴─────────────→ 学習者のスキル/事前知識
    低い
```

##### 設計上の工夫

| 工夫 | 説明 | 実装例 |
| :--- | :--- | :--- |
| **段階的複雑化** | 最初はシンプルに、徐々に変数を追加 | 最初は1パラメータ、慣れたら複合条件に |
| **スキャフォールディング** | ヒントや補助線を提供し、必要に応じて外す | 最初は正解範囲を表示、後で非表示に |
| **適応的難易度** | 読者の反応（正答率、操作時間）に応じて難易度調整 | 将来：動的難易度調整機能 |
| **複数エントリーポイント** | 異なる事前知識レベルに対応した入口を用意 | 「基礎から」「発展から」の選択肢 |
| **予測の明示化** | 操作前に「どうなると思うか」を問い、認知的関与を高める | 予測入力フィールド → 結果比較 |

##### 著者向けチェックリスト（最適な予測誤差）

| チェック項目 | 説明 |
| :--- | :--- |
| ☐ **段階的に難易度が上がるか？** | 最初は単純なケースから始め、成功体験を積ませる |
| ☐ **複数の事前知識レベルに対応しているか？** | 初心者と経験者の両方に価値がある設計か |

### 2.8 ゲーム意味論に基づく設計パターン（論理学・数学向け）

論理学、数学、哲学（論証分析）の分野において、**LAB-STEP（ステップ実行型）**や**LAB-DECISION（意思決定型）**を設計する際の理論的枠組みとして、ゲーム意味論（Game-Theoretic Semantics）が有用である。

#### 理論的背景

**Jaakko Hintikka（1973, 1983）**が提唱した論理学・意味論の理論。命題の真偽を**対話的なゲームのプロセス**として定義する。

| プレイヤー | 役割 |
| :--- | :--- |
| **検証者（Verifier）** | 命題が真であることを主張し、証明する |
| **反駁者（Falsifier）** | 命題が偽であることを示すため、反例を探す |

**量化子に対応するゲームルール**

| 量化子 | ルール | 選択者 |
| :--- | :--- | :--- |
| **全称量化（∀x）** | 「どんなxについても成り立つ」を検証 | 反駁者が個体を選ぶ |
| **存在量化（∃x）** | 「あるxについて成り立つ」を証明 | 検証者が個体を選ぶ |

> **学習設計への示唆**:
> 「すべての○○は△△である」という全称命題の学習では、学習者に反駁者の役割を与え、**反例を能動的に探索させる**ことが効果的。

#### 適用範囲（限定的）

| 適用可能な分野 | Laboratoryカテゴリ | 具体例 |
| :--- | :--- | :--- |
| **論理学（量化論理）** | LAB-STEP | 証明のステップ実行、反例探索 |
| **数学（証明・定理）** | LAB-STEP, LAB-DECISION | 証明の検証ゲーム、定理の境界探索 |
| **哲学（論証分析）** | LAB-DECISION | 論証の妥当性検証、反論構築 |

> **適用外の分野**: 物理シミュレーション、化学反応、生態系モデルなど、命題論理に還元されない現象にはゲーム意味論の枠組みは直接適用されない。

#### 設計パターン

| パターン | 説明 | 実装例 |
| :--- | :--- | :--- |
| **反例探索ゲーム** | 全称命題に対して反例を探させる | 「すべての素数は奇数である」→ 2は反例 |
| **証人探索ゲーム** | 存在命題に対して具体例を見つけさせる | 「100以下に双子素数は存在する」→ (3,5), (5,7)... |
| **論証分解ゲーム** | 論証を前提と結論に分解し、各ステップを検証 | 三段論法の妥当性検証 |

#### Laboratoryでの実装イメージ

**例：数学（素数の性質）**

```
[LAB-STEP: verification_game]
命題: 「すべての素数は奇数である」

1. 学習者が命題を確認
2. 反例を入力: [    2    ]
3. システムが検証: 「2は素数だが偶数 → 反例！」
4. 命題を修正: 「2を除くすべての素数は奇数である」
5. システムが再検証を試みる
```

**例：哲学（論証分析）**

```
[LAB-DECISION: argument_analysis]
論証:
  前提1: すべての人間は死ぬ
  前提2: ソクラテスは人間である
  結論: ソクラテスは死ぬ

問い: この論証に対する有効な反論を構築できるか？
選択肢:
  A) 前提1への反例を探す
  B) 前提2への反例を探す  
  C) 推論の誤りを指摘する
  D) 反論不可（妥当な論証）
```

#### 著者向けチェックリスト

| チェック項目 | 説明 |
| :--- | :--- |
| ☐ **命題は明確に定式化されているか？** | 曖昧な表現ではゲームが成立しない |
| ☐ **反例・証人の探索空間は適切か？** | 広すぎると探索不能、狭すぎると自明 |
| ☐ **学習者の役割（検証者/反駁者）は明確か？** | どちら側でプレイするかを明示 |
| ☐ **段階的なヒントが用意されているか？** | 行き詰まった際の救済策 |

### 4.1 技術アーキテクチャ

#### 設計方針

単一技術での統一は非現実的である。**Lit Web Components をシェルとし、内部で適切な技術を使い分ける**レイヤードアーキテクチャを採用する。

```
┌─────────────────────────────────────────────────────────┐
│  <lab-container>  (Lit Web Component - 共通シェル)       │
│  - 共通UI（ツールバー、ヘルプ、アクセシビリティ設定）      │
│  - 遅延読み込み / エラー境界 / 状態管理                  │
├─────────────────────────────────────────────────────────┤
│  内部レンダリング技術（カテゴリに応じて選択）             │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ DOM/CSS  │  │ SVG/D3   │  │ Canvas2D │              │
│  │ COMPARE  │  │ NETWORK  │  │ SIM      │              │
│  │ DECISION │  │ TIMELINE │  │ BUILD    │              │
│  │ CASE     │  │ MAP      │  │          │              │
│  │ STEP     │  │          │  │          │              │
│  └──────────┘  └──────────┘  └──────────┘              │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ WebGL    │  │ CodeSand │  │ WebAudio │              │
│  │ VISUAL   │  │ CODE     │  │ AUDIO    │              │
│  │ (3D)     │  │ (外部)   │  │          │              │
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘
```

#### LAB-CODE 技術選定（安定性・パフォーマンス・プライバシー重視）

**設計方針**: 実装コストより**安定性・Web Vitals・プライバシー（安全性）**を優先する。

##### 読み込み戦略

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 静的コンテンツ（即時表示）                         │
│  - コードはシンタックスハイライトのみ                        │
│  - 出力は事前生成（ビルド時）してHTMLに埋め込み              │
│  - Web Vitals: 最高（LCP < 1s）                              │
└─────────────────────────────────────────────────────────────┘
                               │
                               ▼ ユーザーが「実行」ボタンを押した時のみ
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: インタラクティブ実行（遅延読み込み）               │
│  - WASM実行環境を遅延読み込み                               │
│  - Web Workerで実行（メインスレッド非ブロック）             │
│  - 外部通信なし（完全セルフホスト）                         │
└─────────────────────────────────────────────────────────────┘
```

##### 言語別技術選定

| 言語 | 技術 | WASMサイズ | サンドボックス | セルフホスト | 理由 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **JavaScript** | QuickJS-Emscripten | ~500KB | ✅ WASM内 | ✅ | 軽量、ES2023対応、eval不要で安全 |
| **Python** | Pyodide | ~10MB | ✅ WASM内 | ✅ | CPython完全互換、科学計算ライブラリ対応 |
| **R** | webR | ~20MB | ✅ WASM内 | ✅ | R完全互換、統計分析・データ可視化 |
| **SQL** | PGlite (PostgreSQL) | ~3MB | ✅ WASM内 | ✅ | PostgreSQL完全互換、業界標準 |

**コンパイル言語（将来対応）**:

| 言語 | 現状 | 将来の技術 | ユースケース |
| :--- | :--- | :--- | :--- |
| **Rust** | 静的出力のみ | セルフホストrustc → WASM実行 | システムプログラミング、WASM開発 |
| **Go** | 静的出力のみ | セルフホストgc → WASM実行 | バックエンド開発、並行処理 |
| **C/C++** | 静的出力のみ | セルフホストclang → WASM実行 | 低レイヤープログラミング |
| **Java** | 静的出力のみ | 検討中 | エンタープライズ開発 |

> **注記**: コンパイル言語はインタプリタ言語と異なり、ブラウザ内での実行にはサーバーサイドでのコンパイルが必要。現時点ではコード表示＋事前生成された出力のみをサポート。

> **注記**: JavaScriptをブラウザの`eval()`で直接実行せず、WASM内のQuickJSで実行することで、無限ループ・メモリ爆発・XSS等のリスクを完全に排除する。

##### SQL実行環境の設計方針

**基本方針**: **PostgreSQL（PGlite）を標準とし、他DBMSの構文差異は記事内のテキストで補足**する。

| 方針 | 説明 |
| :--- | :--- |
| **実行環境** | PGlite（PostgreSQL）のみ提供 |
| **選定理由** | PostgreSQLは標準SQLに最も近く、実務で広く使用されている |
| **構文差異** | MySQL、SQL Server、Oracle等との違いは記事内の注記で説明 |
| **標準SQL重視** | ISO/ANSI SQLを教え、方言は「補足」として扱う |

**構文差異の説明例**:

```sql
-- PostgreSQL（PGliteで実行可能）
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100)
);

-- 📝 他のデータベースでの違い
-- MySQL: id INT AUTO_INCREMENT PRIMARY KEY
-- SQL Server: id INT IDENTITY(1,1) PRIMARY KEY
-- Oracle: id NUMBER + SEQUENCE
```

**他DBMSのWASM版状況**:

| DBMS | WASM版 | 状況 |
| :--- | :--- | :--- |
| PostgreSQL | PGlite ✅ | 利用可能 |
| SQLite | sql.js ✅ | 利用可能（PostgreSQLを優先） |
| MySQL | なし ❌ | WASM版は存在しない |
| SQL Server | なし ❌ | WASM版は存在しない |
| Oracle | なし ❌ | WASM版は存在しない |

##### 設計原則

| 原則 | 説明 | Web Vitals効果 |
| :--- | :--- | :--- |
| **プリレンダリング** | ビルド時にコード出力を事前生成し、初期表示は静的HTML | LCP向上 |
| **遅延読み込み** | 「実行」ボタン押下時に初めてWASM環境を読み込み | 初期ペイロード削減 |
| **Web Worker化** | コード実行はWeb Workerで行い、メインスレッドをブロックしない | INP向上 |
| **外部通信なし** | すべてセルフホスト、外部サービス依存なし | プライバシー保護 |
| **タイムアウト** | 実行時間上限（例: 5秒）を設け、無限ループを防止 | 安全性 |
| **メモリ上限** | WASMメモリ上限を設定し、メモリ爆発を防止 | 安全性 |

### 4.2.1 コンポーネントの再利用性 (Reusability)

Laboratoryは「特定記事専用」ではなく、**複数の記事・分野で再利用可能な汎用ツール**として設計する。

#### 設計原則

| 原則 | 説明 | 例 |
| :--- | :--- | :--- |
| **ロジックとコンテンツの分離** | シミュレーションのロジックはコンポーネント内に、データ・ラベル・初期値は外部から注入 | 確率分布シミュレータ：分布の種類やパラメータはMarkdownから指定 |

### 4.3 アクセシビリティ対応方針

Isidoricaは「誰もがアクセスできる知識」を目指すため、Laboratoryも WCAG 2.2 AA 準拠を原則とする。

#### カテゴリ別対応方針

| カテゴリ | 主な課題 | 対応方針 |
| :--- | :--- | :--- |
| **VISUAL (3D/Canvas)** | 視覚依存 | 代替テキスト、テキストによる説明、音声化（Sonification） |
| **AUDIO** | 聴覚依存 | 書き起こし、視覚的波形表示 |
| **SIM / PARAM** | マウス操作依存 | キーボード操作対応、スクリーンリーダーでの状態読み上げ |
| **BUILD** | ドラッグ&ドロップ依存 | キーボードによる要素選択・配置 |
| **CODE** | 視覚的フィードバック依存 | スクリーンリーダー対応エディタ、結果の音声読み上げ |
| **全カテゴリ共通** | アニメーション | `prefers-reduced-motion` 対応、アニメーション停止オプション |

#### 代替コンテンツの原則
1. **同等の学習体験**: 視覚的Laboratoryには、同じ認知的目標を達成できる代替手段を用意
2. **段階的対応**: まず静的な説明・図解を提供し、可能な範囲でインタラクティブ性を追加
3. **ユーザー選択**: ユーザーが好みのモダリティ（視覚/聴覚/テキスト）を選択可能に

#### 認知的アクセシビリティ (Cognitive Accessibility)
LEARNING_FOUNDATIONS.mdに基づき、認知的障壁を下げるための以下の設計を採用する。

1.  **強制の排除 (No Forced Interaction)**:
    *   比較や探索を「必須タスク」とせず、あくまで学習を深めるための「ボーナス/パズル」として提示する。ユーザーの自律性を尊重し、離脱を防ぐ。

2.  **失敗の正規化 (Normalization of Failure)**:
    *   シミュレーションで予期せぬ結果（バランス崩壊など）が出た場合、それを「不正解（Error）」として赤色で警告するのではなく、「興味深い発見（Discovery）」として中立的または肯定的にフィードバックする。
    *   UI例：「Oops!」ではなく、「興味深いですね。パラメータXの影響過多かもしれません」と表示。

### 4.5 レガシーブラウザ対応

#### 背景

Laboratoryは以下の技術に依存しており、レガシーブラウザでは動作しない。

| 技術 | 必要なブラウザ機能 | サポート開始 |
| :--- | :--- | :--- |
| **Lit Web Components** | Custom Elements v1 | Chrome 67+, Firefox 63+, Safari 10.1+ |
| **Constructable Stylesheets** | `adoptedStyleSheets` | Chrome 73+, Firefox 101+, Safari 16.4+ |
| **WebAssembly** | LAB-CODE用 | Chrome 57+, Firefox 52+ |

#### 方針

| 項目 | 決定 |
| :--- | :--- |
| **レガシーブラウザでのLaboratory** | 静的プレビューを表示 |
| **Polyfill導入** | ❌ 不採用（Litのフル機能再現は非現実的） |
| **完全非表示** | ❌ 不採用（学習体験の格差が大きい） |

#### 静的プレビューの構成

| 要素 | 内容 | 必須 |
| :--- | :--- | :--- |
| **スクリーンショット** | Laboratoryの初期状態または代表的な状態の画像（WebP形式） | ✅ 必須 |
| **キャプション** | Laboratoryの目的と学習ポイントの説明 | ✅ 必須 |
| **代替テキスト（alt）** | スクリーンリーダー用の詳細説明 | ✅ 必須 |
| **ブラウザ更新案内** | 「最新のブラウザで完全な体験ができます」（控えめに表示） | ⚪ 推奨 |

#### 実装方法

##### ブラウザ判定

### 5.5 品質管理

Laboratoryのコンテンツは、記事本文とは異なる観点での品質管理が必要である。

#### レビュー観点

| 観点 | 説明 | レビュワーの確認事項 |
| :--- | :--- | :--- |
| **セキュリティ** | 悪意のあるコードが含まれていないか | LAB-CODEで危険なコード（無限ループ、メモリ爆発、外部通信）がないか |
| **教育的価値** | 記事の学習目標に寄与しているか | 「予測誤差の喚起」「認知的目標」が達成されるか |
| **アクセシビリティ** | 多様なユーザーが利用可能か | キーボード操作、スクリーンリーダー対応、代替コンテンツ |
| **正確性** | シミュレーション結果が正確か | 物理法則、数式、データに誤りがないか |
| **パフォーマンス** | Web Vitalsを悪化させないか | 遅延読み込みが適切に実装されているか |

#### セキュリティチェック

LAB-CODEのコードは以下の観点でレビューする。

| チェック項目 | 説明 | 対応 |
| :--- | :--- | :--- |
| **無限ループ** | `while(true)` 等のループ | タイムアウト設定で防止（技術的対策）、レビュー時に確認 |
| **メモリ爆発** | 大量のメモリ確保 | メモリ上限設定で防止（技術的対策）、レビュー時に確認 |
| **外部通信** | `fetch`、`XMLHttpRequest` 等 | WASM内での実行により遮断（技術的対策） |
| **悪意のあるコード** | XSS、フィッシング等 | WASM内での隔離（技術的対策）、レビュー時に確認 |

> **技術的保護**: LAB-CODEはWASM内（QuickJS/Pyodide等）で実行されるため、ブラウザのDOM/ネットワークへのアクセスは原則遮断される。ただし、レビュー時のコード確認は必須。

#### 教育的価値のチェック

| チェック項目 | 説明 |
| :--- | :--- |
| **認知的目標との整合** | Laboratoryが記事の学習目標に寄与しているか |
| **予測誤差の喚起** | 読者の素朴な理解を揺さぶる設計になっているか |
| **過度な複雑さ** | 操作が複雑すぎて本質を見失わないか |
| **正統なミニチュア** | 実世界の現象を適切に単純化しているか（子供騙しになっていないか） |

#### アクセシビリティチェック

| チェック項目 | 説明 |
| :--- | :--- |
| **キーボード操作** | すべての操作がキーボードで可能か |
| **フォーカス管理** | フォーカス順序が論理的か、フォーカスリングが見えるか |
| **スクリーンリーダー** | 状態変化がアナウンスされるか（`aria-live`） |
| **代替コンテンツ** | 視覚/聴覚依存のLaboratoryに代替手段があるか |
| **prefers-reduced-motion** | アニメーションの停止オプションがあるか |

#### 品質基準

| 基準 | 必須/推奨 | 説明 |
| :--- | :--- | :--- |
| **セキュリティ** | ✅ 必須 | 悪意のあるコードは絶対に許容しない |
| **正確性** | ✅ 必須 | 誤った結果を出力するLaboratoryはマージしない |
| **キーボード操作** | ✅ 必須 | WCAG 2.2 AA準拠 |
